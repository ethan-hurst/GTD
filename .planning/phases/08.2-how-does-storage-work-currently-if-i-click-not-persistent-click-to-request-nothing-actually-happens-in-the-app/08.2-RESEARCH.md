# Research for Phase 08.2: Storage Persistence Investigation

**Objective:** Understand why the "click to request" action for persistent storage appears to do nothing, and provide a clear path to fix it. This involves investigating the existing "Persistent Storage API" (from phase 08-04) and its implementation.

## 1. Executive Summary

The issue stems from a misunderstanding of how the browser's Persistent Storage API (`navigator.storage.persist()`) works. It is not a guaranteed user-facing prompt. The browser can silently grant or deny the request based on heuristics (site engagement, PWA installation status, etc.). The current implementation likely calls this API but fails to handle the returned promise, so the UI never updates to reflect the outcome (granted, denied, or already active). The user correctly observes that "nothing happens" because there is no feedback loop.

The fix will involve:
1.  Identifying the UI component that makes the request.
2.  Properly handling the `true`/`false` result from the `navigator.storage.persist()` promise.
3.  Updating a central store (`storage.svelte.ts`) with the definitive persistence status.
4.  Ensuring the UI reacts to changes in this store to show the user what happened.

## 3. Technical Deep Dive: `StorageManager` API

The core of this feature is the `navigator.storage` object, which is part of the Web Storage API.

*   **`navigator.storage.persisted()`**: This method returns a `Promise<boolean>`.
    *   It resolves to `true` if storage for the site is already marked as persistent.
    *   It resolves to `false` otherwise.
    *   This should be called on app load to determine the initial UI state.

*   **`navigator.storage.persist()`**: This method returns a `Promise<boolean>`.
    *   It requests that the browser make the site's storage persistent.
    *   It resolves to `true` if persistence was successfully enabled.
    *   It resolves to `false` if the browser denied the request. **Crucially, the browser is not required to show a prompt to the user and can deny the request silently.**

The most likely failure mode is that the code calls `navigator.storage.persist()` but doesn't do anything with the `false` result, leaving the UI in its initial state.

## 4. Actionable Insights for Planning

To resolve this, the planning phase should generate tasks to implement the following logic:

1.  **State Management (`storage.svelte.ts`):**
    *   The store should hold at least three states: `'UNKNOWN'`, `'GRANTED'`, `'DENIED'`.
    *   Create an async function `checkPersistence()` that calls `navigator.storage.persisted()` and updates the store to either `'GRANTED'` or `'UNKNOWN'`. This should be run when the app initializes.

2.  **UI Component (to be located):**
    *   On mount, the component should call `checkPersistence()`.
    *   The UI should display different content based on the store's state (e.g., "Storage is persistent", "Enable persistent storage?", "Persistent storage not available").
    *   The "Enable" button should trigger a new function, `requestPersistence()`.

3.  **Request Logic (`storage.svelte.ts`):**
    *   The `requestPersistence()` function should call `navigator.storage.persist()`.
    *   It **must** handle the promise result:
        *   If it resolves `true`, update the store state to `'GRANTED'`.
        *   If it resolves `false`, update the store state to `'DENIED'`.
    *   Add console logs for both success and failure cases to aid debugging.

4.  **Verification Plan:**
    *   The plan must include steps to manually verify the UI changes for all three states. Since forcing a browser to deny persistence is difficult, the verification may need to rely on checking console logs and mocking the API's response if possible.

This research provides a clear, technically-grounded path forward for the planner to create concrete, executable tasks.
