# Phase 08.2: Storage Persistence Bug - Context

**Gathered:** 2026-01-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the broken "click to request" persistent storage flow in the StatusBar. The `navigator.storage.persist()` promise result isn't handled, so the UI never updates. Add proper state management, user feedback, and a Storage dashboard in Settings.

</domain>

<decisions>
## Implementation Decisions

### Feedback pattern
- Success feedback: Claude's discretion (toast + status update vs status update only — pick what fits existing app patterns using `svelte-5-french-toast`)
- Failure feedback: Toast with browser-specific explanation + red indicator in StatusBar
- Loading state: Show spinner on button + disable during `persist()` request
- On-load confirmation: Show toast every app load when storage IS persistent ("Storage is persistent" or similar)
- On-load non-persistent warning: Claude's discretion on whether to show a warning toast or let the amber indicator suffice

### State display
- StatusBar states: green dot (persistent), amber dot + button (not persistent), red dot (denied), gray dot (API unavailable)
- Quota display: Only show storage quota (% used, MB) when storage IS persistent
- Settings page: Add full Storage dashboard section with status, visual quota bar, last save time, request/retry button, and explanation of what persistence means
- Denied state in StatusBar: Claude's discretion on exact styling — should include retry option

### Denied/recovery flow
- Retry available in both StatusBar and Settings page
- Denial guidance: Browser-specific tips (detect browser, show relevant advice — Chrome: bookmark, Safari: add to home screen, etc.)
- Retry cooldown: Claude's discretion on whether to add a brief cooldown between attempts
- API unavailable: Show gray indicator with "Storage API not available" in both StatusBar and Settings

### Toast system
- Use existing `svelte-5-french-toast` library (already in app at `<Toaster position="top-center" />`)
- Toast style: Claude's discretion (standard `toast.success()`/`toast.error()` or custom — match app patterns)
- Toast duration: Claude's discretion based on message importance (existing app uses 1500ms for captures)

### Claude's Discretion
- Exact toast wording and duration
- Whether to show a non-persistent warning toast on load (vs just the amber indicator)
- Retry cooldown timing (if any)
- Denied state StatusBar styling details
- Toast styling (standard vs custom icons)
- Settings dashboard visual layout

</decisions>

<specifics>
## Specific Ideas

- User explicitly wants red indicator at bottom of site when denied
- User wants confirmation toast on every app load when persistent (reassurance that data is safe)
- Browser-specific guidance is important — not just generic "try later"
- Full dashboard in Settings: status + quota bar + last save + request button + explanation
- Spinner + disable on the request button during persist() call

</specifics>
