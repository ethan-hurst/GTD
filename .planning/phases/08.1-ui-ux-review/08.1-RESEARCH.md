# Phase 08.1: UI/UX Review - Research

**Researched:** 2026-01-31
**Domain:** UI/UX consistency, design system audit, accessibility
**Confidence:** HIGH

## Summary

This research phase audited the existing GTD app codebase to understand current UI patterns, identify inconsistencies, and establish design principles for v1.0 UI/UX polish before layering Outlook sync features.

The app is built with SvelteKit 2, Svelte 5, and Tailwind v4 with class-based dark mode. Analysis of 25 components and 8 route pages revealed a mostly consistent but ad-hoc design system. Key findings:

1. **Spacing/typography are mostly consistent** but lack formal design tokens
2. **Dark mode coverage is comprehensive** (367 instances of dark: variants) but needs contrast validation
3. **Interactive states exist** (hover, focus) but focus ring patterns are inconsistent
4. **Empty states are well-implemented** across all views
5. **No responsive breakpoints** — app is desktop-only (only 1 breakpoint found)
6. **Button patterns vary** — 3-4 different button style combinations for primary/secondary actions

**Primary recommendation:** Audit and standardize spacing scale, typography hierarchy, button patterns, and focus ring implementation. Establish formal design tokens in Tailwind config. Validate dark mode contrast ratios.

### Anti-Patterns to Avoid

**Don't create component libraries mid-project**
This is a polish phase, not a rewrite. Don't extract every button into `<Button variant="primary">`. That's premature abstraction. Instead, document and standardize the Tailwind class patterns.

**Don't introduce new color values**
The app uses gray/blue/green/red/yellow/purple/amber semantic colors. Stick with these. Don't add new brand colors without user approval.

**Don't change layout paradigms**
The app is desktop-first with fixed sidebar. Don't introduce mobile hamburger menus or responsive grids without explicit requirement.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Color contrast validation | Manual eyeballing | Browser DevTools Contrast Checker, WAVE evaluator | WCAG requires 4.5:1 for normal text, 3:1 for large text |
| Dark mode color inversion | Swap light/dark values | Audit with real users, test tools | Pure inversion breaks hierarchy; needs manual tuning |
| Design token system | Custom CSS variables | Tailwind 4's @theme directive | Already in app.css; extend rather than replace |
| Focus ring styles | Custom outlines | Tailwind's focus:ring-* utilities | Browser defaults work; augment with ring-2 ring-blue-500 |
| Responsive breakpoints | Inline media queries | Tailwind's sm:/md:/lg: prefixes | Already available; app just doesn't use them |

**Key insight:** Tailwind CSS already provides design tokens, focus utilities, and responsive patterns. This phase is about USING them consistently, not building alternatives.

## Common Pitfalls

### Pitfall 1: Inconsistent Spacing Scale
**What goes wrong:** Different components use different padding/margin values (p-4, p-6, p-8, px-3, px-4) without clear hierarchy
**Why it happens:** Developers choose values that "look right" without consulting a spacing scale
**How to avoid:**
- Audit all spacing values across components
- Document standard spacing scale: sm (2), md (4), lg (6), xl (8)
- Replace ad-hoc values with scale values
**Warning signs:** Grep shows `px-4` (75), `px-3` (45), `px-2` (20) — inconsistent

### Pitfall 2: Missing Focus Rings on Interactive Elements
**What goes wrong:** Keyboard users can't see which element is focused
**Why it happens:** Developers test with mouse, forget keyboard navigation
**How to avoid:**
- Every clickable element needs visible focus state
- Use `focus:ring-2 focus:ring-blue-500` for custom elements
- Test with Tab key navigation
**Warning signs:** Found some buttons without explicit focus: styles (relying on browser defaults)

### Pitfall 3: Dark Mode Contrast Issues
**What goes wrong:** Text becomes unreadable in dark mode (e.g., gray-400 on gray-900)
**Why it happens:** Assuming light mode color pairs work in dark mode
**How to avoid:**
- Test every text/background pair with contrast checker
- Use softer dark backgrounds (#121212, not #000000)
- Avoid pure white text on dark mode
**Warning signs:** Some instances of `text-gray-400` which may fail 4.5:1 contrast on `bg-gray-900`

### Pitfall 4: Inconsistent Button Patterns
**What goes wrong:** Primary actions look different across pages
**Why it happens:** Copy-paste from different components, no central pattern
**How to avoid:**
- Document 3 button sizes: sm (py-1.5 px-3), md (py-2.5 px-4), lg (py-3 px-6)
- Standardize border-radius: md for standard buttons, lg for hero CTAs
- Use same hover/dark transitions everywhere
**Warning signs:** Found 4 different primary button patterns (see Architecture Patterns)

### Pitfall 5: No Responsive Behavior
**What goes wrong:** App breaks on narrow screens, tablet users get desktop layout
**Why it happens:** Desktop-first development without mobile testing
**How to avoid:**
- Define minimum supported screen width (e.g., 1024px)
- OR implement responsive patterns with Tailwind breakpoints
- Test on 1024px, 768px, 375px widths
**Warning signs:** Only 1 responsive class found in entire codebase

### Pitfall 6: Typography Hierarchy Not Clear
**What goes wrong:** H1/H2/H3 sizes don't clearly differentiate importance
**Why it happens:** No documented type scale
**How to avoid:**
- Define scale: H1 (text-3xl), H2 (text-2xl), H3 (text-xl), Body (text-base), Small (text-sm), Caption (text-xs)
- Pair with font-weight: Headings get font-bold, body gets font-normal, labels get font-medium
**Warning signs:** Grep shows text-2xl (10), text-xl (2), text-base (15) — needs validation

## State of the Art

UI/UX best practices for GTD and productivity apps in 2026:

| Old Approach (v1.0 current state) | Current Best Practice (2026) | Impact |
|-----------------------------------|------------------------------|--------|
| Ad-hoc Tailwind classes | Design tokens in @theme directive | Consistency, easier updates |
| Pure black backgrounds (#000) | Softer dark grays (#121212, gray-950) | Reduces eye strain, better for dyslexia/astigmatism |
| Browser-default focus outlines | Custom focus rings (ring-2 ring-blue-500) | Clearer keyboard navigation |
| Desktop-only layout | Responsive or defined minimum width | User expectations in 2026 |
| Color-only status indicators | Color + icon/text | Accessibility for colorblind users |
| 4.5:1 minimum contrast | Comfort-based contrast (exceed WCAG minimums) | Better readability for all users |

**Deprecated/outdated:**
- Pure color inversion for dark mode (breaks visual hierarchy)
- Hardcoded pixel values instead of design tokens
- Focus states that match hover exactly (should be distinct)

## Design System Best Practices (2026)

Based on research findings:

### Design Token System
**Source:** [Tailwind CSS Best Practices 2025-2026](https://www.frontendtools.tech/blog/tailwind-css-best-practices-design-system-patterns)

Define design tokens in Tailwind 4's @theme directive:
- **Spacing:** Use 4px base scale (0.5, 1, 1.5, 2, 2.5, 3, 4, 5, 6, 8, 10, 12, 16, 20, 24)
- **Typography:** Define semantic font sizes (xs, sm, base, lg, xl, 2xl, 3xl) with line heights
- **Colors:** Use semantic color names (primary, secondary, danger, success) that map to gray/blue/red/green

All components should reference tokens, not arbitrary values, allowing system-wide updates.

### Consistency Principles
**Source:** [Design Consistency Guide UI and UX Best Practices](https://www.uxpin.com/studio/blog/guide-design-consistency-best-practices-ui-ux-designers/)

When buttons, colors, and interactions remain uniform throughout, users build familiarity and confidence. Same spacing, typography, and color usage across every screen creates perceived quality.

Consistency limits representations — users don't learn new patterns for each task. Established design norms (like platform conventions) let users complete tasks without learning new toolsets.

### Dark Mode Accessibility
**Source:** [Dark Mode Design Best Practices in 2026](https://www.tech-rz.com/blog/dark-mode-design-best-practices-in-2026/)

**Key requirements:**
- Avoid pure black (#000000) — causes eye strain and "halation effect"
- Use dark grays like #121212 (gray-950 in Tailwind)
- Replace pure white text with softer off-whites like #E0E0E0
- Maintain 4.5:1 contrast for body text, 3:1 for large headings
- Accommodate users with dyslexia, astigmatism, light sensitivity

Tools: WAVE evaluator, Chrome DevTools Contrast Checker

### Interactive Element States
**Source:** [A guide to designing accessible, WCAG-conformant focus indicators](https://www.sarasoueidan.com/blog/focus-indicators/)

Every interactive element requires 5 states: default, hover, focus, active, disabled.

**Focus ring requirements (WCAG 2.1):**
- If keyboard-operable, must have visible focus (SC 2.4.7)
- Focus indicators should be consistent with hover states
- Pair color with icons/shapes, not color-only cues
- Include focus indicators early in design process
- Use `outline` or `ring` utilities (don't affect layout)

Best practice: Hover and focus states can share styling (underlines, bg changes, animations) but focus should have additional ring for clarity.

### GTD-Specific UX Patterns
**Source:** [Design zen: GTD in UI/UX productivity guide](https://medium.com/design-bootcamp/design-zen-unleashing-the-power-of-gtd-in-ui-ux-a-productivity-guide-for-creative-flow-ec86a63e01fd)

GTD apps require:
- **Seamless capture:** Fast input with keyboard shortcuts (app has this: `/` to focus inbox)
- **Context filtering:** Filter by @home, @office, @computer (app has this in ContextList)
- **Quick processing:** Minimal clicks to triage items (app has this: ProcessingFlow wizard)
- **Information density vs whitespace:** Balance readability with overview (needs audit)
- **Status visualization:** Clear affordances for completion, delegation, etc. (needs consistency check)

## Open Questions

Things that couldn't be fully resolved:

### 1. Responsive Strategy
- **What we know:** App has virtually no responsive breakpoints (only 1 found)
- **What's unclear:** Is this desktop-only by design, or an oversight?
- **Recommendation:** Define minimum supported width (suggest 1024px). If mobile support is out of scope for v1.1, document this explicitly. If needed, add responsive patterns to calendar and sidebar.

### 2. Component Extraction vs Documentation
- **What we know:** 4+ button pattern variations exist
- **What's unclear:** Should we extract to Svelte components (`<Button variant="primary">`) or just document Tailwind patterns?
- **Recommendation:** Start with documentation. Component extraction is premature until patterns stabilize. Create a UI_PATTERNS.md guide instead.

### 3. Animation/Transition Timing
- **What we know:** Found `transition-colors` on many elements, `slide` transitions on panels
- **What's unclear:** Are transition durations consistent? Should they be?
- **Recommendation:** Audit transition timing. Tailwind defaults are 150ms — check if custom durations are needed.

### 4. Click Target Size
- **What we know:** WCAG requires 44x44px minimum for touch targets
- **What's unclear:** Desktop app may not need touch targets, but what about accessibility?
- **Recommendation:** Verify all interactive elements are at least 44px tall (current buttons with py-2.5 are ~40px)
