# Phase 08.3: Left Nav Bar UX Improvement - Research

**Researched:** 2026-01-31
**Domain:** Sidebar Navigation UI/UX, Vertical Space Optimization
**Confidence:** HIGH

## Summary

This research investigates techniques for eliminating the scrollbar in the left navigation sidebar on a 14" MacBook Pro at full screen (approximately 900-950px viewport height), while preserving all existing GTD navigation functionality. The sidebar currently contains 11 items: app header, 8 nav links (Inbox, Next Actions, Projects, Waiting For, Someday/Maybe, Weekly Review, Calendar, Settings), context list (3-5 items typically), and theme toggle footer.

The standard approach combines multiple techniques: reducing vertical spacing density, making the context list collapsible by default, and potentially adding a sidebar collapse toggle. Modern CSS provides native solutions via the `<details>`/`<summary>` elements for accessible accordions, and Tailwind CSS 4 provides clean utility patterns for implementing collapsible states and responsive behavior.

**Primary recommendation:** Use a combination of tighter vertical spacing (reduce padding from py-2 to py-1.5), make the context list collapsible via native `<details>`/`<summary>` HTML (collapsed by default), and add an optional sidebar collapse toggle using Svelte 5 $state runes for clean state management. This should eliminate the scrollbar on 14" MBP while maintaining professional appearance and all functionality.

### Pattern 1: Spacing Reduction

**What:** Reduce padding on navigation items from py-2 (8px) to py-1.5 (6px)

**When to use:** First step - eliminates ~16-24px of total height with minimal visual impact

**Calculation:**
- Current: 11 nav items × 8px padding × 2 = 176px + 5 separators × 8px margin = 216px
- Optimized: 11 items × 6px padding × 2 = 132px + 5 separators × 4px margin = 152px
- **Savings: 64px**

**Example:**
```svelte
<!-- Before -->
<a class="block px-4 py-2 rounded-md ...">Next Actions</a>

<!-- After -->
<a class="block px-4 py-1.5 rounded-md ...">Next Actions</a>
```

### Pattern 2: Collapsible Context List (Native HTML)

**What:** Use `<details>`/`<summary>` for context list, collapsed by default

**When to use:** Primary space-saving technique - most contexts only needed when filtering

**Benefits:**
- Native HTML accordion - built-in accessibility (keyboard, screen reader)
- Zero JavaScript required for collapse/expand behavior
- Browser handles state management automatically
- Exclusive accordions via `name` attribute (supported in modern browsers 2026)

**Example:**
```svelte
<!-- Source: https://developer.mozilla.org/en-US/blog/html-details-exclusive-accordions/ -->
<details class="space-y-1">
	<summary class="flex items-center justify-between px-4 py-1.5 cursor-pointer
		text-xs font-semibold text-gray-500 dark:text-gray-400 uppercase tracking-wider
		hover:bg-gray-100/70 dark:hover:bg-gray-800/70 rounded-md">
		<span>Contexts</span>
		<svg class="w-4 h-4 transition-transform duration-150" ...>
			<!-- Chevron icon rotates on open -->
		</svg>
	</summary>

	<div class="pl-2 space-y-1">
		<!-- Context items here -->
	</div>
</details>
```

**Styling details marker:**
```css
/* Hide default arrow, use custom icon */
summary::-webkit-details-marker {
	display: none;
}
summary::marker {
	content: "";
}

/* Rotate icon when open */
details[open] summary svg {
	transform: rotate(180deg);
}
```

### Pattern 3: Sidebar Collapse Toggle

**What:** Add hamburger/collapse button to toggle sidebar between 240px and 64px (icon-only)

**When to use:** Nice-to-have feature for extra content space

**State management with Svelte 5:**
```svelte
<script>
	import { writable } from 'svelte/store';

	// Persist in localStorage
	let isCollapsed = $state(
		typeof window !== 'undefined'
			? localStorage.getItem('sidebar-collapsed') === 'true'
			: false
	);

	function toggleSidebar() {
		isCollapsed = !isCollapsed;
		if (typeof window !== 'undefined') {
			localStorage.setItem('sidebar-collapsed', String(isCollapsed));
		}
	}
</script>

<aside class="{isCollapsed ? 'w-16' : 'w-60'} transition-all duration-150 ...">
	<!-- Conditional rendering for collapsed state -->
</aside>
```

**Collapsed state considerations:**
- Show only icons for nav items (requires adding icons to all links)
- Hide text labels, context list completely
- Keep theme toggle as icon-only
- Use Tailwind `group-hover` for tooltip-like labels

### Pattern 4: Visual Grouping (Replace Separators)

**What:** Replace horizontal line separators with subtle spacing and optional section headers

**When to use:** Reduces visual clutter and saves 2-4px per separator (5 separators = 10-20px)

**Example:**
```svelte
<!-- Before: Separator line (my-2 = 8px margin) -->
<div class="border-t border-gray-200/50 dark:border-gray-700/30 my-2"></div>

<!-- After: Just spacing -->
<div class="mt-3"></div>

<!-- Alternative: Section header (more semantic) -->
<div class="px-4 py-1 text-xs font-semibold text-gray-500 uppercase tracking-wider">
	Organize
</div>
```

### Anti-Patterns to Avoid

- **Removing functionality to save space:** Don't hide nav items or remove features - compress presentation instead
- **Overflow hidden without scroll affordance:** If content still overflows, don't hide scrollbar - indicates design issue
- **Inconsistent spacing:** Use systematic spacing scale (4px, 6px, 8px, 12px, 16px) - don't mix random values
- **Breaking GTD hierarchy:** Inbox, Next Actions, Projects are primary - don't de-emphasize them
- **Auto-expanding context list:** Defeats space-saving purpose - keep collapsed by default

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Accessible accordion | Custom JS with ARIA attributes | Native `<details>`/`<summary>` | Built-in keyboard navigation, screen reader support, simpler code |
| Scrollbar styling | Manual overflow handling | CSS `scrollbar-width` + `::-webkit-scrollbar` | Cross-browser, standard approach |
| Sidebar state persistence | Custom storage logic | Svelte $state + localStorage | Reactive updates, clean code |
| Smooth transitions | Manual CSS animations | Tailwind `transition-all duration-150` | Consistent with existing codebase |
| Icon management | SVG in each component | Consider icon component library | If adding icons to all nav items for collapse mode |

**Key insight:** Native HTML and CSS have evolved significantly - `<details>` element is now production-ready with excellent accessibility support. Don't reinvent these with custom JavaScript.

## Common Pitfalls

### Pitfall 1: Overflow Calculation Errors

**What goes wrong:** Forgetting to account for header, footer, and border heights when calculating available space for nav content

**Why it happens:** Only measuring nav item heights, not total sidebar content

**How to avoid:**
- MacBook Pro 14" viewport: ~900-950px height (varies by browser chrome)
- Sidebar structure breakdown:
  - Header: ~56px (py-4 on container + text height + border)
  - Footer: ~60px (p-4 + theme toggle height + border)
  - Available for nav: ~784-834px
  - Current nav height estimate: 11 items × (32px item + 8px margin) + 5 separators × 16px = ~520px + context list expansion
  - **Context list expansion is the problem:** 3-5 contexts × 40px + add form = 120-200px+ **overflow**

**Warning signs:** Scrollbar appears/disappears when context list expands or when window resizes slightly

**Solution:** Make context list collapsible (saves 120-200px when collapsed) - this alone should solve the issue

### Pitfall 2: Accessibility Regression with Hidden Scrollbar

**What goes wrong:** Hiding scrollbar with CSS removes visual indicator that content is scrollable, confusing keyboard users

**Why it happens:** Attempting `overflow: auto` + `scrollbar-width: none` as a quick fix instead of addressing root cause

**How to avoid:**
- Don't hide the scrollbar - eliminate the need for scrolling instead
- If overflow is unavoidable (edge case: user adds 20+ contexts), keep scrollbar visible
- Never use `overflow: hidden` if content is actually truncated

**Warning signs:** Content is cut off, keyboard users can't reach items, complaints about missing nav links

**Solution:** Properly size content to fit viewport, use collapsible sections to manage overflow

### Pitfall 3: Breaking Svelte Reactivity with Direct DOM Manipulation

**What goes wrong:** Using vanilla JS to toggle classes for collapse instead of Svelte $state, breaking reactive updates

**Why it happens:** Copying jQuery/vanilla JS patterns from web search results

**How to avoid:**
- Use Svelte 5 $state runes for all stateful UI
- Let Svelte handle DOM updates via reactive declarations
- Avoid `document.querySelector` and manual `classList.toggle`

**Warning signs:** State and UI get out of sync, toggling doesn't work after navigation, console errors

**Example (correct approach):**
```svelte
<script>
	let isCollapsed = $state(false);
	// Svelte handles the class reactivity automatically
</script>

<aside class="{isCollapsed ? 'w-16' : 'w-60'}">
```

### Pitfall 4: Poor Mobile/Tablet Experience

**What goes wrong:** Focusing only on 14" MBP desktop, creating a sidebar that's unusable on smaller screens

**Why it happens:** Phase context specifies 14" MBP as primary target, ignoring responsive design

**How to avoid:**
- Current implementation already has `overflow-y-auto` for smaller viewports - maintain this
- Sidebar collapse toggle is especially valuable on tablets (1024px width)
- Test on common tablet viewport: 768×1024 and 834×1194 (iPad)

**Warning signs:** Sidebar takes up too much horizontal space on tablet, can't reach footer on small laptops

**Solution:** Keep responsive design - don't make it 14" MBP exclusive. The optimizations should work across screen sizes.

### Pitfall 5: Losing Context List Usability

**What goes wrong:** Making context list so hidden/hard to access that users stop using contexts altogether

**Why it happens:** Over-optimizing for space at expense of functionality

**How to avoid:**
- Make summary/header clearly clickable (cursor-pointer, hover state)
- Use clear chevron icon to indicate expand/collapse affordance
- Persist open/closed state in localStorage if user manually expands
- Consider showing count badge even when collapsed: "Contexts (3)"

**Warning signs:** User feedback that contexts are "missing", confusion about where to filter by context

**Solution:** Balance space-saving with discoverability - collapsed by default but obvious how to expand

### Sidebar Collapse Toggle with Svelte 5

```svelte
<script>
	import { onMount } from 'svelte';

	// Initialize from localStorage
	let isCollapsed = $state(false);

	onMount(() => {
		const stored = localStorage.getItem('sidebar-collapsed');
		if (stored !== null) {
			isCollapsed = stored === 'true';
		}
	});

	function toggleSidebar() {
		isCollapsed = !isCollapsed;
		localStorage.setItem('sidebar-collapsed', String(isCollapsed));
	}
</script>

<aside class="{isCollapsed ? 'w-16' : 'w-60'}
	border-r border-gray-200/70 dark:border-gray-800/50
	bg-gray-50/80 dark:bg-gray-900/95
	backdrop-blur-sm flex flex-col
	transition-all duration-150">

	<!-- Header with collapse toggle -->
	<div class="px-5 py-4 border-b border-gray-200/50 dark:border-gray-800/50
		flex items-center justify-between">
		{#if !isCollapsed}
			<h1 class="gtd-logo text-xl font-bold">GTD</h1>
		{/if}
		<button
			onclick={toggleSidebar}
			class="p-2 rounded-md hover:bg-gray-100/70 dark:hover:bg-gray-800/70
				focus:outline-none focus:ring-2 focus:ring-blue-500/40
				transition-all duration-150"
			title={isCollapsed ? "Expand sidebar" : "Collapse sidebar"}>
			<svg class="w-5 h-5" fill="none" stroke="currentColor" viewBox="0 0 24 24">
				{#if isCollapsed}
					<!-- Expand icon (chevron right) -->
					<path stroke-linecap="round" stroke-linejoin="round" stroke-width="2" d="M9 5l7 7-7 7" />
				{:else}
					<!-- Collapse icon (chevron left) -->
					<path stroke-linecap="round" stroke-linejoin="round" stroke-width="2" d="M15 19l-7-7 7-7" />
				{/if}
			</svg>
		</button>
	</div>

	<!-- Nav content with conditional rendering -->
	<nav class="flex-1 overflow-y-auto p-4 space-y-1">
		{#if isCollapsed}
			<!-- Icon-only nav items -->
			<a href="/" class="flex items-center justify-center p-2 ..." title="Inbox">
				<!-- Icon only -->
			</a>
		{:else}
			<!-- Full nav items (existing implementation) -->
		{/if}
	</nav>
</aside>
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Custom JS accordions with ARIA | Native `<details>`/`<summary>` | 2024-2025 | Exclusive accordions via `name` attribute now in all major browsers |
| jQuery slideToggle | CSS transitions + Svelte reactivity | 2020+ | Simpler code, better performance, no jQuery dependency |
| Vendor-prefixed scrollbar styling | Standard `scrollbar-width` property | 2023+ | Firefox standard support, cleaner CSS |
| Fixed sidebar widths | Collapsible sidebars with localStorage state | 2022+ | Better UX on tablets and smaller displays |
| Complex CSS grid layouts | Flexbox for vertical nav layouts | 2020+ | Simpler mental model, better browser support |

**Deprecated/outdated:**
- **jQuery-based sidebar toggles:** Use Svelte $state instead for reactive updates
- **ARIA-heavy custom accordions:** Native `<details>` is more accessible and simpler
- **Fixed 240px sidebars:** User-collapsible sidebars are now standard in modern apps
- **Overflow hidden with fade gradients:** Better to properly size content than fake truncation

## Open Questions

Things that couldn't be fully resolved:

1. **Exact viewport height for 14" MBP in full-screen browser**
   - What we know: Native resolution 3024×1964 (254 PPI), viewport typically 100-150px less due to browser chrome
   - What's unclear: Exact pixel height varies by browser (Safari vs Chrome vs Firefox) and toolbar visibility
   - Recommendation: Target 900px as conservative estimate, test actual device. Optimizations should work for 850-950px range.

2. **Icon library for collapsed sidebar mode (if implemented)**
   - What we know: Current nav items don't have icons, only text labels
   - What's unclear: Whether to add icon library (heroicons, lucide, etc.) or use inline SVG
   - Recommendation: If implementing sidebar collapse, use inline SVG for ~8 icons to avoid dependency. Icons already used in ContextList for edit/delete.

3. **Context list state persistence**
   - What we know: Native `<details>` doesn't persist open/closed state across page loads
   - What's unclear: Should collapsed state persist, or always start collapsed?
   - Recommendation: Start simple with always-collapsed. If user feedback requests it, add localStorage persistence with `open` attribute.
