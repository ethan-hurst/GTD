# Phase 08.8: Feedback System with Netlify Blobs - Research

**Researched:** 2026-02-02
**Domain:** Replacing Netlify Forms with Netlify Blobs for feedback storage + admin dashboard
**Confidence:** HIGH

## Summary

This research investigates how to replace the current Netlify Forms-based feedback collection with a Netlify Blobs-backed system, adding an admin dashboard page for viewing and managing submissions. The codebase already has a fully working analytics system (`analytics-track.mts`, `analytics-query.mts`, `/analytics` page) that provides the exact template for this implementation.

The approach is straightforward: create a `feedback-submit` Netlify Function that receives POST data and stores it as JSON in a `feedback` Blobs store, create a `feedback-query` Netlify Function that lists/filters feedback items with Basic Auth protection, build an admin page at `/admin/feedback` (or `/feedback-admin` -- see architecture section), and update the existing `FeedbackModal.svelte` to POST JSON to the new function instead of FormData to `/feedback-form.html`.

**Key design decisions:** Store each feedback item as its own blob with key `feedback/{timestamp}-{random}`. Store screenshot data as a separate blob to keep the listing lightweight. Use the same `ANALYTICS_PASSWORD` env var for auth (reuse existing admin credential). Use `setJSON` for structured data and `list({ prefix: 'feedback/' })` for querying.

**Primary recommendation:** Follow the analytics system pattern exactly -- same auth mechanism, same function structure, same admin page layout -- but adapt the data model for feedback items with status management and screenshot storage.

### Pattern 1: Blob Key Structure for Feedback Items

**What:** Each feedback submission gets its own blob. Keys are structured for sortable listing and efficient prefix queries.

**Key format:** `items/{ISO-timestamp}_{4-char-random-hex}`

Example: `items/2026-02-02T14:30:00.000Z_a3f1`

**Why this format:**
- `items/` prefix groups all submissions under one namespace, separate from `screenshots/` and any future prefixes
- ISO timestamp sorts chronologically by default (lexicographic sort = chronological sort)
- Random suffix prevents key collision if two submissions arrive at the same millisecond
- `list({ prefix: 'items/' })` returns all submissions sorted by time

**Screenshot key:** `screenshots/{same-key-suffix}` -- e.g., `screenshots/2026-02-02T14:30:00.000Z_a3f1`

This separation is critical: when listing feedback items, you do NOT want to download multi-hundred-KB screenshot data. List the items (small JSON), and only fetch the screenshot when the admin clicks into a detail view.

**Source:** Follows the same prefix-based pattern as analytics (`daily/YYYY-MM-DD`, `salt/YYYY-MM-DD`) already established in `analytics-utils.ts`.

```typescript
// Generate feedback blob key
function generateFeedbackKey(): string {
  const timestamp = new Date().toISOString();
  const random = Math.random().toString(16).slice(2, 6);
  return `items/${timestamp}_${random}`;
}
```

### Pattern 2: Feedback Data Model

**What:** Each feedback blob stores a JSON object with all submission data except the screenshot binary.

```typescript
interface FeedbackItem {
  id: string;           // Same as blob key suffix: "2026-02-02T14:30:00.000Z_a3f1"
  type: 'bug' | 'feature';
  description: string;
  email?: string;
  status: 'new' | 'reviewed' | 'resolved' | 'archived';
  hasScreenshot: boolean;
  submittedAt: string;  // ISO 8601
  userAgent: string;    // Browser UA for bug context
  reviewedAt?: string;  // ISO 8601, set when status changes
  notes?: string;       // Admin notes
}
```

**Metadata (Blobs metadata, max 2KB):** Use for lightweight filtering fields that can be read without downloading the full blob.

```typescript
// Stored as Blobs metadata (Record<string, string>)
{
  type: 'bug',           // 'bug' | 'feature'
  status: 'new',         // 'new' | 'reviewed' | 'resolved' | 'archived'
  hasScreenshot: 'true', // 'true' | 'false'
  submittedAt: '2026-02-02T14:30:00.000Z'
}
```

**Important:** Blobs metadata values must be strings (Record<string, string>). Booleans and numbers must be stringified.

### Pattern 3: Netlify Function Structure (Matching Analytics Pattern)

**What:** Follow the exact same function structure as `analytics-track.mts` and `analytics-query.mts`.

**Submit function (`feedback-submit.mts`):**

```typescript
import { getStore } from '@netlify/blobs';
import type { Context } from '@netlify/functions';

export default async (req: Request, context: Context) => {
  if (req.method !== 'POST') {
    return new Response(JSON.stringify({ error: 'Method not allowed' }), {
      status: 405,
      headers: { 'Content-Type': 'application/json' }
    });
  }

  try {
    const body = await req.json();
    const { type, description, email, screenshot, botField } = body;

    // Honeypot spam check
    if (botField) {
      // Silently accept but don't store (bot detected)
      return new Response(JSON.stringify({ success: true }), {
        status: 200,
        headers: { 'Content-Type': 'application/json' }
      });
    }

    // Validate required fields
    if (!type || !['bug', 'feature'].includes(type)) {
      return new Response(JSON.stringify({ error: 'Invalid type' }), {
        status: 400,
        headers: { 'Content-Type': 'application/json' }
      });
    }

    if (!description || typeof description !== 'string' || description.length < 10) {
      return new Response(JSON.stringify({ error: 'Description required (min 10 chars)' }), {
        status: 400,
        headers: { 'Content-Type': 'application/json' }
      });
    }

    // Generate key
    const timestamp = new Date().toISOString();
    const random = Math.random().toString(16).slice(2, 6);
    const id = `${timestamp}_${random}`;

    const store = getStore('feedback');

    // Build feedback item
    const feedbackItem = {
      id,
      type,
      description,
      email: email || undefined,
      status: 'new',
      hasScreenshot: !!screenshot,
      submittedAt: timestamp,
      userAgent: req.headers.get('user-agent') || 'unknown'
    };

    // Store feedback item as JSON
    await store.setJSON(`items/${id}`, feedbackItem, {
      metadata: {
        type,
        status: 'new',
        hasScreenshot: screenshot ? 'true' : 'false',
        submittedAt: timestamp
      }
    });

    // Store screenshot separately if provided
    if (screenshot && typeof screenshot === 'string') {
      await store.set(`screenshots/${id}`, screenshot);
    }

    return new Response(JSON.stringify({ success: true, id }), {
      status: 200,
      headers: { 'Content-Type': 'application/json' }
    });
  } catch (error: any) {
    console.error('feedback-submit error:', error);
    return new Response(JSON.stringify({ error: 'Internal server error' }), {
      status: 500,
      headers: { 'Content-Type': 'application/json' }
    });
  }
};
```

**Query function (`feedback-query.mts`):**

```typescript
import { getStore } from '@netlify/blobs';
import type { Context } from '@netlify/functions';

export default async (req: Request, context: Context) => {
  if (req.method !== 'GET') {
    return new Response(JSON.stringify({ error: 'Method not allowed' }), {
      status: 405,
      headers: { 'Content-Type': 'application/json' }
    });
  }

  try {
    // Auth check -- SAME pattern as analytics-query.mts
    const analyticsPassword = process.env.ANALYTICS_PASSWORD;
    if (!analyticsPassword) {
      return new Response(JSON.stringify({ error: 'Admin not configured' }), {
        status: 503,
        headers: { 'Content-Type': 'application/json' }
      });
    }

    const authHeader = req.headers.get('Authorization');
    if (!authHeader || !authHeader.startsWith('Basic ')) {
      return new Response(JSON.stringify({ error: 'Authentication required' }), {
        status: 401,
        headers: { 'Content-Type': 'application/json', 'WWW-Authenticate': 'Basic realm="Admin"' }
      });
    }

    const credentials = atob(authHeader.slice(6));
    const [, password] = credentials.split(':');
    if (password !== analyticsPassword) {
      return new Response(JSON.stringify({ error: 'Invalid credentials' }), {
        status: 401,
        headers: { 'Content-Type': 'application/json', 'WWW-Authenticate': 'Basic realm="Admin"' }
      });
    }

    // Parse query params
    const url = new URL(req.url);
    const statusFilter = url.searchParams.get('status');    // 'new', 'reviewed', 'resolved', 'archived'
    const typeFilter = url.searchParams.get('type');          // 'bug', 'feature'
    const itemId = url.searchParams.get('id');               // Specific item (for detail view)
    const includeScreenshot = url.searchParams.get('screenshot') === 'true';

    const store = getStore('feedback');

    // Detail view: fetch single item with optional screenshot
    if (itemId) {
      const item = await store.get(`items/${itemId}`, { type: 'json' });
      if (!item) {
        return new Response(JSON.stringify({ error: 'Not found' }), { status: 404, headers: { 'Content-Type': 'application/json' } });
      }

      let screenshot = undefined;
      if (includeScreenshot && item.hasScreenshot) {
        screenshot = await store.get(`screenshots/${itemId}`, { type: 'text' });
      }

      return new Response(JSON.stringify({ item, screenshot }), {
        status: 200,
        headers: { 'Content-Type': 'application/json' }
      });
    }

    // List view: get all feedback items
    const result = await store.list({ prefix: 'items/' });

    // Fetch all items in parallel
    const items = await Promise.all(
      result.blobs.map(blob => store.get(blob.key, { type: 'json' }))
    );

    // Filter (client-side since Blobs doesn't support query filters)
    let filtered = items.filter(Boolean);
    if (statusFilter) {
      filtered = filtered.filter((item: any) => item.status === statusFilter);
    }
    if (typeFilter) {
      filtered = filtered.filter((item: any) => item.type === typeFilter);
    }

    // Sort by submittedAt descending (newest first)
    filtered.sort((a: any, b: any) => b.submittedAt.localeCompare(a.submittedAt));

    return new Response(JSON.stringify({
      items: filtered,
      total: filtered.length,
      generatedAt: new Date().toISOString()
    }), {
      status: 200,
      headers: { 'Content-Type': 'application/json' }
    });
  } catch (error: any) {
    console.error('feedback-query error:', error);
    return new Response(JSON.stringify({ error: 'Internal server error' }), {
      status: 500,
      headers: { 'Content-Type': 'application/json' }
    });
  }
};
```

### Anti-Patterns to Avoid

- **Storing screenshot data inline with feedback JSON:** Screenshots as base64 are 100KB-500KB. Listing 50 items would download 5-25MB of data. Always store screenshots as separate blobs.
- **Using a single aggregate blob for all feedback:** Unlike analytics (where daily aggregation makes sense because you always view totals), feedback items are individual records. One blob per item allows individual CRUD operations.
- **Fetching all blobs when you only need metadata:** For a list view showing type/status badges, `getMetadata` on each blob would be cheaper than `get` with full JSON. However, since feedback items are small (< 1KB each without screenshots), and the total count is expected to be low (dozens, not thousands), fetching full JSON is acceptable and simpler.
- **Creating a new env var for feedback admin password:** Reuse `ANALYTICS_PASSWORD`. One admin password for all admin pages reduces configuration burden.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Blob key generation | UUID library or complex ID scheme | ISO timestamp + 4-char hex | Simple, sortable, no dependencies, collision-resistant enough for low-volume feedback |
| Auth middleware | Custom session/JWT system | Basic Auth with ANALYTICS_PASSWORD (same as analytics) | Already proven in analytics-query.mts, password held in memory, re-prompt on refresh |
| Screenshot optimization | Server-side image processing | Client-side JPEG compression (already in FeedbackModal) | html2canvas already compresses to 0.7 quality, further to 0.3 if >500KB; server doesn't need to process |
| Pagination | Complex cursor-based pagination | Load all items (expected <200 total) | For low-volume feedback, loading all items is fine; Blobs list() returns all with prefix; add pagination only if volume exceeds expectations |
| Search/filtering | Server-side query engine | Client-side array filtering | Feedback items are small JSON; filtering 100 items in the browser is instant |
| Spam protection | Server-side ML/Akismet | Honeypot field check in submit function | The honeypot pattern (`botField` non-empty = bot) catches simple bots; for a low-traffic admin tool, this is sufficient |

**Key insight:** The analytics system already solved all the hard problems (auth, Blobs storage, admin page layout). This implementation is almost entirely copy-adapt from existing code.

## Common Pitfalls

### Pitfall 1: Screenshot Data Exceeding Useful Limits

**What goes wrong:** Base64 screenshots stored as text blobs can be very large (500KB-2MB), making the feedback store consume significant space and slowing down screenshot retrieval.

**Why it happens:** html2canvas captures the full viewport at device pixel ratio, producing large canvases. Even with JPEG compression, base64 encoding adds ~33% overhead.

**How to avoid:**
1. The existing FeedbackModal already compresses to JPEG 0.7 quality and further to 0.3 if >500KB -- keep this
2. Store screenshots as separate blobs (never inline with feedback JSON)
3. Only fetch screenshot data when admin explicitly clicks to view a specific submission
4. Consider adding a max size check in the submit function (reject screenshots > 1MB)

**Warning signs:** Feedback query endpoint becomes slow; Blobs storage usage grows unexpectedly.

### Pitfall 2: Listing Performance with Many Items

**What goes wrong:** `store.list({ prefix: 'items/' })` returns all items, then `Promise.all` fetches each one. With 500+ items, this creates hundreds of parallel requests.

**Why it happens:** Netlify Blobs `list()` only returns keys and metadata, not blob content. You must fetch each blob individually.

**How to avoid:**
1. For the initial implementation (expected <200 items), this is fine
2. If volume grows, use `list()` with the `paginate` option and fetch only the current page of items
3. Store lightweight summary data in blob metadata (type, status, submittedAt) and use `getMetadata` for list views instead of fetching full JSON
4. Alternatively, maintain a single "index" blob that contains an array of all item summaries, updated on each submit/status-change

**Warning signs:** Admin page takes >5 seconds to load; Netlify function timeouts.

### Pitfall 3: Forgetting to Update Both JSON and Metadata

**What goes wrong:** Status update changes the JSON blob but forgets to update the Blobs metadata, causing inconsistency between what `getMetadata` returns and what the actual data contains.

**Why it happens:** Metadata and blob content are set independently. When updating via `setJSON`, you must also pass the updated metadata.

**How to avoid:** Always pass the `metadata` option when calling `setJSON` during updates. Create a helper function that builds metadata from the feedback item to ensure consistency.

```typescript
function feedbackMetadata(item: FeedbackItem): Record<string, string> {
  return {
    type: item.type,
    status: item.status,
    hasScreenshot: item.hasScreenshot ? 'true' : 'false',
    submittedAt: item.submittedAt
  };
}
```

### Pitfall 4: Service Worker and Feedback Queue Still Targeting Old Endpoint

**What goes wrong:** The offline queue in `feedback-queue.ts` and `service-worker.js` still sends to `/feedback` with `form-name=feedback` URLSearchParams format. The new Blobs function expects JSON at `/.netlify/functions/feedback-submit`.

**Why it happens:** Three separate files need updating: `FeedbackModal.svelte` (online path), `feedback-queue.ts` (offline sync path), and `service-worker.js` (background sync path). Missing any one leaves a broken code path.

**How to avoid:** Update ALL three files together:
1. `FeedbackModal.svelte`: Change `fetch('/feedback-form.html', { body: formData })` to `fetch('/.netlify/functions/feedback-submit', { body: JSON.stringify(...) })`
2. `feedback-queue.ts` `syncQueuedFeedback()`: Change URLSearchParams POST to JSON POST
3. `service-worker.js` `syncFeedbackQueue()`: Change URLSearchParams POST to JSON POST

**Warning signs:** Online submissions work but offline-queued submissions fail when syncing.

### Pitfall 5: Blobs Metadata Values Must Be Strings

**What goes wrong:** Passing `{ hasScreenshot: true }` (boolean) or `{ retryCount: 3 }` (number) as metadata causes silent failures or type errors.

**Why it happens:** Netlify Blobs metadata is typed as `Record<string, string>`. All values must be strings.

**How to avoid:** Always stringify non-string metadata values: `hasScreenshot: 'true'`, not `hasScreenshot: true`.

### Admin Page Data Fetching

Following the exact pattern from `src/routes/analytics/+page.svelte`:

```typescript
async function fetchFeedback() {
  loading = true;
  error = '';

  try {
    const params = new URLSearchParams();
    if (statusFilter) params.set('status', statusFilter);
    if (typeFilter) params.set('type', typeFilter);

    const response = await fetch(
      `/.netlify/functions/feedback-query?${params.toString()}`,
      {
        headers: {
          Authorization: `Basic ${btoa(`admin:${password}`)}`
        }
      }
    );

    if (response.status === 401) {
      error = 'Invalid password';
      loading = false;
      return;
    }

    if (!response.ok) {
      error = 'Failed to load feedback data';
      loading = false;
      return;
    }

    const data = await response.json();
    items = data.items;
    authenticated = true;
  } catch (err) {
    error = 'Network error. Please try again.';
  } finally {
    loading = false;
  }
}
```

## Admin Page Route Decision

The analytics page lives at `/analytics` (not `/admin/analytics`). Two options for the feedback admin page:

**Option A: `/admin/feedback`** -- Creates a new `/admin/` prefix for admin pages
- Pro: Cleaner URL structure going forward
- Con: Inconsistent with `/analytics` which is already at root level

**Option B: `/feedback-admin`** -- Flat route like analytics
- Pro: Consistent with existing `/analytics` pattern
- Con: Less organized long-term

**Recommendation:** Use `/admin/feedback` and consider moving `/analytics` to `/admin/analytics` in a future cleanup. The `/admin/` prefix is the more conventional pattern and provides a natural grouping.

However, since the analytics page already exists at `/analytics` and this is a small app, using `/admin/feedback` is fine even with the inconsistency. The admin will bookmark whichever URLs they use.

## Open Questions

1. **Feedback data retention policy**
   - What we know: Analytics has a 13-month retention with `cleanupOldData()`. Feedback may want different retention.
   - What's unclear: How long to keep feedback items, especially resolved ones.
   - Recommendation: Implement a `cleanupOldFeedback()` that deletes items with status `archived` older than 6 months, run opportunistically like analytics cleanup. Keep non-archived items indefinitely (low volume expected).

2. **Rate limiting on the submit endpoint**
   - What we know: The analytics-track endpoint has no explicit rate limiting. The feedback-submit endpoint is public (no auth).
   - What's unclear: Whether Netlify's built-in function rate limiting is sufficient, or if we need application-level limiting.
   - Recommendation: Start without rate limiting (honeypot handles bots). If abuse is detected, add a simple IP-based rate limit using a Blobs counter key (e.g., `ratelimit/{ip-hash}/{date}` with a max of 10 submissions per day per IP). This can be added later without schema changes.

3. **Whether to use `setJSON` vs `set` with `JSON.stringify`**
   - What we know: The analytics system uses `store.set(key, JSON.stringify(aggregate))`. The `setJSON` method is a convenience that does the same thing internally.
   - What's unclear: Whether `setJSON` has any behavioral differences (content-type handling, etc.).
   - Recommendation: Use `setJSON` for the feedback system -- it is the more modern API and sets the correct content-type automatically. It pairs with `store.get(key, { type: 'json' })` for deserialization.
