# Phase 08.8: User Feedback & Bug Reports - Research

**Researched:** 2026-02-01
**Domain:** In-app feedback collection for web applications
**Confidence:** HIGH

## Summary

Research investigated implementation patterns for collecting user feedback and bug reports in offline-first PWAs without user authentication. The domain has three primary approaches: (1) hosted SaaS services (Canny, FeatureBase, Frill), (2) self-hosted open source solutions (Astuto, Fider), and (3) simple form submissions with backend notification (Netlify Forms, custom endpoints).

For a privacy-focused, offline-first PWA already deployed on Netlify with no user accounts, **Netlify Forms provides the simplest path**: built-in spam protection via Akismet + honeypot, free tier for low-volume feedback (100 submissions/month on legacy plans, credit-based on new plans), zero infrastructure to maintain, and native SvelteKit integration via prerendered routes. The primary limitation is submissions require network connectivity - offline queuing must be implemented separately using IndexedDB + Background Sync API.

The standard pattern pairs a lightweight modal dialog (triggered by FAB or menu item) with SvelteKit's form actions for progressive enhancement. Screenshot capture using FeedbackPlus adds visual context for bug reports. Submissions categorize as "bug report" or "feature request" with optional email for follow-up.

**Primary recommendation:** Use Netlify Forms with SvelteKit form actions, add screenshot capture with FeedbackPlus, implement offline queue with IndexedDB + Background Sync, and categorize submissions with hidden form fields.

## Standard Stack

The established libraries/tools for this domain:

### Core

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|

| FeedbackPlus | 1.3+ | Screenshot capture and editing for feedback forms | Lightweight, uses native browser APIs with html2canvas fallback, visual bug reporting |

### Pattern 1: Netlify Forms with SvelteKit Prerendering

**What:** Netlify Forms require static HTML for form detection during build. SvelteKit routes must be prerendered to generate static output.

**When to use:** Primary pattern for simple feedback collection on Netlify-hosted SvelteKit apps.

**Example:**
```svelte
<!-- src/routes/feedback/+page.svelte -->
<script>
  export const prerender = true; // CRITICAL: Must prerender for Netlify
  import { enhance } from '$app/forms';
</script>

<form
  method="POST"
  name="feedback"
  data-netlify="true"
  netlify-honeypot="bot-field"
  use:enhance
>
  <!-- Hidden fields for Netlify -->
  <input type="hidden" name="form-name" value="feedback" />

  <!-- Honeypot for spam protection -->
  <p class="hidden">
    <label>
      Don't fill this out: <input name="bot-field" type="text" />
    </label>
  </p>

  <!-- Feedback type categorization -->
  <label>
    Type:
    <select name="type" required>
      <option value="bug">Bug Report</option>
      <option value="feature">Feature Request</option>
    </select>
  </label>

  <!-- Description (required) -->
  <label>
    Description:
    <textarea name="description" required minlength="10"></textarea>
  </label>

  <!-- Email (optional for follow-up) -->
  <label>
    Email (optional - only if you'd like us to follow up):
    <input type="email" name="email" />
  </label>

  <!-- Screenshot (optional, added via FeedbackPlus) -->
  <input type="hidden" name="screenshot" id="screenshot-data" />

  <button type="submit">Submit Feedback</button>
</form>
```

### Pattern 2: Progressive Enhancement with use:enhance

**What:** SvelteKit's `use:enhance` directive enables AJAX submission with JavaScript while maintaining fallback to standard POST without JavaScript.

**When to use:** All forms in SvelteKit for better UX - prevents page refresh, preserves scroll position, enables loading states.

**Example:**
```svelte
<script>
  import { enhance } from '$app/forms';

  let submitting = false;
  let submitted = false;
</script>

<form
  method="POST"
  name="feedback"
  data-netlify="true"
  use:enhance={() => {
    submitting = true;

    return async ({ result, update }) => {
      submitting = false;
      if (result.type === 'success') {
        submitted = true;
      }
      await update();
    };
  }}
>
  <!-- form fields -->

  <button type="submit" disabled={submitting}>
    {submitting ? 'Submitting...' : 'Submit Feedback'}
  </button>
</form>

{#if submitted}
  <p class="success">Thank you! We've received your feedback.</p>
{/if}
```

**Source:** SvelteKit official docs - [Form actions](https://svelte.dev/docs/kit/form-actions)

### Pattern 3: Screenshot Capture with FeedbackPlus

**What:** Add screenshot capture and editing to feedback forms using FeedbackPlus library. Uses browser Display Capture API with html2canvas fallback.

**When to use:** Bug reports where visual context is valuable.

**Example:**
```svelte
<script>
  import { onMount } from 'svelte';
  import FeedbackPlus from 'feedbackplus';

  let screenshotData = '';

  onMount(() => {
    // Load html2canvas for better browser support
    import('html2canvas').then(() => {
      // Library now available for FeedbackPlus fallback
    });
  });

  async function captureScreenshot() {
    const feedback = new FeedbackPlus();

    try {
      // Capture screenshot
      const dataUrl = await feedback.capture();

      // Show edit dialog (optional)
      const edited = await feedback.showEditDialog(dataUrl);

      // Store in hidden field
      screenshotData = edited || dataUrl;
      document.getElementById('screenshot-data').value = screenshotData;
    } catch (error) {
      console.error('Screenshot capture failed:', error);
    }
  }
</script>

<button type="button" on:click={captureScreenshot}>
  Add Screenshot
</button>

{#if screenshotData}
  <img src={screenshotData} alt="Screenshot preview" />
{/if}

<input type="hidden" name="screenshot" id="screenshot-data" />
```

### Pattern 4: Offline Queue with IndexedDB + Background Sync

**What:** Queue feedback submissions in IndexedDB when offline, automatically sync when network returns using Background Sync API.

**When to use:** Offline-first PWAs where users must be able to submit feedback without network connectivity.

**Example:**
```typescript
// src/lib/db/feedback-queue.ts
import { db } from './db'; // Existing Dexie instance

export interface QueuedFeedback {
  id?: number;
  type: 'bug' | 'feature';
  description: string;
  email?: string;
  screenshot?: string;
  timestamp: number;
  retryCount: number;
}

export async function queueFeedback(feedback: Omit<QueuedFeedback, 'id' | 'timestamp' | 'retryCount'>) {
  return db.feedbackQueue.add({
    ...feedback,
    timestamp: Date.now(),
    retryCount: 0
  });
}

export async function getQueuedFeedback(): Promise<QueuedFeedback[]> {
  return db.feedbackQueue.toArray();
}

export async function removeFromQueue(id: number) {
  return db.feedbackQueue.delete(id);
}

// src/lib/sync/feedback-sync.ts
export async function registerFeedbackSync() {
  if ('serviceWorker' in navigator && 'SyncManager' in window) {
    const registration = await navigator.serviceWorker.ready;
    await registration.sync.register('sync-feedback');
  }
}

export async function syncQueuedFeedback() {
  const queued = await getQueuedFeedback();

  for (const item of queued) {
    try {
      const formData = new FormData();
      formData.append('form-name', 'feedback');
      formData.append('type', item.type);
      formData.append('description', item.description);
      if (item.email) formData.append('email', item.email);
      if (item.screenshot) formData.append('screenshot', item.screenshot);

      const response = await fetch('/feedback', {
        method: 'POST',
        body: formData
      });

      if (response.ok) {
        await removeFromQueue(item.id!);
      } else {
        // Increment retry count
        await db.feedbackQueue.update(item.id!, {
          retryCount: item.retryCount + 1
        });
      }
    } catch (error) {
      console.error('Sync failed for feedback:', item.id, error);
    }
  }
}
```

```javascript
// src/service-worker.js
self.addEventListener('sync', (event) => {
  if (event.tag === 'sync-feedback') {
    event.waitUntil(
      // Import and call sync function
      import('./lib/sync/feedback-sync.js').then(module =>
        module.syncQueuedFeedback()
      )
    );
  }
});
```

### Anti-Patterns to Avoid

- **Requiring email address:** Makes friction too high, reduces submissions. Email should be optional with clear justification ("only if you'd like us to follow up").
- **Long forms with many fields:** Research shows 5-8 questions maximum. Description + type + optional email is sufficient.
- **Blocking modals that interrupt workflow:** Users will close without reading. Use non-blocking modal that's easy to dismiss.
- **Only marking optional fields with asterisks:** Accessibility research shows this confuses users. Use text "(optional)" instead.
- **No keyboard navigation:** FAB and modal must be fully keyboard accessible with visible focus indicators (3:1 contrast minimum).
- **Using Netlify Forms with dynamic routes:** Forms MUST be prerendered - Netlify's build-time crawler cannot detect forms in dynamic routes.
- **Combining use:enhance with Netlify Forms action attribute:** use:enhance expects SvelteKit action responses. For Netlify Forms, let it submit naturally or handle via custom endpoint.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Spam protection | Custom bot detection logic | Netlify Forms (Akismet + honeypot) or Cloudflare Turnstile | Spam bots evolve rapidly. Akismet has massive training dataset. Honeypots catch simple bots but miss sophisticated ones. reCAPTCHA has accessibility issues - Cloudflare Turnstile is better. |
| Screenshot capture | Custom Canvas API implementation | FeedbackPlus with html2canvas fallback | Cross-browser screenshot APIs are inconsistent. html2canvas handles edge cases (iframes, CORS, webfonts). Screenshot editing UI is complex (drawing tools, redaction). |
| Offline form queue | Custom IndexedDB sync logic | Background Sync API + Dexie | Background Sync handles network transitions automatically, retries failed requests, manages battery/data constraints. Manual implementation misses edge cases (app closed, multiple tabs, partial failures). |
| Email notifications | Custom SMTP implementation in Netlify Functions | Netlify Forms notifications or SendGrid/Resend | SMTP configuration is fragile (ports, TLS, authentication). Deliverability requires SPF/DKIM/DMARC setup. Transactional email services handle bounce/retry logic and maintain reputation. |
| Form validation | Custom validation logic | HTML5 required/minlength/pattern + SvelteKit validation | HTML5 provides client-side validation with native browser UI. SvelteKit fail() function returns field-specific errors. Custom validation duplicates browser features and misses accessibility (aria-invalid, error announcements). |

**Key insight:** Feedback collection has hidden complexity in spam prevention, cross-browser compatibility, offline reliability, and accessibility. Established libraries solve edge cases discovered through years of production use. For a simple feedback form, the full stack (Netlify Forms + FeedbackPlus + Background Sync API) requires ~50 lines of code vs. 500+ for custom implementation.

## Common Pitfalls

### Pitfall 1: Netlify Forms Not Detected During Build

**What goes wrong:** Form submissions return 404 or "Form not found" errors even with correct data-netlify attribute.

**Why it happens:** Netlify's build process crawls static HTML output to detect forms. SvelteKit routes without `export const prerender = true` generate pages dynamically at runtime, so Netlify's crawler never sees the form HTML.

**How to avoid:**
1. Add `export const prerender = true` to the `+page.svelte` or `+page.ts` file containing the form
2. Verify form appears in static build output (`.svelte-kit/output/prerendered/pages/`)
3. Check Netlify build logs for "X form(s) detected" message

**Warning signs:**
- Form submissions fail only after deployment (work locally with SvelteKit dev server)
- Netlify admin panel shows 0 forms detected
- Build output doesn't include static HTML file for form route

**Source:** Netlify blog - [Easy HTML Forms in SvelteKit with Netlify Forms](https://www.netlify.com/blog/easy-html-forms-in-sveltekit-with-netlify-forms/)

### Pitfall 2: use:enhance Incompatible with Netlify Forms Default Behavior

**What goes wrong:** Form appears to submit successfully but no data reaches Netlify. Or form redirects to success page without client-side feedback.

**Why it happens:** `use:enhance` intercepts form submission and expects SvelteKit action response format. Netlify Forms submit to Netlify's servers (not SvelteKit actions) and redirect to `/` or a success page. The two mechanisms conflict.

**How to avoid:**
1. **Option A:** Don't use `use:enhance` on Netlify Forms - let them submit naturally with full page reload
2. **Option B:** Use custom `use:enhance` callback that POSTs FormData to Netlify endpoint, then shows success message client-side without expecting SvelteKit action response
3. **Option C:** Submit to SvelteKit action that forwards to Netlify Forms via fetch (hybrid approach)

**Warning signs:**
- Console shows network error when form submits
- Form state doesn't update after submission
- Submissions don't appear in Netlify admin panel

**Sources:**
- GitHub issue - [use:enhance with Netlify Forms not working?](https://github.com/sveltejs/kit/discussions/9678)
- SvelteKit docs - [Customizing use:enhance](https://svelte.dev/tutorial/kit/customizing-use-enhance)

### Pitfall 3: Screenshot Data Exceeds Form Submission Size Limits

**What goes wrong:** Form submission fails silently or with cryptic error when screenshot is included. Large screenshots cause timeouts.

**Why it happens:** Screenshots as base64 data URLs are typically 500KB-2MB. Netlify Forms has 10MB limit per submission, but large payloads increase latency and risk timeouts. Some platforms have smaller limits.

**How to avoid:**
1. Compress screenshots before submission using Canvas API with reduced quality
2. Resize large screenshots (max 1920px width recommended)
3. Store screenshots in Netlify Blob or external storage, submit URL instead of data
4. Show file size warning if screenshot exceeds threshold (e.g., 1MB)

**Warning signs:**
- Form submits successfully for text-only feedback but fails with screenshots
- Network tab shows submission taking 5+ seconds
- Intermittent failures on slower connections

**Code example:**
```typescript
async function compressScreenshot(dataUrl: string, maxSizeKB: number = 500): Promise<string> {
  const img = new Image();
  img.src = dataUrl;
  await img.decode();

  const canvas = document.createElement('canvas');
  const MAX_WIDTH = 1920;
  const scale = Math.min(1, MAX_WIDTH / img.width);

  canvas.width = img.width * scale;
  canvas.height = img.height * scale;

  const ctx = canvas.getContext('2d')!;
  ctx.drawImage(img, 0, 0, canvas.width, canvas.height);

  // Try reducing quality until under size limit
  for (let quality = 0.9; quality > 0.1; quality -= 0.1) {
    const compressed = canvas.toDataURL('image/jpeg', quality);
    const sizeKB = compressed.length / 1024;
    if (sizeKB < maxSizeKB) return compressed;
  }

  return canvas.toDataURL('image/jpeg', 0.1);
}
```

### Pitfall 4: Accessibility - Keyboard Trap in Modal

**What goes wrong:** Keyboard users can tab out of feedback modal to underlying page content. Or can't close modal with Escape key. Or focus indicator invisible against background.

**Why it happens:** Modal dialogs require explicit focus management that's not automatic in HTML. Without focus trap, Tab key cycles through all page elements. Without keyboard handlers, Escape key does nothing.

**How to avoid:**
1. Trap focus within modal - when Tab reaches last focusable element, wrap to first
2. Handle Escape key to close modal
3. Return focus to trigger button when modal closes
4. Ensure focus indicators have 3:1 contrast minimum (WCAG 2.1.1)
5. Add `role="dialog"` and `aria-labelledby` for screen readers

**Warning signs:**
- Manual keyboard testing - tab moves to page content behind modal
- Escape key doesn't close modal
- Screen reader announces incorrect modal purpose
- Focus indicator hard to see on dark backgrounds

**Code example:**
```svelte
<script>
  import { onMount } from 'svelte';

  let modalElement: HTMLElement;
  let previouslyFocused: HTMLElement;

  onMount(() => {
    previouslyFocused = document.activeElement as HTMLElement;

    // Focus first focusable element
    const firstFocusable = modalElement.querySelector('input, button, textarea, select');
    (firstFocusable as HTMLElement)?.focus();

    return () => {
      // Restore focus on unmount
      previouslyFocused?.focus();
    };
  });

  function handleKeydown(event: KeyboardEvent) {
    if (event.key === 'Escape') {
      close();
    }

    if (event.key === 'Tab') {
      const focusable = modalElement.querySelectorAll(
        'input, button, textarea, select, [tabindex]:not([tabindex="-1"])'
      );
      const first = focusable[0] as HTMLElement;
      const last = focusable[focusable.length - 1] as HTMLElement;

      if (event.shiftKey && document.activeElement === first) {
        event.preventDefault();
        last.focus();
      } else if (!event.shiftKey && document.activeElement === last) {
        event.preventDefault();
        first.focus();
      }
    }
  }
</script>

<div
  bind:this={modalElement}
  role="dialog"
  aria-labelledby="feedback-title"
  aria-modal="true"
  on:keydown={handleKeydown}
>
  <h2 id="feedback-title">Send Feedback</h2>
  <!-- form content -->
</div>
```

### Pitfall 5: Ignoring Spam Protection Until After Launch

**What goes wrong:** Feedback form gets overwhelmed with spam submissions immediately after launch. Team spends hours manually deleting spam instead of reading real feedback.

**Why it happens:** Bots aggressively crawl new deployments looking for unprotected forms. Even simple bots fill out all fields. Without protection, 80%+ of submissions are spam within days of launch.

**How to avoid:**
1. Implement honeypot field from day one (zero UX impact, catches simple bots)
2. Enable Netlify's built-in Akismet filtering (automatic, no configuration)
3. Consider adding Cloudflare Turnstile for high-value forms (better UX than reCAPTCHA)
4. Monitor spam vs. verified submission ratio in Netlify admin panel

**Warning signs:**
- Submissions with gibberish text or links in description field
- Same email/description repeated multiple times
- Submissions at regular intervals (every 5 minutes)
- Forms submitted instantly after page load

**Source:** Netlify docs - [Spam filters](https://docs.netlify.com/manage/forms/spam-filters/)

### Pitfall 6: Making Email Required Kills Submission Rate

**What goes wrong:** Feedback form completion rate drops by 50-70% when email is required field. Users abandon form rather than provide email.

**Why it happens:** Email is perceived as "high friction" - users fear spam, don't want to be contacted, or simply don't want to share. Privacy-conscious users especially resistant. For simple feedback, requiring email feels disproportionate.

**How to avoid:**
1. Make email optional with clear, compelling justification: "(optional - only if you'd like us to follow up)"
2. Position email field last, after user has invested effort in description
3. Never use email for marketing without explicit opt-in
4. Consider collecting just type + description as minimum viable feedback

**Warning signs:**
- High form abandonment rate (analytics show many people start but don't submit)
- Users entering fake emails (test@test.com, asdf@asdf.com)
- Comments like "why do you need my email for a bug report?"

**Source:** UX Tigers - [Required Fields in Forms: Best Design Practices](https://www.uxtigers.com/post/required-fields)

### Complete Feedback Modal Component

```svelte
<!-- src/lib/components/FeedbackModal.svelte -->
<script lang="ts">
  import { createEventDispatcher, onMount } from 'svelte';
  import { enhance } from '$app/forms';
  import FeedbackPlus from 'feedbackplus';
  import { queueFeedback, registerFeedbackSync } from '$lib/sync/feedback-sync';

  const dispatch = createEventDispatcher();

  let modalElement: HTMLElement;
  let previouslyFocused: HTMLElement;
  let submitting = false;
  let submitted = false;
  let screenshotData = '';
  let isOnline = navigator.onLine;

  onMount(() => {
    previouslyFocused = document.activeElement as HTMLElement;
    modalElement.querySelector('input, button, textarea, select')?.focus();

    // Load html2canvas for better browser support
    import('html2canvas');

    // Listen for online/offline
    const handleOnline = () => isOnline = true;
    const handleOffline = () => isOnline = false;
    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);

    return () => {
      previouslyFocused?.focus();
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  });

  async function captureScreenshot() {
    const feedback = new FeedbackPlus();
    try {
      const dataUrl = await feedback.capture();
      const edited = await feedback.showEditDialog(dataUrl);
      screenshotData = edited || dataUrl;

      // Compress if too large
      if (screenshotData.length > 500 * 1024) {
        screenshotData = await compressScreenshot(screenshotData);
      }
    } catch (error) {
      console.error('Screenshot capture failed:', error);
    }
  }

  async function compressScreenshot(dataUrl: string): Promise<string> {
    const img = new Image();
    img.src = dataUrl;
    await img.decode();

    const canvas = document.createElement('canvas');
    const MAX_WIDTH = 1920;
    const scale = Math.min(1, MAX_WIDTH / img.width);

    canvas.width = img.width * scale;
    canvas.height = img.height * scale;

    const ctx = canvas.getContext('2d')!;
    ctx.drawImage(img, 0, 0, canvas.width, canvas.height);

    return canvas.toDataURL('image/jpeg', 0.8);
  }

  function handleKeydown(event: KeyboardEvent) {
    if (event.key === 'Escape') {
      close();
    }

    if (event.key === 'Tab') {
      const focusable = modalElement.querySelectorAll(
        'input:not([type="hidden"]), button, textarea, select, [tabindex]:not([tabindex="-1"])'
      );
      const first = focusable[0] as HTMLElement;
      const last = focusable[focusable.length - 1] as HTMLElement;

      if (event.shiftKey && document.activeElement === first) {
        event.preventDefault();
        last.focus();
      } else if (!event.shiftKey && document.activeElement === last) {
        event.preventDefault();
        first.focus();
      }
    }
  }

  function close() {
    dispatch('close');
  }

  async function handleOfflineSubmit(event: SubmitEvent) {
    const formData = new FormData(event.currentTarget as HTMLFormElement);

    await queueFeedback({
      type: formData.get('type') as 'bug' | 'feature',
      description: formData.get('description') as string,
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| mailto: links for feedback | In-app forms with backend collection | ~2015 | Email clients inconsistent, can't attach screenshots, no spam protection, data not centralized |
| Custom AJAX to PHP/Node backend | Native framework form actions (SvelteKit, Next.js, Remix) | 2020-2023 | Progressive enhancement, type-safe validation, automatic revalidation, better DX |
| jQuery plugins for screenshot | Native browser APIs + lightweight libraries | 2020+ | Modern browsers support display capture natively, Canvas API more reliable, no jQuery dependency |
| reCAPTCHA v2 "I'm not a robot" | Cloudflare Turnstile or invisible CAPTCHA | 2022-2024 | Better UX (no checkbox clicking), better accessibility, privacy-friendly (no Google tracking) |
| Manual spam filtering | AI-powered filters (Akismet, built-in platform tools) | Ongoing | Akismet trained on billions of spam samples, catches sophisticated attacks, reduces manual moderation |
| Separate feedback portal on subdomain | In-app lightweight modal | 2018+ | Lower friction, contextual (can capture app state), doesn't break workflow, higher completion rate |

**Deprecated/outdated:**
- **Embedded full-featured feedback portals (UserVoice widget):** Too heavy, slow page load, poor mobile UX. Replaced by lightweight modals for submission + separate public roadmap page.
- **Flash-based screenshot tools:** Dead with Flash EOL in 2020. Replaced by Canvas API and native display capture.
- **Form to email (mail() function in PHP):** Unreliable delivery, no spam protection, emails land in spam folder. Replaced by transactional email services (SendGrid, Resend, Postmark).
- **Manual database + admin panel:** High maintenance, security vulnerabilities, scalability issues. Replaced by form platforms (Netlify Forms, Formspree) or headless CMS.

## Open Questions

Things that couldn't be fully resolved:

1. **Netlify Forms submission limits on new credit-based plans**
   - What we know: New accounts after Sept 2025 use credit-based pricing (1 credit per submission). Free tier has hard limits that pause all sites when exceeded.
   - What's unclear: Exact free tier credit allocation, whether spam submissions consume credits (documentation says "verified submissions" but unclear if "verified" means "not auto-flagged as spam" or "manually verified by user").
   - Recommendation: Check current plan in Netlify admin panel. For low-volume feedback (estimated <50 submissions/month), free tier should be sufficient. Monitor credit usage first month after launch. If free tier insufficient, consider custom Netlify Function endpoint with Dexie Cloud or Astuto self-hosted as alternatives.

2. **Background Sync API browser support and reliability**
   - What we know: Background Sync supported in Chrome/Edge (90%+ market share), not in Safari/Firefox. Service worker retries when network returns.
   - What's unclear: Behavior when app closed vs. minimized, retry limits/backoff strategy, battery/data restrictions on mobile.
   - Recommendation: Implement as progressive enhancement. Show clear UI for queued items. Add manual "retry now" button in settings for users on unsupported browsers. Test thoroughly on iOS Safari (largest unsupported browser).

3. **Screenshot storage and retention**
   - What we know: Screenshots stored as base64 in Netlify Forms submissions. Netlify Forms retains submissions indefinitely on paid plans, unclear on free tier.
   - What's unclear: Storage limits for large screenshots, whether Netlify compresses/optimizes images, retention policy for spam submissions, ability to bulk export/delete.
   - Recommendation: Compress screenshots to <500KB before submission. For high-volume feedback, consider storing screenshots in Netlify Blob storage and submitting URL instead of base64 data. Document retention policy in privacy policy if collecting email addresses.

4. **Accessibility of FeedbackPlus screenshot editor**
   - What we know: FeedbackPlus provides screenshot editing UI with drawing tools and redaction. Not mentioned in documentation whether keyboard accessible.
   - What's unclear: Can users navigate edit tools with keyboard? Are color contrast ratios sufficient? Do screen readers announce tool state?
   - Recommendation: Make screenshot optional (not required for bug reports). Provide text field for users who can't use screenshot tools. Manual accessibility testing required before launch. Consider this technical debt to address in future if users report issues.

5. **GDPR compliance for feedback with email addresses**
   - What we know: Collecting email addresses (even optional) makes feedback personally identifiable. GDPR requires consent, right to access, right to deletion.
   - What's unclear: Does Netlify Forms provide GDPR-compliant data access/deletion tools? What's required for legitimate interest vs. consent basis?
   - Recommendation: Add privacy policy link near email field. Document retention period (suggest 1 year for feedback). Netlify provides manual deletion in admin panel. For GDPR compliance, may need custom endpoint with database to support automated data access/deletion requests. Consider whether email is truly necessary - anonymous feedback simpler legally.
