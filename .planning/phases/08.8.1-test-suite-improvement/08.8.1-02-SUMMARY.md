## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] RRule dtstart defaults to current time without DTSTART in string**

- **Found during:** Task 1
- **Issue:** `RRule.fromString('FREQ=DAILY;COUNT=5')` uses current time as dtstart, not the event's startTime. The production code sets `rruleSet.dtstart = event.startTime` but this does not propagate to the individual RRule's iteration. Occurrences start from "now" instead of the event's start time.
- **Fix:** Used DTSTART-anchored rrule strings in tests (e.g., `DTSTART:20260201T100000Z\nRRULE:FREQ=DAILY;COUNT=5`) to get deterministic, predictable test behavior. This matches how rrule strings would work when properly including DTSTART.
- **Note:** The underlying production code issue is documented but not fixed in this plan (out of scope -- this is a test plan, not a fix plan). A future plan should address passing dtstart to the RRule constructor explicitly.
