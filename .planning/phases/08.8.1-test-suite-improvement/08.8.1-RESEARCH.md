# Phase 08.8.1: Test Suite Improvement - Research

**Researched:** 2026-02-02
**Domain:** Vitest browser mode testing for Svelte 5 + SvelteKit 2 applications
**Confidence:** HIGH

## Summary

This research investigates testing strategies for improving test coverage in a SvelteKit 2 + Svelte 5 application using Vitest 6.x with browser mode. The testing infrastructure is already established (Phase 08.6) with real browser IndexedDB, Chromium via Playwright, and vitest-browser-svelte for component rendering. The focus is on testing untested business logic, Svelte 5 $state runes-based stores, Netlify Functions, and components.

The standard approach for testing Svelte 5 runes requires running tests in a real browser environment (not jsdom) because runes only work in browser/component contexts. Vitest's browser mode with Playwright provides real IndexedDB, crypto APIs, localStorage, and sessionStorage without mocking. For stores using $state runes, tests must use `flushSync()` from Svelte to trigger DOM updates after state changes, as locators alone won't wait for external state mutations.

Testing should be prioritized by risk: HIGH risk items (sync orchestration, device pairing) require comprehensive test coverage with edge cases and error scenarios; MEDIUM risk items (recurrence logic, ICS parsing, feedback queue) need solid functional coverage; LOW risk items (analytics, admin functions) can have basic happy-path coverage. Coverage thresholds of 70-85% are realistic for the untested codebase, with critical business logic paths targeted for 90%+.

**Primary recommendation:** Test business logic in isolation from components where possible (sync logic, crypto, merge algorithms, recurrence expansion, ICS parsing), mock external dependencies (fetch, Netlify Blobs) at module level with resets in beforeEach, and use vitest-browser-svelte with flushSync for store and component testing that requires runes reactivity.

### Pattern 1: Testing Isolated Business Logic (Sync, Crypto, Merge)
**What:** Test pure functions and orchestration logic without UI dependencies
**When to use:** High-risk business logic like sync orchestration, cryptography, merge algorithms
**Example:**
```typescript
// src/lib/sync/sync.test.ts
import { describe, it, expect, beforeEach } from 'vitest';
import { exportForSync, importFromSync, computeContentHash } from './sync';
import { db } from '$lib/db/schema';

beforeEach(async () => {
  if (db.isOpen()) db.close();
  await db.delete();
  await db.open();
});

describe('exportForSync', () => {
  it('includes all database tables', async () => {
    // Add test data to multiple tables
    await db.items.add({ id: 'test-1', title: 'Test', type: 'inbox', created: new Date(), modified: new Date() });
    await db.settings.put({ key: 'theme', value: 'dark' });

    const payload = await exportForSync();

    expect(payload.version).toBe(1);
    expect(payload.data.items).toHaveLength(1);
    expect(payload.data.settings).toHaveLength(1);
  });
});

describe('computeContentHash', () => {
  it('produces deterministic hash for same data', async () => {
    const data = {
      items: [{ id: 'a', title: 'Test' }],
      settings: [{ key: 'theme', value: 'dark' }]
    };

    const hash1 = await computeContentHash(data);
    const hash2 = await computeContentHash(data);

    expect(hash1).toBe(hash2);
    expect(hash1).toMatch(/^[0-9a-f]{64}$/);
  });

  it('produces different hash for different data', async () => {
    const data1 = { items: [{ id: 'a', title: 'Test 1' }] };
    const data2 = { items: [{ id: 'a', title: 'Test 2' }] };

    const hash1 = await computeContentHash(data1);
    const hash2 = await computeContentHash(data2);

    expect(hash1).not.toBe(hash2);
  });
});
```

### Pattern 2: Testing Svelte 5 $state Runes Stores
**What:** Test stores using $state runes in browser environment with flushSync
**When to use:** Testing reactive stores that use Svelte 5 runes (sync.svelte.ts, theme.svelte.ts, etc.)
**Example:**
```typescript
// src/lib/stores/sync.svelte.test.ts
// CRITICAL: Filename must include .svelte to use runes in tests
import { describe, it, expect, beforeEach, vi } from 'vitest';
import { flushSync } from 'svelte';
import { syncStore } from './sync.svelte';

// Mock dependencies
vi.mock('$lib/sync/sync', () => ({
  performSync: vi.fn().mockResolvedValue({ success: true }),
  checkForUpdates: vi.fn().mockResolvedValue(false)
}));

vi.mock('$lib/db/schema', () => ({
  db: {
    isOpen: vi.fn(() => false),
    open: vi.fn().mockResolvedValue(undefined)
  }
}));

beforeEach(() => {
  vi.clearAllMocks();
});

describe('syncStore', () => {
  it('initializes with unpaired state', () => {
    expect(syncStore.isPaired).toBe(false);
    expect(syncStore.syncState).toBe('idle');
  });

  it('updates syncState to syncing when sync starts', async () => {
    syncStore.isPaired = true;

    // Start sync (async operation)
    const syncPromise = syncStore.sync();

    // CRITICAL: flushSync() required to trigger DOM/reactive updates
    flushSync(() => {
      // State should update to 'syncing' immediately
      expect(syncStore.syncState).toBe('syncing');
    });

    await syncPromise;

    flushSync(() => {
      expect(syncStore.syncState).toBe('idle');
    });
  });
});
```

### Pattern 5: Testing Async Debounced Functions with Fake Timers
**What:** Test debounced sync operations using vi.useFakeTimers() and microtask flushing
**When to use:** Testing sync debouncing, auto-push delays, polling intervals
**Example:**
```typescript
// src/lib/sync/sync.test.ts
import { describe, it, expect, beforeEach, vi } from 'vitest';
import { syncDebouncer } from './sync';

vi.mock('$lib/sync/crypto', () => ({
  encrypt: vi.fn().mockResolvedValue('encrypted-data')
}));

describe('syncDebouncer', () => {
  beforeEach(() => {
    vi.useFakeTimers();
  });

  afterEach(() => {
    vi.restoreAllMocks();
  });

  it('debounces multiple rapid calls into single sync', async () => {
    const mockSync = vi.fn().mockResolvedValue(undefined);
    const debouncedSync = syncDebouncer(mockSync, 1000);

    // Call 3 times rapidly
    debouncedSync();
    debouncedSync();
    debouncedSync();

    // CRITICAL: Flush microtasks BEFORE advancing timers
    await Promise.resolve();

    // Fast-forward time by 1000ms
    vi.advanceTimersByTime(1000);

    // Flush microtasks again to let promise resolve
    await Promise.resolve();

    // Should have called sync only once
    expect(mockSync).toHaveBeenCalledOnce();
  });
});
```

### Anti-Patterns to Avoid
- **Testing UI instead of business logic:** Extract business logic into pure functions and test those, not component rendering details
- **Not using flushSync with runes:** Tests for $state runes will fail intermittently without flushSync() to trigger updates
- **Mixing Dexie operations with non-IndexedDB async calls in transactions:** Will cause TransactionInactiveError - use Dexie.waitFor() if unavoidable
- **Using fake-indexeddb in browser mode:** Browser mode provides real IndexedDB - only use fake-indexeddb for Node.js Netlify Functions tests
- **Overusing snapshot tests for logic:** Snapshots are for UI consistency, not for asserting business logic correctness
- **Mocking everything:** Real browser APIs (IndexedDB, crypto, localStorage) work natively in browser mode - don't mock them

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Mocking localStorage/sessionStorage | Custom mock object | Real browser APIs (already available in browser mode) | Browser mode runs in Chromium - real APIs work without mocking |
| Mocking fetch for Netlify tests | Custom fetch wrapper | vi.mock() at module level + mockResolvedValue | Module-level mocking prevents imported function from using real fetch |
| Test data builders | One-off factory functions per test | Shared helper with reasonable defaults | Already have pattern in existing tests (mockContext, makeRequest) |
| Coverage thresholds | Manual file-by-file configuration | coverage.include + glob-based thresholds | Vitest v4+ supports per-directory thresholds via glob patterns |
| Testing async operations | setTimeout + callbacks | async/await + Vitest's built-in async support | Vitest natively handles promises, no need for done() callbacks |
| Component query helpers | Repeated querySelector calls | page.getBy*() locators from @vitest/browser | Auto-retry built in, more reliable than manual DOM queries |

**Key insight:** Vitest browser mode eliminates most mocking needs for web APIs. The real browser environment (Chromium) provides IndexedDB, crypto.subtle, localStorage, sessionStorage, fetch, and DOM APIs natively. Only mock external services (Netlify Blobs) and module dependencies, not browser APIs.

## Common Pitfalls

### Pitfall 1: Forgetting flushSync() with Svelte 5 Runes
**What goes wrong:** Tests for stores using $state runes fail intermittently or can't observe state changes
**Why it happens:** Runes batch updates for performance - locators don't automatically wait for external state changes
**How to avoid:** Import `flushSync` from 'svelte' and wrap assertions that depend on reactive state updates
**Warning signs:** Test failures with "element not found" when state should have updated, race conditions in store tests

### Pitfall 2: Awaiting Non-IndexedDB Operations in Dexie Transactions
**What goes wrong:** TransactionInactiveError thrown when mixing fetch/setTimeout with Dexie operations
**Why it happens:** IndexedDB transactions auto-commit when not used within a tick
**How to avoid:** Keep transactions short with only Dexie operations, or use Dexie.waitFor() for unavoidable external calls (but note: CPU intensive)
**Warning signs:** "TransactionInactiveError: The transaction is inactive or finished" in test output

### Pitfall 3: Using fake-indexeddb in Browser Mode
**What goes wrong:** Tests may pass with fake-indexeddb but fail in production with real IndexedDB
**Why it happens:** fake-indexeddb doesn't fully replicate real IndexedDB edge cases and transaction behavior
**How to avoid:** Browser mode tests use real IndexedDB by default - only use fake-indexeddb for Node.js Netlify Functions tests
**Warning signs:** Tests pass but users report IndexedDB errors in production

### Pitfall 4: Module-Level Mock Timing Issues
**What goes wrong:** Mocked modules return undefined or fail to apply properly
**Why it happens:** vi.mock() must be called BEFORE importing the module under test
**How to avoid:** Always define vi.mock() at the top of the test file before any imports that use the mocked module
**Warning signs:** "TypeError: X is not a function" when calling mocked functions, or mocks not being tracked

### Pitfall 5: Not Clearing Mocks Between Tests
**What goes wrong:** Test state leaks between tests, causing false positives or hard-to-debug failures
**Why it happens:** Mock call history and implementations persist across test runs
**How to avoid:** Call vi.clearAllMocks() in beforeEach(), or enable clearMocks: true in Vitest config
**Warning signs:** toHaveBeenCalledOnce() fails with "expected 1, received 2" when test runs after another

### Pitfall 6: Testing Implementation Details Instead of Behavior
**What goes wrong:** Tests break when refactoring code even though behavior is unchanged
**Why it happens:** Tests assert internal function calls or state instead of observable outcomes
**How to avoid:** Test public API contracts and user-observable behavior, not internal implementation
**Warning signs:** Refactoring requires updating many tests that aren't related to changed functionality

### Pitfall 7: Inadequate Error Scenario Coverage
**What goes wrong:** Tests pass but production code crashes with unhandled errors
**Why it happens:** Tests focus on happy paths without testing network failures, invalid input, race conditions
**How to avoid:** For HIGH risk code (sync, pairing), test error scenarios: network failures, decryption errors, invalid data, concurrent operations
**Warning signs:** Users report crashes in error scenarios that tests never covered

### Pitfall 8: Fake Timers Without Microtask Flushing
**What goes wrong:** Debounced async tests fail or hang when using vi.useFakeTimers()
**Why it happens:** Fake timers only fast-forward macrotasks, not microtasks (promises)
**How to avoid:** Call `await Promise.resolve()` before vi.advanceTimersByTime() to flush microtasks
**Warning signs:** Tests timeout when using fake timers with async/await code

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| @testing-library/svelte + jsdom | Vitest browser mode + vitest-browser-svelte | Svelte 5 (2024) | Runes require real browser reactivity; jsdom can't replicate $state behavior |
| fake-indexeddb for all tests | Real IndexedDB in browser mode, fake-indexeddb only for Node.js | Phase 08.6 (2025) | More realistic testing; catches real IndexedDB edge cases |
| Jest | Vitest | 2022-2023 | Native ESM, faster, better Vite integration, browser mode support |
| Manual mock setup in each test | vi.mock() at module level | Vitest best practice | Consistent mocking, less boilerplate, better test isolation |
| coverage.exclude patterns | coverage.include with glob thresholds | Vitest v4 (2024) | More intuitive, per-directory thresholds |

**Deprecated/outdated:**
- **@testing-library/svelte with jsdom**: Can't test Svelte 5 runes reactivity; use vitest-browser-svelte instead
- **fake-indexeddb in browser mode**: Browser mode provides real IndexedDB; only use for Node.js tests
- **Manual flushSync imports**: Must import from 'svelte' not 'svelte/internal' (changed in Svelte 5)

## Open Questions

Things that couldn't be fully resolved:

1. **Optimal coverage threshold for existing untested codebase**
   - What we know: General best practice is 70-85% for production code, 90%+ for critical paths
   - What's unclear: What specific threshold is realistic given 107 existing tests and ~2000 untested lines
   - Recommendation: Start with coverage.include: ['src/lib/sync/**', 'src/lib/utils/**'] and 75% threshold, measure current baseline first

2. **Testing analytics client with sendBeacon API**
   - What we know: Browser mode provides real navigator.sendBeacon
   - What's unclear: How to verify sendBeacon was called correctly (it returns boolean, doesn't expose sent data)
   - Recommendation: Mock navigator.sendBeacon with vi.spyOn(navigator, 'sendBeacon') to verify calls

3. **Testing polling intervals without real delays**
   - What we know: Fake timers work with setInterval via vi.advanceTimersByTime()
   - What's unclear: Whether polling combined with IndexedDB operations causes timing issues
   - Recommendation: Test polling logic with fake timers, may need to flush microtasks between intervals

4. **Component test coverage contribution**
   - What we know: Vitest counts browser mode test coverage toward overall coverage
   - What's unclear: Whether component tests meaningfully increase coverage for stores/utils they import
   - Recommendation: Focus component tests on UI interaction, not coverage numbers; test business logic separately

5. **Netlify Functions cold start behavior**
   - What we know: Can test function logic with mocked Blobs
   - What's unclear: How to test cold start latency, connection pooling, or Netlify-specific runtime behavior
   - Recommendation: Unit tests for logic only; defer cold start/performance testing to E2E or monitoring
