# Phase 08.7: Site Analytics & Usage Metrics - Research

**Researched:** 2026-02-01
**Domain:** Self-built privacy-first analytics with serverless infrastructure
**Confidence:** MEDIUM

## Summary

Research investigated how to build a privacy-respecting, GDPR-compliant analytics system using the existing Netlify Functions + Blobs infrastructure. The standard approach for self-built analytics involves three core components: (1) a lightweight client-side tracking script that sends event beacons to a serverless endpoint, (2) a serverless function that processes and stores anonymized event data in a key-value store, and (3) a simple dashboard that queries aggregated data for visualization.

The privacy-first approach avoids cookies, persistent cross-session identifiers, and fingerprinting, instead using daily-rotated hashes of IP + User-Agent for same-day visitor deduplication. This matches the pattern used by Plausible Analytics: `hash(daily_salt + domain + ip + user_agent)` with the salt rotated every 24 hours. Data collection is limited to page URLs (path only), referrer, browser/OS (derived from User-Agent), device type, and geographic location (country-level via IP lookup, then IP discarded).

The architecture leverages Netlify Blobs (optimized for frequent reads, infrequent writes) to store aggregated event data as JSON. A simple HTML dashboard with Chart.js can visualize trends without requiring complex BI infrastructure. Event-based schema design tracks "what happens" (events) separately from "what exists" (entities), with each event containing timestamp, event name, properties, and optional anonymous session ID.

**Primary recommendation:** Use Netlify Functions to receive event beacons via POST, hash IP+User-Agent with daily salt for visitor deduplication, store aggregated daily/weekly metrics in Netlify Blobs as JSON, and build a password-protected SvelteKit route with Chart.js for team dashboard access.

### Pattern 1: Privacy-First Visitor Identification (Plausible Pattern)
**What:** Generate daily-changing visitor ID by hashing IP + User-Agent with rotating salt
**When to use:** Need to count unique visitors per day without cookies or persistent tracking
**Example:**
```typescript
// Server-side only (Netlify Function)
// Source: https://plausible.io/data-policy

async function generateVisitorId(
  request: Request,
  siteDomain: string
): Promise<string> {
  const ip = request.headers.get('x-forwarded-for')?.split(',')[0] ||
             request.headers.get('x-real-ip') ||
             'unknown';
  const userAgent = request.headers.get('user-agent') || 'unknown';

  // Daily salt (rotate every 24 hours)
  const today = new Date().toISOString().split('T')[0]; // YYYY-MM-DD
  const dailySalt = await getDailySalt(today);

  // Hash: hash(daily_salt + website_domain + ip_address + user_agent)
  const data = `${dailySalt}${siteDomain}${ip}${userAgent}`;
  const hashBuffer = await crypto.subtle.digest('SHA-256',
    new TextEncoder().encode(data)
  );
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}

// CRITICAL: Never persist IP or User-Agent, only the hash
// CRITICAL: Delete old salts after 24 hours to prevent brute-force
```

**Privacy guarantee:** Original IP addresses cannot be recovered after salt deletion (GDPR Article 17 - Right to Erasure).

### Pattern 2: Event Beacon with navigator.sendBeacon
**What:** Non-blocking event transmission that survives page unload
**When to use:** Track page views, navigation events, and user actions without delaying page interactions
**Example:**
```typescript
// Client-side tracking script
// Source: https://oliverjam.es/articles/diy-analytics-netlify-functions

export function trackEvent(eventName: string, properties?: Record<string, any>) {
  const payload = JSON.stringify({
    event: eventName,
    url: window.location.pathname, // Path only, no query params
    referrer: document.referrer,
    timestamp: new Date().toISOString(),
    ...properties
  });

  const endpoint = '/.netlify/functions/analytics-track';

  // Use sendBeacon for guaranteed delivery (even on page unload)
  if (navigator.sendBeacon) {
    const blob = new Blob([payload], { type: 'application/json' });
    navigator.sendBeacon(endpoint, blob);
  } else {
    // Fallback to fetch (best-effort)
    fetch(endpoint, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: payload,
      keepalive: true // Persist after page unload
    }).catch(() => {}); // Fail silently, don't block user
  }
}

// Track page views on navigation (SvelteKit pattern)
// Source: https://rodneylab.com/tracking-page-views-sveltekit/
import { page } from '$app/stores';

$effect(() => {
  trackEvent('pageview', { path: $page.url.pathname });
});
```

### Pattern 3: Aggregated Storage in Netlify Blobs
**What:** Store pre-aggregated daily metrics instead of raw events
**When to use:** Reduce storage costs and query complexity for time-series analytics
**Example:**
```typescript
// Netlify Function: analytics-track.mts
// Source: https://lynn.zone/blog/serverless-analytics/

import { getStore } from '@netlify/blobs';

export default async (req: Request, context: Context) => {
  const { event, url, referrer, timestamp } = await req.json();
  const visitorId = await generateVisitorId(req, 'gtd.app');

  const store = getStore('analytics');
  const today = new Date().toISOString().split('T')[0]; // YYYY-MM-DD

  // Fetch today's aggregate (or initialize)
  const key = `daily/${today}`;
  const existing = await store.get(key, { type: 'json' }) || {
    date: today,
    pageviews: {},
    events: {},
    visitors: new Set(),
    referrers: {}
  };

  // Update aggregates
  existing.pageviews[url] = (existing.pageviews[url] || 0) + 1;
  existing.events[event] = (existing.events[event] || 0) + 1;
  existing.visitors.add(visitorId);
  if (referrer) {
    existing.referrers[referrer] = (existing.referrers[referrer] || 0) + 1;
  }

  // Convert Set to array for JSON serialization
  const updated = {
    ...existing,
    visitors: Array.from(existing.visitors),
    uniqueVisitors: existing.visitors.size
  };

  await store.set(key, JSON.stringify(updated));

  return new Response(JSON.stringify({ success: true }), {
    status: 200,
    headers: { 'Content-Type': 'application/json' }
  });
};
```

### Anti-Patterns to Avoid
- **Collecting PII before aggregation:** Never store raw IP addresses, emails, or user IDs in analytics events. Hash or anonymize immediately upon ingestion.
- **Using localStorage for visitor tracking:** Creates persistent cross-session identifiers, violates GDPR's cookie-free guarantee. Use daily-rotating server-side hash instead.
- **Fingerprinting browser properties:** Collecting canvas fingerprints, font lists, or WebGL data is explicitly rejected by privacy-first tools and triggers GDPR violations.
- **Storing events indefinitely:** Raw event logs accumulate quickly (storage costs) and violate GDPR data minimization. Aggregate to daily metrics within 24 hours, then delete raw events.
- **Loading tracking script synchronously:** Blocks page rendering. Use async/defer or load in $effect() after hydration.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Time zone conversion | Custom Date math | Intl.DateTimeFormat with timeZone option | Edge cases: DST transitions, historical TZ changes, leap seconds |
| Daily salt rotation | setTimeout scheduler | Netlify Scheduled Functions (cron) | Survives function cold starts, guaranteed execution, built-in retry |
| Visitor geolocation | Parse IP ranges | Request headers (Netlify adds geo data) | Netlify provides country/region via `x-country-code` header for free |
| Chart rendering | SVG path calculations | Chart.js | Handles responsive sizing, animations, tooltips, accessibility |
| Rate limiting | In-memory counters | Netlify Edge rate limiting | Distributed, survives function restarts, prevents DDoS |

**Key insight:** Analytics infrastructure has hidden complexity in privacy compliance, data aggregation, and performance optimization. Use battle-tested patterns (Plausible's hash approach, Chart.js visualization) instead of reinventing.

## Common Pitfalls

### Pitfall 1: Clock Skew in Daily Salt Rotation
**What goes wrong:** Netlify Functions run in distributed data centers with slight clock differences. A visitor's request at 23:59:58 might hash with one salt, while the aggregation function at 00:00:02 uses a different salt, causing duplicate visitor counts.
**Why it happens:** Daily salt rotation uses `new Date().toISOString().split('T')[0]` which depends on server clock accuracy.
**How to avoid:** Store salt generation timestamp in Blobs metadata. Use 24-hour TTL from creation time (not calendar day boundaries). Example: Salt created at 2026-02-01 14:00 expires 2026-02-02 14:00.
**Warning signs:** Unique visitor counts jump 2x on day boundaries, then drop back to normal.

### Pitfall 2: Set Serialization in Netlify Blobs
**What goes wrong:** JavaScript `Set` objects cannot be directly JSON.stringify'd. Storing `{ visitors: new Set([...]) }` results in `{ visitors: {} }` (empty object).
**Why it happens:** JSON spec doesn't support Set type. Sets serialize to empty objects by default.
**How to avoid:** Convert to array before storing: `Array.from(visitorsSet)`. Reconstruct Set on read: `new Set(data.visitors)`.
**Warning signs:** `uniqueVisitors` count is always 0 despite events being tracked.

### Pitfall 3: Consent Banner Not Required = Still Need Privacy Policy
**What goes wrong:** Team assumes cookie-free analytics means no legal obligations. GDPR Article 13 requires transparency regardless of consent requirements.
**Why it happens:** Misconception that "no cookies = no compliance needed."
**How to avoid:** Add Privacy Policy page explaining: (1) What data is collected (page URLs, browser type, country), (2) How visitor IDs are generated (daily hash, not persistent), (3) Data retention (30 days aggregated, no raw events), (4) No third-party sharing.
**Warning signs:** EU regulatory inquiry or user complaint about undisclosed tracking.

### Pitfall 4: Browser Do Not Track (DNT) Header Ignored
**What goes wrong:** Analytics tracks users who have DNT enabled, despite their explicit opt-out signal.
**Why it happens:** DNT header is not legally enforced, but ignoring it violates user trust and privacy-first principles.
**How to avoid:** Check `navigator.doNotTrack` on client before sending beacons. Respect `DNT: 1` header in Netlify Function.
```typescript
// Client-side check
if (navigator.doNotTrack === '1') {
  console.log('User has DNT enabled, skipping analytics');
  return;
}
```
**Warning signs:** Privacy-conscious users discover tracking via browser DevTools, damage to reputation.

### Pitfall 5: SvelteKit Page Transitions Not Tracked
**What goes wrong:** Page views only tracked on full page load (first visit). Client-side navigation (SvelteKit's default) doesn't trigger pageview events.
**Why it happens:** Traditional analytics relies on `window.onload`, but SvelteKit uses client-side routing after hydration.
**How to avoid:** Use `$effect()` with `$page.url.pathname` store to track route changes.
```typescript
import { page } from '$app/stores';

$effect(() => {
  trackEvent('pageview', { path: $page.url.pathname });
});
```
**Warning signs:** Analytics shows only 1 pageview per session, even when users navigate multiple pages.

### Pitfall 6: Analytics Function Cold Start Timeouts
**What goes wrong:** First analytics event after 15+ minutes of inactivity takes 3-5 seconds, causing beacon failures and data loss.
**Why it happens:** Netlify Functions have cold start penalty. Blobs SDK initialization adds ~500ms overhead.
**How to avoid:**
- Use `navigator.sendBeacon` with `keepalive: true` (tolerates slow responses)
- Pre-warm function with scheduled ping every 10 minutes (Netlify Scheduled Functions)
- Set function timeout to 10s (default is 10s, but explicitly configure)
**Warning signs:** Missing pageviews during low-traffic hours (nights, weekends).

### Pitfall 7: GDPR Data Retention Violation
**What goes wrong:** Storing aggregated analytics data indefinitely violates GDPR Article 5(1)(e) - storage limitation principle.
**Why it happens:** Assumption that anonymized data can be kept forever.
**How to avoid:** Define retention policy (recommended: 13 months for year-over-year comparison). Use Netlify Scheduled Function to delete old Blobs:
```typescript
// Delete aggregates older than 13 months
const cutoffDate = new Date();
cutoffDate.setMonth(cutoffDate.getMonth() - 13);
const cutoffKey = cutoffDate.toISOString().split('T')[0];

const store = getStore('analytics');
const { blobs } = await store.list({ prefix: 'daily/' });

for (const blob of blobs) {
  if (blob.key < `daily/${cutoffKey}`) {
    await store.delete(blob.key);
  }
}
```
**Warning signs:** Indefinite data growth, GDPR audit findings.

### Server-Side Visitor Hashing (Privacy-First)
```typescript
// netlify/functions/analytics-utils.ts
// Source: Plausible Analytics data policy

export async function generateVisitorId(
  request: Request,
  siteDomain: string
): Promise<string> {
  // Extract IP from Netlify headers
  const ip = request.headers.get('x-nf-client-connection-ip') ||
             request.headers.get('x-forwarded-for')?.split(',')[0] ||
             'unknown';

  const userAgent = request.headers.get('user-agent') || 'unknown';

  // Get or create today's salt
  const store = getStore('analytics');
  const today = new Date().toISOString().split('T')[0];
  const saltKey = `salt/${today}`;

  let salt = await store.get(saltKey, { type: 'text' });

  if (!salt) {
    // Generate new salt (256-bit random)
    const randomBytes = crypto.getRandomValues(new Uint8Array(32));
    salt = Array.from(randomBytes)
      .map(b => b.toString(16).padStart(2, '0'))
      .join('');

    // Store with 24-hour expiry metadata
    await store.set(saltKey, salt, {
      metadata: { createdAt: Date.now(), expiresAt: Date.now() + 86400000 }
    });
  }

  // Hash: salt + domain + IP + User-Agent
  const data = `${salt}${siteDomain}${ip}${userAgent}`;
  const hashBuffer = await crypto.subtle.digest(
    'SHA-256',
    new TextEncoder().encode(data)
  );

  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}

// Scheduled cleanup (Netlify Scheduled Functions - cron: 0 0 * * *)
export async function cleanupOldSalts() {
  const store = getStore('analytics');
  const { blobs } = await store.list({ prefix: 'salt/' });

  for (const blob of blobs) {
    const metadata = await store.getMetadata(blob.key);
    if (metadata?.expiresAt && Date.now() > metadata.expiresAt) {
      await store.delete(blob.key);
      console.log(`Deleted expired salt: ${blob.key}`);
    }
  }
}
```

### Dashboard Authentication (SvelteKit Hooks)
```typescript
// src/hooks.server.ts
// Protect /analytics route with basic auth

import { error } from '@sveltejs/kit';

const ANALYTICS_PASSWORD = process.env.ANALYTICS_PASSWORD;

export async function handle({ event, resolve }) {
  if (event.url.pathname.startsWith('/analytics')) {
    const authHeader = event.request.headers.get('authorization');

    if (!authHeader || !authHeader.startsWith('Basic ')) {
      return new Response('Unauthorized', {
        status: 401,
        headers: {
          'WWW-Authenticate': 'Basic realm="Analytics Dashboard"'
        }
      });
    }

    const base64Credentials = authHeader.split(' ')[1];
    const credentials = atob(base64Credentials);
    const [username, password] = credentials.split(':');

    if (username !== 'admin' || password !== ANALYTICS_PASSWORD) {
      throw error(403, 'Invalid credentials');
    }
  }

  return resolve(event);
}
```

## Open Questions

Things that couldn't be fully resolved:

1. **Geolocation accuracy from Netlify headers**
   - What we know: Netlify provides `x-country-code` header (ISO 3166-1 alpha-2). Some sources mention `x-nf-geo` with city-level data.
   - What's unclear: Whether city-level geo data is available on all Netlify plans, or requires Pro/Enterprise. Official docs don't specify granularity.
   - Recommendation: Test with `console.log(Object.fromEntries(request.headers))` in analytics function. Fall back to country-only if city data unavailable. Document in plan which geo precision is achievable.

2. **Scheduled Function execution guarantees**
   - What we know: Netlify Scheduled Functions run on cron syntax (e.g., `0 0 * * *` for midnight daily). Documentation states "best-effort delivery."
   - What's unclear: If midnight cleanup fails (function timeout, Blobs API error), does it retry? Is there a dead-letter queue?
   - Recommendation: Add idempotent cleanup logic (safe to run multiple times). Log failures to Blobs for manual review. Consider redundant cleanup on analytics-query.mts startup.

3. **Blobs consistency model for concurrent writes**
   - What we know: Netlify Blobs uses "last-write-wins" consistency. Multiple concurrent `store.set()` calls to same key may result in one overwriting another.
   - What's unclear: If two analytics events arrive simultaneously and both try to update `daily/2026-02-01`, will one update be lost?
   - Recommendation: Accept potential minor data loss (analytics is approximate, not transactional). For critical events (signup, activation), consider append-only log pattern: `daily/2026-02-01/${timestamp}-${uuid}` instead of single aggregate. Then aggregate on read.

4. **Chart.js bundle size impact**
   - What we know: Chart.js is 60KB minified (per search results). Dashboard is admin-only route.
   - What's unclear: Whether to lazy-load Chart.js only on /analytics route, or accept bundle size increase for all users.
   - Recommendation: Use SvelteKit's dynamic import in +page.svelte: `const Chart = (await import('chart.js/auto')).default;` in `onMount()`. This code-splits Chart.js into separate chunk loaded only when dashboard accessed.

5. **DNT header reliability**
   - What we know: `navigator.doNotTrack` returns `'1'` (enabled), `'0'` (disabled), or `null` (unset). Not all browsers support it (Safari removed in 2019).
   - What's unclear: Should analytics respect DNT if only 5% of users have it enabled? Is checking DNT sufficient for "privacy-first" claim?
   - Recommendation: Check DNT as baseline. Add opt-out link in Privacy Policy that sets `localStorage.analytics_opted_out = 'true'` for additional user control. Document both mechanisms.
