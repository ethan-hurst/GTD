  <files>src/lib/sync/merge.test.ts</files>
  <action>
Create `src/lib/sync/merge.test.ts` with comprehensive tests for the LWW merge system. Import `mergeTable`, `mergePayloads`, `mergeData` from './merge'.

Test groups for `mergeTable`:
1. **Basic merging:**
   - Empty local + empty remote = empty result
   - Items only in local = local items preserved
   - Items only in remote = remote items added
   - Non-overlapping items from both sides = all items present

2. **LWW conflict resolution:**
   - Same ID, remote newer = remote wins (compare modified timestamps)
   - Same ID, local newer = local wins
   - Same ID, equal timestamps = local wins (tie-breaker)
   - Same ID, remote has ISO string dates (not Date objects) = still compares correctly after deserialization

3. **Tombstone handling:**
   - Remote has deleted=true with newer timestamp = tombstone preserved in result
   - Local has deleted=true with newer timestamp = tombstone preserved
   - Both have tombstones = newer tombstone wins

4. **Date deserialization:**
   - Remote records with ISO string dates get deserialized to Date objects
   - All DATE_FIELDS are properly deserialized (modified, created, deletedAt, startTime, endTime, followUpDate, completedAt)

5. **Custom key function:**
   - Using custom keyFn (e.g., `record => record.key`) works for settings table

Test groups for `mergePayloads`:
1. Multiple tables merge independently
2. Settings table uses 'key' field (not 'id') for merge key
3. Tables only in local or only in remote are preserved
4. Empty tables result in empty arrays

Test groups for `mergeData`:
1. Delegates to mergePayloads (single integration test)

Use concrete test data with specific dates:
- older: new Date('2026-01-01T00:00:00Z')
- newer: new Date('2026-01-15T00:00:00Z')

Each test should be self-contained with inline test data (no shared fixtures that could leak between tests).
