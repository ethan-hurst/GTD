# Phase 08.6: Backend, Frontend & Integration Tests - Research

**Researched:** 2026-02-01
**Domain:** Testing SvelteKit 2 + Svelte 5 applications with IndexedDB and Netlify Functions
**Confidence:** HIGH

## Summary

Modern SvelteKit 2 + Svelte 5 testing has shifted dramatically from jsdom-based approaches to real browser testing. The standard stack centers on **Vitest Browser Mode** with Playwright for component and unit tests, using `vitest-browser-svelte` for rendering Svelte 5 components with runes support. For IndexedDB testing (Dexie.js), `fake-indexeddb` provides in-memory mocking with proper test isolation. Netlify Functions are tested locally via `netlify dev` and `netlify functions:invoke`. End-to-end tests use Playwright to simulate full user journeys.

**Key architectural shift:** Svelte 5 runes (`$state`, `$derived`, `$effect`) require browser environments to function. Jsdom cannot properly simulate rune reactivity, making Vitest Browser Mode mandatory for testing components that use runes.

**Primary recommendation:** Use Vitest Browser Mode + Playwright as the test runner, `vitest-browser-svelte` for component rendering, `fake-indexeddb` for IndexedDB mocking, and Playwright's E2E mode for integration tests. This provides a unified testing stack with real browser execution.

### Pattern 1: Testing Svelte 5 Runes with Browser Mode
**What:** Svelte 5 runes (`$state`, `$derived`, `$effect`) only work in browser/component contexts, not in Node.js. Browser mode provides real browser execution.

**When to use:** Any test involving components or stores that use runes.

**Example:**
```typescript
// Source: https://github.com/vitest-dev/vitest-browser-svelte
import { render } from 'vitest-browser-svelte'
import { expect, test } from 'vitest'
import Counter from './Counter.svelte'

test('counter increments on click', async () => {
  const screen = render(Counter, { initialCount: 0 })
  const button = screen.getByRole('button', { name: /increment/i })

  await button.click()

  const count = screen.getByText(/count: 1/i)
  await expect.element(count).toBeInTheDocument()
})
```

### Pattern 2: Testing Svelte 5 Runes in Standalone Files
**What:** Test files with `.svelte.ts` extension can use runes directly if using browser mode.

**When to use:** Testing Svelte 5 stores/state that use runes (like your `sync.svelte.ts`).

**Example:**
```typescript
// Source: https://svelte.dev/docs/svelte/testing
// File: sync.test.ts (runs in browser mode)
import { test, expect } from 'vitest'
import { flushSync } from 'svelte'

test('sync state updates reactively', () => {
  const state = $state({ count: 0 })

  const cleanup = $effect.root(() => {
    $effect(() => {
      // Effect code here
    })

    return () => { /* cleanup */ }
  })

  state.count = 1
  flushSync() // Force synchronous update for testing

  expect(state.count).toBe(1)
  cleanup()
})
```

### Pattern 3: Mocking IndexedDB with Dexie
**What:** Use `fake-indexeddb` to provide in-memory IndexedDB for Dexie tests without browser overhead.

**When to use:** Testing database operations, avoiding real IndexedDB in unit tests.

**Example:**
```typescript
// Source: https://github.com/dumbmatter/fakeIndexedDB
// File: vitest.config.ts
import { defineConfig } from 'vitest/config'

export default defineConfig({
  test: {
    setupFiles: ['./src/tests/setup.ts'],
    // ...
  }
})

// File: src/tests/setup.ts
import 'fake-indexeddb/auto'

// File: operations.test.ts
import { beforeEach, test, expect } from 'vitest'
import { IDBFactory } from 'fake-indexeddb'
import { db } from '$lib/db'

beforeEach(() => {
  // Reset IndexedDB between tests for isolation
  indexedDB = new IDBFactory()
})

test('creates GTD item', async () => {
  const item = await db.gtdItems.add({
    title: 'Test task',
    type: 'action',
    // ...
  })

  expect(item).toBeDefined()
  const retrieved = await db.gtdItems.get(item.id!)
  expect(retrieved?.title).toBe('Test task')
})
```

**When to use:** Testing serverless function logic before deployment.

### Pattern 5: End-to-End Testing with Playwright
**What:** Full user flows tested against running application.

**When to use:** Verifying complete features (sync flow, CRUD operations across UI).

**Example:**
```typescript
// Source: https://svelte.dev/docs/svelte/testing
// File: tests/e2e/sync-flow.test.ts
import { test, expect } from '@playwright/test'

test('complete sync flow', async ({ page }) => {
  await page.goto('/')

  // Create item
  await page.getByRole('button', { name: 'New Action' }).click()
  await page.getByPlaceholder('What needs to be done?').fill('Test task')
  await page.getByRole('button', { name: 'Save' }).click()

  // Trigger sync
  await page.getByRole('button', { name: 'Sync' }).click()

  // Verify sync state
  await expect(page.getByText(/synced/i)).toBeVisible()
})
```

### Pattern 6: Testing Web Crypto API
**What:** Web Crypto API is available in browser mode but requires mocking in Node.js environments.

**When to use:** Testing encryption/decryption logic in sync layer.

**Example:**
```typescript
// Source: Vitest Browser Mode provides real crypto.subtle
// File: crypto.test.ts (runs in browser mode)
import { test, expect } from 'vitest'
import { encryptData, decryptData } from '$lib/sync/crypto'

test('encrypts and decrypts data', async () => {
  const data = { test: 'value' }
  const password = 'test-password'

  const encrypted = await encryptData(data, password)
  expect(encrypted).toBeDefined()

  const decrypted = await decryptData(encrypted, password)
  expect(decrypted).toEqual(data)
})
```

### Anti-Patterns to Avoid
- **Testing implementation details:** Don't test internal state variables or private methods. Test user-visible behavior and public APIs.
- **Not resetting IndexedDB between tests:** Causes data leakage between tests. Always reset with `new IDBFactory()`.
- **Using jsdom for Svelte 5 components:** Jsdom cannot execute runes properly. Use browser mode.
- **Mocking too aggressively:** Don't mock Dexie or IndexedDB in integration tests - use the real libraries with fake-indexeddb.
- **Ignoring async cleanup:** Always cleanup effects and components to prevent test pollution.
- **Testing after navigation without waiting:** Use `await` for all user interactions and DOM queries.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| IndexedDB mocking | Custom in-memory storage | fake-indexeddb | Passes 82.8% of Web Platform Tests, handles edge cases (transactions, cursors, indexes) |
| DOM assertions | Custom matchers | Vitest Browser Mode built-in matchers | Already includes forked @testing-library/jest-dom with retry-ability |
| Component cleanup | Manual beforeEach/afterEach | vitest-browser-svelte auto-cleanup | Handles Svelte-specific cleanup, prevents memory leaks |
| Browser automation | Custom Selenium setup | Playwright | Official SvelteKit recommendation, better API, built-in waiting |
| Test isolation in browser mode | Global state management | Vitest's test context isolation | Each test runs in isolated context automatically |
| Service worker testing | Manual browser testing | Playwright E2E tests | Real browser environment, can test SW registration and caching |
| Coverage reporting | Custom instrumentation | @vitest/coverage-v8 | AST-based remapping (v3.2.0+) = Istanbul accuracy with V8 speed |

**Key insight:** Testing infrastructure is complex with subtle edge cases. Vitest Browser Mode and Playwright have solved browser testing problems that took years to stabilize. Don't rebuild what the ecosystem has already solved.

## Common Pitfalls

### Pitfall 1: Forgetting to Install Playwright Browsers
**What goes wrong:** Tests fail with "No browsers found" error despite having `@vitest/browser-playwright` installed.

**Why it happens:** Playwright requires a separate installation step for browser binaries.

**How to avoid:** Run `npx playwright install chromium` after installing packages.

**Warning signs:** Error message: "browserType.launch: Executable doesn't exist at ..."

### Pitfall 2: Testing Runes in Node.js Environment
**What goes wrong:** Runes (`$state`, `$derived`, `$effect`) fail with cryptic errors or don't react properly.

**Why it happens:** Runes only work in Svelte component context or browser mode, not in Node.js.

**How to avoid:** Use `browser.enabled: true` in vitest config for any tests involving runes. Test files can use `.svelte.ts` extension.

**Warning signs:** "Cannot read properties of undefined" when accessing rune state, or state doesn't update reactively.

### Pitfall 3: Not Resetting IndexedDB Between Tests
**What goes wrong:** Tests pass in isolation but fail when run together. Data from one test affects another.

**Why it happens:** fake-indexeddb maintains state across tests unless explicitly reset.

**How to avoid:** Add `beforeEach(() => { indexedDB = new IDBFactory() })` to test files using IndexedDB.

**Warning signs:** Tests fail when run with other tests but pass in isolation. "Unique constraint" errors.

### Pitfall 4: Missing flushSync() in Rune Tests
**What goes wrong:** Assertions fail because reactive updates haven't executed yet.

**Why it happens:** Svelte batches reactive updates. In tests, you need to force synchronous execution.

**How to avoid:** Import `flushSync` from 'svelte' and call it before assertions that depend on reactive updates.

**Warning signs:** Intermittent test failures, "expected X but got Y" where Y is the old value.

### Pitfall 5: Testing Netlify Functions in Wrong Environment
**What goes wrong:** Functions work locally but fail in production, or tests don't catch environment-specific issues.

**Why it happens:** `netlify dev` simulates but doesn't perfectly replicate production environment.

**How to avoid:**
- Unit test pure logic separately
- Use `netlify dev` for integration testing
- Test against deploy previews for pre-production validation
- Never rely solely on local testing

**Warning signs:** "Works on my machine" - functions pass local tests but fail on Netlify.

### Pitfall 6: Not Awaiting Async User Interactions
**What goes wrong:** Tests fail with "element not found" or check state before interactions complete.

**Why it happens:** Browser interactions are asynchronous; queries need to wait for DOM updates.

**How to avoid:** Always `await` button.click(), form submissions, and use `await expect.element()` for assertions.

**Warning signs:** Flaky tests that sometimes pass, "element not visible" errors.

### Pitfall 7: Using @testing-library/user-event in Browser Mode
**What goes wrong:** Events are simulated instead of actually triggered, missing real browser behavior.

**Why it happens:** @testing-library/user-event simulates events; it doesn't use actual browser protocols.

**How to avoid:** Use `screen.getByRole('button').click()` (via vitest-browser-svelte) which uses Chrome DevTools Protocol or Webdriver.

**Warning signs:** Tests pass but real user interactions fail differently.

### Pitfall 8: Mixing jsdom and Browser Mode
**What goes wrong:** Some tests run in jsdom, others in browser mode, causing inconsistent behavior.

**Why it happens:** Default Vitest environment is 'node', can be overridden per-file with comments.

**How to avoid:** Set `browser.enabled: true` globally or use consistent per-file directives. Don't mix paradigms.

**Warning signs:** Tests behave differently in different files, crypto API available in some tests but not others.

```typescript
        effectTriggered = true
      }

    store.syncState = 'syncing'
    flushSync()

    return () => {} // cleanup
```

### Playwright E2E Test
```typescript
// Source: https://svelte.dev/docs/svelte/testing + https://www.okupter.com/blog/e2e-testing-with-sveltekit-and-playwright
// File: tests/e2e/sync-workflow.test.ts
import { test, expect } from '@playwright/test'

test.describe('Sync workflow', () => {
  test('pairs devices and syncs data', async ({ page, context }) => {
    // Device 1: Generate pairing code
    await page.goto('/settings')
    await page.getByRole('button', { name: 'Enable Sync' }).click()

    await device2.goto('/settings')
    await device2.getByRole('button', { name: 'Pair Device' }).click()
    await device2.getByPlaceholder('Enter code').fill(pairingCode)
    await device2.getByRole('button', { name: 'Pair' }).click()

    await page.goto('/actions')
    await page.getByRole('button', { name: 'New Action' }).click()

    await page.getByRole('button', { name: 'Sync Now' }).click()

    await device2.goto('/actions')
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| @testing-library/svelte + jsdom | vitest-browser-svelte + Browser Mode | 2025 (Svelte 5) | Jsdom cannot properly execute runes; browser mode required for Svelte 5 |
| Jest | Vitest | 2022+ | Vite-native testing, no separate build config, faster HMR |
| c8 coverage | @vitest/coverage-v8 with AST remapping | Vitest 3.2.0 (2024) | V8 speed with Istanbul accuracy |
| Separate test runners for unit/e2e | Unified Vitest + Playwright stack | 2024-2025 | Single test framework philosophy, shared config |
| Manual browser testing for Netlify Functions | netlify dev + netlify functions:invoke | Netlify CLI 3.0+ (2021) | Local testing parity with production |
| Testing Library user-event | Vitest Browser Mode native interactions | 2025 | Real browser events via CDP/Webdriver vs simulated |

**Deprecated/outdated:**
- **@testing-library/svelte with jsdom for Svelte 5:** Runes don't work properly in jsdom. Community has moved to browser mode.
- **Jest with SvelteKit:** Poor Vite integration, slow, requires complex configuration. Vitest is official recommendation.
- **c8 standalone:** Integrated into Vitest as @vitest/coverage-v8 with improved accuracy.
- **Testing libraries with manual act() calls:** vitest-browser-svelte includes retry-ability, no act() needed.

## Open Questions

Things that couldn't be fully resolved:

1. **Service Worker Testing Strategy**
   - What we know: Vitest Browser Mode supports service workers; Playwright E2E can test SW registration
   - What's unclear: Best practice for testing SW cache strategies and offline behavior in this stack
   - Recommendation: Start with E2E tests for SW registration/basic offline. Deep cache testing may require manual browser testing or vite-plugin-pwa test utilities.

2. **Optimal Coverage Thresholds**
   - What we know: @vitest/coverage-v8 provides accurate coverage reporting
   - What's unclear: Reasonable coverage targets for SvelteKit apps (backend vs frontend vs integration)
   - Recommendation: Start with baseline coverage reporting, no thresholds. Establish targets after seeing actual coverage patterns.

3. **Testing @netlify/blobs in Local Environment**
   - What we know: `netlify dev` provides local environment for functions
   - What's unclear: Whether @netlify/blobs works identically locally vs production, or needs mocking
   - Recommendation: Test blob operations against real Netlify environment (deploy previews) for integration tests. Mock for unit tests.

4. **Performance Testing IndexedDB Operations**
   - What we know: fake-indexeddb provides functional testing
   - What's unclear: Performance characteristics vs real IndexedDB, whether to add performance regression tests
   - Recommendation: Use fake-indexeddb for correctness testing. Add manual browser performance testing if IndexedDB performance becomes critical.
