Add a beforeEach block that:

   (The IDBFactory reset from setup.ts gives us a clean IndexedDB, but Dexie needs to reconnect)

Import from `$lib/db/operations`: addItem, updateItem, deleteItem, getItem, getAllInbox (and other query functions as available).
Import `db` from `$lib/db/schema`.

Test groups:

1. **addItem:**
   - Creates item with UUID id, created/modified dates auto-set
   - Returns the string UUID id
   - Item is retrievable from database after add
   - Item type is preserved (inbox, next-action, etc.)

2. **updateItem:**
   - Updates title and modified timestamp changes
   - Partial updates preserve other fields
   - Returns update count (1 for success)

3. **deleteItem (soft-delete):**
   - Sets deleted=true on the item (does NOT remove from DB)
   - Sets deletedAt timestamp
   - Updates modified timestamp
   - Item still exists in DB with deleted=true

4. **getItem:**
   - Returns item by ID
   - Returns undefined for non-existent ID

5. **getAllInbox (query function):**
   - Returns only items with type='inbox'
   - Excludes soft-deleted items (deleted=true)
   - Returns empty array when no inbox items exist

Use simple test data:

**Create `src/lib/sync/crypto.test.ts`** with tests for the crypto module. These tests run in browser mode so Web Crypto API (`crypto.subtle`) is available natively.

IMPORTANT: The crypto module's `encrypt` and `decrypt` functions depend on `initIVCounter` being called first (it uses getSetting/setSetting from db/operations). For unit tests, you need to either:
- Option A: Mock the IV counter initialization by directly testing `deriveKey`, `generateSalt`, and the encrypt/decrypt functions after initializing the counter
- Option B: Initialize the database (from fake-indexeddb) and call `initIVCounter` with a test device ID before encryption tests

Use Option B (real integration with fake-indexeddb) since the setup already provides it.

Test groups:

1. **generateSalt:**
   - Returns Uint8Array of length 16
   - Two calls produce different salts (random)

2. **deriveKey:**
   - Returns a CryptoKey object
   - Same pairing code + same salt = same derived key behavior (encrypt then decrypt works)

3. **encrypt + decrypt round-trip:**
   - beforeEach: reset DB, open it, call `initIVCounter('test-device-id')`
   - Encrypt a JSON string with a pairing code, decrypt with same code, get original back
   - Encrypted output is a JSON string containing salt, iv, ciphertext fields
   - Decrypt with wrong pairing code throws error
   - Multiple encryptions produce different ciphertexts (different IVs)

4. **getNextIV:**
   - After initIVCounter, getNextIV returns Uint8Array of length 12
   - Consecutive calls return different IVs (counter increments)

**Create `src/lib/db/export.test.ts`** with tests for database export/import.

Test groups:

1. **exportDatabase:**
   - beforeEach: reset DB, open it, add some test items
   - Export returns valid JSON string
   - Parsed JSON has version, exported (ISO date), and data properties
   - data contains items table with added items

2. **importDatabase:**
   - Import clears existing data and loads new data
   - After import, items are retrievable from DB
   - Invalid JSON throws error
   - Missing data property throws error

3. **Round-trip:**
   - Export, then import into fresh DB, then verify all items match

For export tests, add test data via db.items.add() directly (bypassing operations layer for simpler setup).
