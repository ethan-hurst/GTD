**Create `tests/e2e/inbox-crud.test.ts`:**
Import `test` and `expect` from '@playwright/test'.

IMPORTANT: Before writing these tests, you need to understand the app's UI flow. Read these source files to understand the DOM structure:
- `src/routes/+page.svelte` (or the inbox page route)
- `src/lib/components/InboxCapture.svelte`
- `src/lib/components/InboxList.svelte`

Tests:
1. 'can add item to inbox' - Navigate to the inbox page. Find the capture input. Type a task title. Submit (Enter or button click). Verify the new item appears in the inbox list.
2. 'inbox shows empty state when no items' - Navigate to inbox, verify empty state message is shown (before adding any items).
3. 'can add multiple items' - Add 2-3 items, verify all appear in the list.

Use Playwright's auto-waiting and retry-able assertions:
- `await expect(page.getByText('My Task')).toBeVisible()` (auto-retries)
- `await page.getByPlaceholder('...').fill('My Task')`
- `await page.getByRole('button', { name: '...' }).click()`

IMPORTANT: The app may show an onboarding wizard on first visit. If so, the E2E test may need to dismiss it first. Check if there's a way to bypass onboarding (e.g., set a localStorage flag) or click through it quickly.

If the dev server is not already running, Playwright will start it automatically (configured in playwright.config.ts webServer). Tests may take longer on first run.
