# Phase 08.7: Site Analytics & Usage Metrics - Research

**Researched:** 2026-02-01
**Domain:** Privacy-respecting web analytics for SvelteKit PWAs
**Confidence:** HIGH

## Summary

Privacy-focused analytics in 2026 has matured significantly with three established platforms dominating the space: Plausible, Umami, and Fathom. All three are GDPR-compliant, cookie-free, and designed to work without consent banners while providing the essential metrics needed to understand user behavior.

For a SvelteKit PWA hosted on Netlify with offline-first architecture, the standard approach is to use a lightweight, privacy-respecting analytics solution that tracks pageviews and custom events client-side, with optional server-side proxy routing to bypass ad blockers. The key technical challenge is ensuring analytics events are queued when offline and transmitted when connectivity returns, following established PWA patterns using service workers and IndexedDB.

The 2026 regulatory landscape has tightened considerably, with the EU Digital Omnibus initiative and increased enforcement making cookie-free, privacy-first analytics no longer optional but essential. Solutions that collect no personal data and use no cookies avoid consent requirements entirely under GDPR Article 5(3).

**Primary recommendation:** Use Plausible Analytics (cloud or self-hosted) with the @accuser/svelte-plausible-analytics integration package, implement custom event tracking for key user journeys, and add service worker-based offline queuing for analytics events to ensure no data loss during offline periods.

### Pattern 2: Custom Event Tracking

**What:** Track specific user interactions beyond pageviews (feature usage, conversions, onboarding steps)
**When to use:** For measuring engagement and user journeys
**Example:**

```typescript
// src/lib/analytics/events.ts
import { pa } from '@accuser/svelte-plausible-analytics';

// Define event names as constants to avoid typos
export const EVENTS = {
  ONBOARDING_STARTED: 'onboarding_started',
  ONBOARDING_COMPLETED: 'onboarding_completed',
  TASK_CREATED: 'task_created',
  TASK_COMPLETED: 'task_completed',
  SYNC_INITIATED: 'sync_initiated',
  SYNC_COMPLETED: 'sync_completed',
  OFFLINE_MODE_ENTERED: 'offline_mode_entered',
} as const;

// Track custom events with properties
export function trackEvent(eventName: string, props?: Record<string, string | number>) {
  if (typeof window !== 'undefined' && pa) {
    pa.addEvent(eventName, { props });
  }
}

// Usage in components
export function trackOnboardingComplete() {
  trackEvent(EVENTS.ONBOARDING_COMPLETED);
}

export function trackTaskCreated(taskType: string) {
  trackEvent(EVENTS.TASK_CREATED, { type: taskType });
}
```

### Pattern 3: Offline Event Queuing with Service Worker

**What:** Queue analytics events when offline and transmit when connectivity returns
**When to use:** For PWAs to ensure no analytics data loss during offline periods
**Example:**

```typescript
// src/service-worker.ts
import { BackgroundSyncPlugin } from 'workbox-background-sync';
import { NetworkOnly } from 'workbox-strategies';
import { registerRoute } from 'workbox-routing';

// Queue analytics requests that fail due to network issues
const bgSyncPlugin = new BackgroundSyncPlugin('analytics-queue', {
  maxRetentionTime: 24 * 60 // Retry for up to 24 hours
});

// Register route for Plausible analytics
registerRoute(
  ({ url }) => url.hostname === 'plausible.io' && url.pathname === '/api/event',
  new NetworkOnly({
    plugins: [bgSyncPlugin]
  }),
  'POST'
);

// Listen for sync event
self.addEventListener('sync', (event) => {
  if (event.tag === 'analytics-queue') {
    event.waitUntil(bgSyncPlugin.replayRequests());
  }
});
```

### Pattern 4: Proxy to Bypass Ad Blockers (Optional)

**What:** Route analytics requests through your own domain to bypass ad blocker detection
**When to use:** When analytics accuracy is critical and ad blocker bypass is acceptable
**Example:**

```typescript
// src/routes/api/analytics/proxy/+server.ts
import type { RequestHandler } from './$types';

export const POST: RequestHandler = async ({ request, fetch }) => {
  // Forward the analytics event to Plausible under your domain
  const response = await fetch('https://plausible.io/api/event', {
    method: 'POST',
    headers: {
      'Content-Type': request.headers.get('Content-Type') || 'application/json',
      'User-Agent': request.headers.get('User-Agent') || '',
      'X-Forwarded-For': request.headers.get('X-Forwarded-For') || ''
    },
    body: await request.text()
  });

  return new Response(null, { status: response.status });
};

// Configure Plausible to use the proxy endpoint
// src/routes/+layout.svelte
<PlausibleAnalytics
  domain="yourdomain.com"
  apiHost="/api/analytics/proxy"
/>
```

**Important:** Use non-obvious proxy paths (not `/analytics` or `/tracking`) as ad blockers detect common patterns. Only works with SSR (adapter-auto or adapter-netlify), not adapter-static.

### Pattern 5: SPA Route Change Tracking

**What:** Manually trigger pageview events on SvelteKit navigation (client-side routing)
**When to use:** SvelteKit handles this automatically with proper integration, but manual control needed for custom routing
**Example:**

```typescript
// src/routes/+layout.svelte
<script>
  import { page } from '$app/stores';
  import { pa } from '@accuser/svelte-plausible-analytics';

  // Track route changes in SPA
  $: if (browser && $page.url.pathname) {
    pa?.addEvent('pageview', {
      url: $page.url.pathname
    });
  }
</script>
```

Note: The @accuser/svelte-plausible-analytics package handles this automatically when placed in the root layout.

### Anti-Patterns to Avoid

- **Using Google Analytics without consent banner:** Violates GDPR in 2026, requires cookies, massive script size, blocked by ad blockers
- **Tracking before user interaction:** Don't track on first page load if user hasn't interacted; some DPAs require "strictly necessary" justification
- **Hand-rolling analytics infrastructure:** Session tracking, unique visitor counting, and geographic lookups are complex with edge cases
- **Storing PII in event properties:** Never pass email, name, IP, or user IDs in custom event properties
- **Obvious proxy paths:** Using `/analytics`, `/tracking`, or `/telemetry` defeats ad blocker bypass
- **Not queuing offline events:** PWAs will lose analytics data during offline periods without proper queuing
- **Tracking everything:** Focus on business-critical events; avoid data overload and meaningless metrics

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Unique visitor counting | Hash-based visitor IDs, daily rotation | Plausible/Umami built-in | Requires IP + User-Agent hashing with rotating salts, timezone considerations, privacy compliance |
| Geographic location lookup | IP geolocation database | Plausible/Umami built-in | MaxMind GeoIP database requires licensing, regular updates, GDPR compliance for IP storage |
| Device/browser detection | User-Agent parsing | Plausible/Umami built-in | 1000+ device signatures, constant updates for new devices, fallback logic |
| Session tracking | Custom session logic | Plausible/Umami built-in | Session timeout logic, cross-tab coordination, timezone handling |
| Analytics dashboard | Custom reporting UI | Plausible/Umami dashboard | Real-time updates, filtering, date ranges, data visualization, export formats |
| Offline event queuing | Custom IndexedDB queue | Workbox Background Sync | Retry logic, exponential backoff, request deduplication, storage limits |
| GDPR compliance infrastructure | Custom consent management | Cookie-free analytics | Legal requirements, consent storage, audit logs, data deletion |

**Key insight:** Analytics infrastructure has deep complexity in edge cases (bot detection, referrer spam, timezone handling, privacy compliance). The 2026 standard is to use established privacy-focused platforms rather than building custom solutions. Self-hosting is viable for infrastructure control, but don't build the analytics engine itself.

## Common Pitfalls

### Pitfall 1: Missing Offline Event Queue
**What goes wrong:** Analytics events are lost when PWA is used offline, creating gaps in usage data and underreporting active users
**Why it happens:** Default analytics scripts expect network connectivity; failed requests are discarded
**How to avoid:** Implement service worker-based Background Sync to queue failed analytics requests and replay when online
**Warning signs:** Lower-than-expected usage metrics, missing data during commute times or areas with poor connectivity

### Pitfall 2: Not Configuring Custom Events Before Use
**What goes wrong:** Custom events don't appear in dashboard; tracking code seems to work but no data is recorded
**Why it happens:** Plausible, Umami, and Fathom require custom events to be defined as "goals" in the dashboard before they will record data
**How to avoid:** Configure all custom event names in the analytics dashboard settings under Goals/Events before deploying tracking code
**Warning signs:** `pa.addEvent()` calls execute without errors but no data appears in reports

### Pitfall 3: Ad Blocker Blind Spot
**What goes wrong:** 30-50% of users have ad blockers, leading to severely underreported metrics
**Why it happens:** Ad blockers detect analytics scripts by domain (plausible.io, umami.is) and request patterns
**How to avoid:** Implement proxy endpoint under your domain OR accept reduced visibility into ad blocker users (many teams choose the latter for simplicity)
**Warning signs:** Analytics show lower traffic than server logs, discrepancy between reported users and backend API calls

### Pitfall 4: GDPR Consent Banner on Cookie-Free Analytics
**What goes wrong:** Implementing unnecessary consent banner creates friction and reduces opt-in rate despite using cookie-free analytics
**Why it happens:** Developers assume all analytics require consent; legal teams don't understand cookie-free exception
**How to avoid:** Document that cookie-free analytics (Plausible/Umami/Fathom) don't require consent under GDPR Article 5(3); get legal review if needed
**Warning signs:** Bounce rate increases after adding consent banner, opt-out rate is high, legal questions about compliance

### Pitfall 5: Tracking Meaningless Events
**What goes wrong:** Dashboard filled with hundreds of events that don't inform decisions; analysis paralysis, no actionable insights
**Why it happens:** "Track everything" mentality without asking "will this inform a decision?"
**How to avoid:** Start with 5-10 business-critical events (signup started, onboarding completed, key feature used, sync completed); add more only when specific questions arise
**Warning signs:** Analytics dashboard is overwhelming, team doesn't check metrics regularly, no actions taken based on data

### Pitfall 6: Not Accounting for "Signup" Semantics
**What goes wrong:** Confusion about what "signup funnel" means for a clientside-only app with no authentication
**Why it happens:** Analytics typically track account creation, but this app has no accounts
**How to avoid:** Define "signup" as "new user activation" - track first visit, onboarding start, onboarding completion, first task created as your funnel
**Warning signs:** Stakeholders ask about "signups" but metrics show zero because there's no signup flow

### Pitfall 7: SPA Route Changes Not Tracked
**What goes wrong:** Only first pageview is tracked; navigation to different routes doesn't register as pageviews
**Why it happens:** Traditional analytics expect full page loads; SvelteKit uses client-side routing
**How to avoid:** Use SvelteKit-specific integration package (like @accuser/svelte-plausible-analytics) that automatically tracks route changes
**Warning signs:** Pageview count matches unique visitor count (1:1 ratio), only entry pages show in dashboard

### Pitfall 8: Missing Server-Side Context in Analytics
**What goes wrong:** Can't correlate analytics events with server-side actions (sync attempts, API errors)
**Why it happens:** Client-side analytics only see browser events, not backend activity
**How to avoid:** Use Plausible Events API from Netlify Functions to track server-side events (sync initiated, sync failed, pairing completed) with proper IP forwarding
**Warning signs:** User reports sync issues but analytics show no errors, can't diagnose backend problems from usage patterns

### Custom Event Tracking Module

```typescript
// src/lib/analytics/events.ts
// Pattern: Centralized event tracking with type safety
import { pa } from '@accuser/svelte-plausible-analytics';

export const ANALYTICS_EVENTS = {
  // Onboarding funnel
  ONBOARDING_STARTED: 'onboarding_started',
  ONBOARDING_STEP_COMPLETED: 'onboarding_step_completed',
  ONBOARDING_COMPLETED: 'onboarding_completed',

  // Task interactions
  TASK_CREATED: 'task_created',
  TASK_COMPLETED: 'task_completed',
  TASK_DELETED: 'task_deleted',

  // Feature usage
  SYNC_INITIATED: 'sync_initiated',
  SYNC_COMPLETED: 'sync_completed',
  SYNC_FAILED: 'sync_failed',

  // Engagement
  SESSION_DURATION: 'session_duration',
  FEATURE_DISCOVERED: 'feature_discovered'
} as const;

type EventName = typeof ANALYTICS_EVENTS[keyof typeof ANALYTICS_EVENTS];
type EventProps = Record<string, string | number | boolean>;

export function trackEvent(name: EventName, props?: EventProps) {
  // Only track in browser context
  if (typeof window === 'undefined') return;

  try {
    pa?.addEvent(name, { props });
  } catch (error) {
    // Fail silently - analytics should never break the app
    console.warn('Analytics tracking failed:', error);
  }
}

// Convenience functions for common events
export const analytics = {
  onboardingStarted: () => trackEvent(ANALYTICS_EVENTS.ONBOARDING_STARTED),

  onboardingStepCompleted: (step: number) =>
    trackEvent(ANALYTICS_EVENTS.ONBOARDING_STEP_COMPLETED, { step }),

  onboardingCompleted: () => trackEvent(ANALYTICS_EVENTS.ONBOARDING_COMPLETED),

  taskCreated: (type: string) =>
    trackEvent(ANALYTICS_EVENTS.TASK_CREATED, { type }),

  syncCompleted: (itemCount: number, duration: number) =>
    trackEvent(ANALYTICS_EVENTS.SYNC_COMPLETED, { itemCount, duration }),

  syncFailed: (error: string) =>
    trackEvent(ANALYTICS_EVENTS.SYNC_FAILED, { error })
};
```

import type { Handler } from '@netlify/functions';

async function trackServerEvent(
  eventName: string,
  url: string,
  userAgent: string,
  clientIp: string,

      domain: process.env.PLAUSIBLE_DOMAIN,
      name: eventName,
      url,
      props
    })

  return response.ok;

    event.headers['user-agent'] || '',
    event.headers['x-forwarded-for'] || event.headers['client-ip'] || '',
    {
      items_synced: itemCount,
      duration_ms: duration
    }
  );

  // Return sync response...
};

### Offline Analytics Queue (Service Worker)

```typescript
// src/service-worker.ts
// Source: https://developers.chrome.com/docs/workbox/modules/workbox-background-sync
/// <reference types="@sveltejs/kit" />
/// <reference no-default-lib="true"/>
/// <reference lib="esnext" />
/// <reference lib="webworker" />

import { BackgroundSyncPlugin } from 'workbox-background-sync';
import { NetworkOnly } from 'workbox-strategies';
import { registerRoute } from 'workbox-routing';

const sw = self as unknown as ServiceWorkerGlobalScope;

// Queue failed analytics requests
const analyticsQueuePlugin = new BackgroundSyncPlugin('analytics-queue', {
  maxRetentionTime: 24 * 60, // Retry for up to 24 hours
  onSync: async ({ queue }) => {
    let entry;
    while ((entry = await queue.shiftRequest())) {
      try {
        await fetch(entry.request);
        console.log('Replay successful for request', entry.request.url);
      } catch (error) {
        console.error('Replay failed for request', entry.request.url);
        await queue.unshiftRequest(entry);
        throw error;
      }
    }
  }
});

// Register analytics endpoints for background sync
registerRoute(
  ({ url }) =>
    (url.hostname === 'plausible.io' && url.pathname === '/api/event') ||
    (url.hostname === 'yourdomain.com' && url.pathname.startsWith('/api/analytics')),
  new NetworkOnly({
    plugins: [analyticsQueuePlugin]
  }),
  'POST'
);

sw.addEventListener('install', (event) => {
  event.waitUntil(sw.skipWaiting());
});

sw.addEventListener('activate', (event) => {
  event.waitUntil(sw.clients.claim());
});
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Google Analytics Universal | Privacy-focused cookie-free analytics (Plausible/Umami/Fathom) | 2023-2025 | No consent banners needed, GDPR compliant by default, faster page loads |
| Cookie-based tracking | Rotating hash-based visitor IDs | 2022-2023 | Privacy compliance without sacrificing unique visitor counts |
| Manual SPA pageview tracking | Automatic route change detection | 2023-2024 | Framework-specific integrations handle SPA routing automatically |
| Client-only analytics | Hybrid client + server-side tracking | 2024-2026 | More accurate data, tracks backend events, bypasses ad blockers |
| Track everything | Business-aligned event tracking | 2024-2026 | Focus on actionable metrics, avoid data overload |
| Ignore offline users | Service worker queuing with Background Sync | 2024-2026 | No data loss for PWA offline usage |
| EU-only GDPR | Global privacy regulations (GDPR, CCPA, PECR, EU AI Act) | 2025-2026 | Privacy-first is global requirement, not regional |

**Deprecated/outdated:**
- **Google Analytics Universal Analytics (UA):** Sunset in 2023, replaced by GA4 which still has privacy/GDPR concerns
- **Cookie-based session tracking:** Requires consent banners, blocked by browsers, violates privacy regulations
- **Third-party hosted fonts/scripts from analytics:** Considered data transfer to third party under GDPR; use self-hosted or proxied
- **Fingerprinting techniques:** Explicitly banned under GDPR; rotating visitor hashes are the compliant alternative
- **Consent-free Google Analytics:** All major EU DPAs now require consent for GA4; exemption no longer exists

## Open Questions

Things that couldn't be fully resolved:

1. **Umami SvelteKit Integration Package**
   - What we know: Umami has official tracker script and REST API; community integration exists (github.com/SaschaLucius/svelte-umami)
   - What's unclear: Current maintenance status of svelte-umami package, compatibility with Umami v3.x features, official support status
   - Recommendation: If choosing Umami, verify svelte-umami package is actively maintained or implement direct script integration via +layout.svelte

2. **Self-Hosted Analytics on Netlify**
   - What we know: Plausible Community Edition and Umami can be self-hosted via Docker; Netlify hosting is primarily for static sites + Functions
   - What's unclear: Cost-effectiveness of running Plausible (requires Elixir + PostgreSQL + ClickHouse) or Umami (Next.js + PostgreSQL) separately vs using cloud service
   - Recommendation: For this project, start with Plausible Cloud ($9/month for <10k pageviews) or Umami Cloud ($20/month); defer self-hosting decision until traffic justifies infrastructure costs

3. **Historical Data Retention**
   - What we know: Plausible Cloud retains data indefinitely, Umami Cloud free tier retains 6 months, Fathom retains indefinitely
   - What's unclear: Whether 6-month retention is sufficient for "trends over time" success criteria, or if unlimited retention is required
   - Recommendation: Clarify retention requirements with stakeholders; if >6 months needed, choose Plausible Cloud, Fathom, or self-hosted Umami

4. **Offline Queue Storage Limits**
   - What we know: Background Sync queues failed requests in IndexedDB with configurable retention time (default 24 hours)
   - What's unclear: Storage quota limits in browsers for queued analytics events, behavior when quota exceeded
   - Recommendation: Implement queue monitoring and fallback to discard oldest events if storage quota warning is triggered; analytics data loss is preferable to app storage failure

5. **Cross-Device Journey Tracking**
   - What we know: Cookie-free analytics track daily unique visitors, not persistent user identities
   - What's unclear: How to track user journey across devices (e.g., discovers app on desktop, uses on mobile) without user accounts
   - Recommendation: Accept limitation of cookie-free approach; cross-device tracking would require user accounts or privacy-invasive fingerprinting. Track device-specific journeys as separate funnels.
