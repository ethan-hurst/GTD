## What Was Built

### Analytics Foundation

1. **Configuration Module** (`src/lib/analytics/config.ts`)
   - Reads `VITE_PLAUSIBLE_DOMAIN` from environment (defaults to 'localhost')
   - Reads `VITE_PLAUSIBLE_API_HOST` for optional proxy configuration
   - Clean environment variable access with sensible defaults

2. **Event Tracking Module** (`src/lib/analytics/events.ts`)
   - Typed `ANALYTICS_EVENTS` constants for all custom events:
     - Onboarding: `onboarding_started`, `onboarding_completed`
     - Tasks: `task_created`, `task_completed`, `task_deleted`
     - Sync: `sync_initiated`, `sync_completed`
     - Review: `review_started`, `review_completed`
     - Engagement: `feature_discovered`
   - `trackEvent(name, props?)` function with SSR safety and error handling
   - `analytics` convenience object with named methods (e.g., `analytics.taskCreated('inbox')`)
   - TypeScript declaration for `window.plausible` global

3. **PlausibleAnalytics Component** (`src/lib/components/PlausibleAnalytics.svelte`)
   - Dynamically injects Plausible script tag with domain configuration
   - Automatically tracks SPA route changes as pageviews using `afterNavigate`
   - Supports custom API host for proxy configurations
   - Script cleanup on component unmount

4. **Root Layout Integration** (`src/routes/+layout.svelte`)
   - PlausibleAnalytics component renders before Toaster (top of render tree)
   - Receives domain and API host from config module
   - Tracks all pageviews and route changes automatically

5. **Offline Analytics Queue** (`src/service-worker.js`)
   - IndexedDB-based queue (`analytics-offline` database)
   - Intercepts failed `plausible.io/api/event` POST requests
   - Queues failed requests with metadata (url, body, headers, timestamp)
   - Replays queue on service worker activation and online events
   - 24-hour retention policy (discards older entries)
   - Returns 202 Accepted for queued requests (prevents client retries)

## Decisions Made

### 1. Manual Plausible Integration (not @accuser package)

**Context:** Plan specified `@accuser/svelte-plausible-analytics` package, but it has peer dependency on Svelte 4.

**Decision:** Implement manual Plausible integration with custom component.

**Rationale:**
- Package incompatibility with Svelte 5 (peer dependency conflict)
- Manual integration is straightforward (script injection + route tracking)
- Avoids `--legacy-peer-deps` workarounds and potential runtime issues
- Full control over implementation and updates

**Impact:** Custom PlausibleAnalytics component, but cleaner dependency tree and guaranteed compatibility.

### 2. Vanilla IndexedDB for Analytics Queue (not Workbox)

**Context:** Service worker needs offline queuing for analytics events.

**Decision:** Implement queue using raw IndexedDB API directly in service worker.

**Rationale:**
- Current service worker is vanilla JS (no Workbox)
- Adding Workbox just for analytics would increase bundle size significantly
- Analytics queue is simple (store failed requests, replay on reconnect)
- IndexedDB provides all needed functionality

**Impact:** Lightweight implementation (~150 lines) vs. heavy Workbox dependency.

### 3. 24-Hour Retention for Queued Events

**Context:** Offline analytics queue needs retention policy.

**Decision:** Discard events older than 24 hours during replay.

**Rationale:**
- Most offline periods are hours, not days
- Analytics events lose value after 24 hours (pageviews, session tracking)
- Prevents unbounded queue growth
- Balances retry persistence with storage efficiency

**Impact:** Events queued for >24 hours are lost, but this is acceptable for analytics use case.
