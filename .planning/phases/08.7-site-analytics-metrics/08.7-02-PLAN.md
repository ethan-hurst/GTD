   - Call `analytics.taskCreated('inbox')` after a new item is successfully added to the inbox. Look for the submit/add handler that calls the database operation, and add the analytics call AFTER the DB write succeeds.
   - This is the key activation event — "first task created" is the conversion moment.

3. Modify `src/lib/components/ProcessingFlow.svelte`:
   - Add import: `import { analytics } from '$lib/analytics/events';`
   - When an item is processed into a specific category (next action, project, waiting for, someday/maybe, reference, trash), call `analytics.taskCompleted()`. Look for the handler that finalizes processing and transitions the item. A single `taskCompleted()` call at the processing completion point is sufficient — do not track each individual processing decision.
   - If there's a "delete/trash" outcome in processing, call `analytics.taskDeleted()` instead of `taskCompleted()` for that path.

IMPORTANT: All analytics calls must be non-blocking — they should not be awaited and should not affect the user flow. Place them AFTER the primary action succeeds, never before. If the analytics call fails silently (which it will, thanks to the try/catch in trackEvent), the user action must still complete successfully.

IMPORTANT: Read each component file FIRST to understand its structure before modifying. Look for existing patterns (onMount, event handlers, state transitions) and add analytics calls at the natural integration points.
