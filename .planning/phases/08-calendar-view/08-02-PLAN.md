  <files>src/lib/utils/ics-parser.ts</files>
  <action>
Create `src/lib/utils/ics-parser.ts` that wraps ical.js to parse .ics file content into CalendarEvent objects.

```typescript
import ICAL from 'ical.js';
import type { CalendarEvent } from '$lib/db/schema';
```

Export a function `parseICS(icsText: string, source?: string): Omit<CalendarEvent, 'id' | 'created' | 'modified'>[]` that:

1. Parses the ICS text using `ICAL.parse()` to get a jCal data structure
2. Creates a `ICAL.Component` from the parsed data
3. Iterates over all `VEVENT` subcomponents
4. For each VEVENT, extracts:
   - `title` from SUMMARY property
   - `startTime` from DTSTART (convert ICAL.Time to JS Date)
   - `endTime` from DTEND (convert ICAL.Time to JS Date, fall back to startTime + 1 hour if missing)
   - `allDay` from DTSTART value type (DATE vs DATE-TIME)
   - `location` from LOCATION property (optional)
   - `notes` from DESCRIPTION property (optional)
   - `rrule` from RRULE property (store as string if present)
   - `exceptionDates` from EXDATE properties (collect as ISO date strings)
   - `source` parameter passed in (defaults to 'ics-import')
5. Returns array of CalendarEvent objects (without id, created, modified — those are added by operations.ts)

Handle edge cases:
- Missing DTEND: default to startTime + 1 hour for timed events, startTime + 1 day for all-day events
- Missing SUMMARY: default to "(No title)"
- VTIMEZONE handling: ical.js handles timezone conversion natively via ICAL.TimezoneService
- Multiple VEVENTs in one file: return all of them
- Invalid/malformed ICS: wrap in try/catch, throw descriptive error

Also export a helper `validateICS(icsText: string): boolean` that returns true if the text starts with "BEGIN:VCALENDAR" (basic validation before full parse).

Note on ical.js import: The library may need `import ICAL from 'ical.js'` or `import * as ICAL from 'ical.js'` depending on its module export. Check node_modules/ical.js after install. If it uses default export, use default import. If not, use namespace import.

<task type="auto">
  <name>Task 2: Create recurrence expansion utility</name>
  <files>src/lib/utils/recurrence.ts</files>
  <action>
Create `src/lib/utils/recurrence.ts` that wraps rrule.js to expand recurrence rules into individual event instances.

```typescript
import { RRule, RRuleSet } from 'rrule';
import type { CalendarEvent } from '$lib/db/schema';
```

Export function `expandRecurrence(event: CalendarEvent, rangeStart: Date, rangeEnd: Date): CalendarEvent[]` that:

1. If event has no `rrule` string, returns `[event]` (single non-recurring event)
2. Parses the RRULE string using `RRule.fromString(event.rrule)` — note: the rrule.js library expects the RRULE string without the "RRULE:" prefix, so strip it if present
3. Creates an `RRuleSet` to handle exception dates:
   - Add the RRule to the set
   - If `event.exceptionDates` exists, add each as an exclusion date via `rruleSet.exdate()`
4. Use `rruleSet.between(rangeStart, rangeEnd, true)` to get all occurrence dates within the visible range
5. For each occurrence date, create a new CalendarEvent object:
   - Copy all fields from the base event
   - Set `startTime` to the occurrence date
   - Compute `endTime` by adding the original event duration (endTime - startTime) to the occurrence date
   - Set `recurrenceId` to the base event's id (so UI knows it's an expanded instance)
   - Keep the same `id` as base event (the UI layer will handle distinguishing instances by start date)
6. Return array of expanded CalendarEvent instances

Also export `expandAllRecurrences(events: CalendarEvent[], rangeStart: Date, rangeEnd: Date): CalendarEvent[]` that:
1. Separates events into recurring and non-recurring
2. Filters non-recurring events that fall within the range
3. Expands recurring events using `expandRecurrence`
4. Returns combined array, sorted by startTime

Important: Do NOT store expanded instances in the database. Expansion is computed on-demand for the visible date range only. This is a pure function.

Note on rrule.js: The library requires Date objects in UTC. Use `RRule.fromString()` or construct RRule with `dtstart` option set to the event's startTime. Ensure timezone handling is consistent — convert to UTC before passing to rrule, convert back for display.

Exception dates are respected. Duration is preserved across occurrences. Expansion is pure (no DB writes).</done>
