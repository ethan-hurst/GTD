# Phase 08.5 Plan 01: Soft-Delete Tombstones and Sync Types Summary

**One-liner:** Soft-delete tombstone support with query filtering and complete sync type definitions for encrypted per-record merge.

## What Was Built

### Schema Changes (Dexie v7)

Extended three core interfaces with soft-delete tombstone fields:

**GTDItem:**
- `deleted?: boolean` - Tombstone marker
- `deletedAt?: Date` - Deletion timestamp for compaction

**CalendarEvent:**
- `deleted?: boolean` - Tombstone marker
- `deletedAt?: Date` - Deletion timestamp for compaction

**Context:**
- `deleted?: boolean` - Tombstone marker
- `deletedAt?: Date` - Deletion timestamp for compaction

**Dexie Schema Migration:**
- Migrated to version 7
- Added `deleted` field to indexed fields on items table
- Added `deleted` field to indexed fields on events table
- Migration is automatic on next app load via Dexie's version system

### Soft-Delete Implementation

**Converted three delete operations to soft-delete:**

1. `deleteItem(id)` - Updates item with `{ deleted: true, deletedAt: now, modified: now }`
2. `deleteEvent(id)` - Updates event with `{ deleted: true, deletedAt: now, modified: now }`
3. `deleteContext(id)` - Clears references, then updates context with `{ deleted: true, deletedAt: now }`
4. `bulkDeleteItems(ids)` - Batch soft-delete for multiple items

**Zero breaking changes:** All function signatures remain identical. Calling code unchanged.

### Query Filtering

**Updated 13 query functions to filter out deleted records:**

**Items:**
- `getAllInbox()` - Added `.filter(item => !item.deleted)`
- `getItemsByType(type)` - Added `.filter(item => !item.deleted)`
- `getAllNextActions()` - Added `!item.deleted` to existing filter
- `getActionsByContext(contexts)` - Added `!item.deleted` to filter
- `getAllProjects()` - Added `!item.deleted` to filter
- `getActionsByProject(projectId)` - Added `!item.deleted` to filter
- `getStalledProjects()` - Added `!item.deleted` to both project and action filters
- `getAllWaitingFor()` - Added `!item.deleted` to filter
- `getAllSomedayMaybe()` - Added `!item.deleted` to filter

**Contexts:**
- `getAllContexts()` - Added `.filter(ctx => !ctx.deleted)`

**Events:**
- `getEventsInRange(start, end)` - Added `.filter(e => !e.deleted)`
- `getRecurringEvents()` - Added `!event.deleted` to filter
- `getAllEvents()` - Added `.filter(e => !e.deleted)`

### Tombstone Compaction

**New function:** `compactTombstones(maxAgeDays = 30)`

- Hard-deletes items/events/contexts where `deleted === true` and `deletedAt` is older than threshold
- Returns count of compacted records
- Default 30-day retention window allows month-long sync window
- Run periodically to prevent tombstone accumulation

**Implementation:**
- Batch query deleted records older than cutoff
- Use `bulkDelete` for efficient removal
- Process items, events, and contexts separately
- Return total count for observability

### Sync Type Definitions

**Created `src/lib/sync/types.ts` with complete type system:**

**State Machine:**
- `SyncState` - `'idle' | 'pulling' | 'pushing' | 'merging' | 'error'`

**All types exported and ready for crypto, merge, and sync modules.**

## Technical Insights

### Soft-Delete Pattern

**Why soft-delete instead of hard-delete:**
- Sync requires propagating deletions across devices
- Without tombstones, Device A can't tell Device B that item was deleted
- Deleted items must survive long enough for all paired devices to sync

**Query filtering impact:**
- Every query now filters `!item.deleted`
- Minimal performance impact (indexed field)
- Clean separation: deleted items invisible to UI, but present for sync

### Tombstone Retention Window

**30-day default balances two concerns:**
1. **Sync window:** Devices that haven't synced in 30 days will still receive deletions
2. **Storage efficiency:** Prevents unbounded tombstone accumulation

**Configurable via `compactTombstones(maxAgeDays)`:**
- Weekly Review could call with default 30 days
- Manual cleanup could call with shorter window
- Enterprise deployments could extend to 90+ days

### Type System Design

**Sync types provide shared contract:**
- Crypto module encrypts/decrypts EncryptedBlob
- Merge module resolves conflicts in SyncPayload
- Sync module orchestrates state machine via SyncStatus

**Separation of concerns:**
- Types define protocol, not implementation
- Each module can evolve independently
- Clear interfaces for testing

## Performance Impact

**Query Performance:**
- Added `!item.deleted` filter to 13 functions
- `deleted` field is indexed - minimal performance impact
- Filter happens in-memory after index lookup (Dexie pattern)

**Storage Impact:**
- Tombstones accumulate until `compactTombstones` called
- Worst case: 30 days of deletion history
- Typical user: <100 deleted items = negligible storage

**Migration Performance:**
- Dexie v6 → v7 migration is instant (adds index, no data changes)
- Existing records have `deleted: undefined` (falsy, correct behavior)

## Lessons Learned

**TypeScript strictness caught edge cases:**
- Initial `item.deleted === true` failed type checking
- Changed to `!!item.deleted` for truthy check
- Prevents bugs where `deleted: undefined` could be mishandled

**Optional fields are cleaner than nullable:**
- `deleted?: boolean` is cleaner than `deleted: boolean | null`
- Undefined = not deleted (default state)
- True = deleted (explicit state)
- No need to handle null case

**Zero breaking changes was critical:**
- 30+ components call delete operations
- Changing signatures would require massive refactor
- Internal behavior change preserved all calling code
