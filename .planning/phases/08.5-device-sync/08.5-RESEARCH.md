# Phase 08.5: Device Sync - Research

**Researched:** 2026-01-31
**Domain:** Client-side device sync with encrypted cloud storage (Netlify Functions + Blobs, Web Crypto API, IndexedDB merge)
**Confidence:** MEDIUM

## Summary

This research investigated the implementation approach for multi-device sync using Netlify Functions + Netlify Blobs as the backend, client-side encryption with Web Crypto API (AES-GCM + PBKDF2), and per-record merge strategies for IndexedDB. The planned architecture involves pairing devices with a short code, auto-syncing on app open, and encrypting data before upload.

The standard approach uses Netlify Blobs as a zero-configuration key-value store accessed from serverless functions, Web Crypto API's built-in SubtleCrypto interface for AES-GCM authenticated encryption with PBKDF2 key derivation, and last-write-wins (LWW) timestamp-based merge for conflict resolution. The main architectural challenge is preventing nonce reuse in AES-GCM encryption across multiple devices, handling clock skew in timestamp-based merging, and managing soft-delete tombstones for deletion propagation.

Key findings include: Netlify Blobs has individual object size limit of 5 GB with 100 GB free tier (unofficial pricing), Web Crypto API requires secure context (HTTPS) and unique IV for every AES-GCM encryption, PBKDF2 should use minimum 100,000 iterations with SHA-256, and debouncing sync operations should combine time-based and count-based thresholds to optimize API calls.

**Primary recommendation:** Use Netlify Blobs for ephemeral sync storage, implement counter-based IV generation to prevent nonce reuse, derive encryption keys from pairing code using PBKDF2 with 100k+ iterations, and implement last-write-wins merge by comparing record `modified` timestamps with tombstone tracking for deletions.

### Pattern 2: AES-GCM Encryption with PBKDF2 Key Derivation

**What:** Derive encryption key from pairing code, encrypt data with AES-GCM

**When to use:** Client-side encryption before uploading to cloud storage

**Example:**
```typescript
// Source: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto/deriveKey
// src/lib/sync/crypto.ts

async function deriveKey(pairingCode: string, salt: Uint8Array): Promise<CryptoKey> {
  // Import pairing code as key material
  const enc = new TextEncoder();
  const keyMaterial = await crypto.subtle.importKey(
    'raw',
    enc.encode(pairingCode),
    'PBKDF2',
    false,
    ['deriveBits', 'deriveKey']
  );

  // Derive AES-GCM key using PBKDF2
  return crypto.subtle.deriveKey(
    {
      name: 'PBKDF2',
      salt,
      iterations: 100000, // NIST minimum recommendation
      hash: 'SHA-256'
    },
    keyMaterial,
    { name: 'AES-GCM', length: 256 },
    false, // not extractable
    ['encrypt', 'decrypt']
  );
}

async function encrypt(data: string, key: CryptoKey, iv: Uint8Array): Promise<ArrayBuffer> {
  const enc = new TextEncoder();
  return crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    enc.encode(data)
  );
}
```

### Pattern 3: Last-Write-Wins Merge with Tombstones

**What:** Compare timestamps to resolve conflicts, track deletions with tombstone records

**When to use:** Syncing IndexedDB data across multiple devices without server-side merge logic

**Example:**
```typescript
// Source: Pattern synthesized from https://app.studyraid.com/en/read/11356/355149/conflict-resolution-strategies
// src/lib/sync/merge.ts

interface SyncRecord {
  id: number;
  modified: Date;
  deleted?: boolean; // Tombstone flag
  [key: string]: any;
}

function mergeRecords(local: SyncRecord[], remote: SyncRecord[]): SyncRecord[] {
  const merged = new Map<number, SyncRecord>();

  // Add all local records
  for (const record of local) {
    merged.set(record.id, record);
  }

  // Merge remote records using last-write-wins
  for (const remoteRecord of remote) {
    const localRecord = merged.get(remoteRecord.id);

    if (!localRecord || remoteRecord.modified > localRecord.modified) {
      // Remote is newer, use it
      merged.set(remoteRecord.id, remoteRecord);
    }
  }

  // Filter out tombstones (deleted records)
  return Array.from(merged.values()).filter(r => !r.deleted);
}
```

### Pattern 4: Debounced Sync with Combined Thresholds

**What:** Debounce sync calls using both time delay and change count thresholds

**When to use:** Reducing API calls while ensuring timely sync after batch edits

**Example:**
```typescript
// Source: Inspired by https://shaxadd.medium.com/tanstack-pacer-solving-debounce-throttle-and-batching-the-right-way-94d699befc8a
// src/lib/sync/sync.ts

class SyncDebouncer {
  private changeCount = 0;
  private timeoutId: number | null = null;
  private readonly MAX_CHANGES = 5;
  private readonly MAX_DELAY_MS = 2000;

  queueSync(syncFn: () => Promise<void>) {
    this.changeCount++;

    // Clear existing timeout
    if (this.timeoutId) clearTimeout(this.timeoutId);

    // Sync immediately if change threshold reached
    if (this.changeCount >= this.MAX_CHANGES) {
      this.executeSync(syncFn);
      return;
    }

    // Otherwise, debounce with time delay
    this.timeoutId = setTimeout(() => {
      this.executeSync(syncFn);
    }, this.MAX_DELAY_MS);
  }

  private async executeSync(syncFn: () => Promise<void>) {
    this.changeCount = 0;
    this.timeoutId = null;
    await syncFn();
  }
}
```

### Pattern 5: Counter-Based IV Generation (Prevent Nonce Reuse)

**What:** Use a deterministic counter for IV/nonce instead of random generation to prevent collisions

**When to use:** Multi-device scenarios where nonce reuse could occur with random IVs

**Example:**
```typescript
// Source: https://news.ycombinator.com/item?id=40653125
// src/lib/sync/crypto.ts

class IVCounter {
  private counter = 0n;

  async loadCounter(db: Dexie) {
    // Load counter from IndexedDB settings
    const setting = await db.settings.get({ key: 'iv-counter' });
    this.counter = setting ? BigInt(setting.value) : 0n;
  }

  async getNextIV(db: Dexie): Promise<Uint8Array> {
    this.counter++;

    // Persist incremented counter
    await db.settings.put({
      key: 'iv-counter',
      value: this.counter.toString(),
      updatedAt: new Date()
    });

    // Convert counter to 12-byte IV (96 bits)
    const iv = new Uint8Array(12);
    const view = new DataView(iv.buffer);
    view.setBigUint64(0, this.counter, false); // Big-endian
    return iv;
  }
}
```

### Anti-Patterns to Avoid

- **Random IVs without collision tracking:** Random 96-bit IVs have 50% collision probability after 2^48 messages. Use counter-based IVs or track used IVs.
- **Storing salt/IV separately from ciphertext:** Salt and IV must be transmitted/stored alongside encrypted data for decryption. Store as single blob.
- **Sync on every keystroke:** Debounce aggressively to prevent excessive API calls and blob storage operations.
- **Hard-coding pairing code:** Never store pairing code in code or config. Generate dynamically and store in IndexedDB settings.
- **Using Date.now() for timestamps:** Client clocks may be skewed. Consider adding logical clock (Lamport timestamp) if clock skew becomes an issue.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| AES encryption | Custom crypto library | Web Crypto API (crypto.subtle) | Browser-native, audited, FIPS-compliant, no bundle size |
| Key derivation from password | SHA-256 hash of pairing code | PBKDF2 with 100k+ iterations | Prevents brute-force, industry standard (NIST SP 800-132) |
| Blob storage authentication | Custom API keys, JWT tokens | Netlify Blobs (implicit auth) | Zero-config, automatic in Netlify Functions |
| JSON serialization for sync | Custom DB walker | Existing `exportDatabase()` function | Already handles all tables, Date serialization |
| Random number generation | Math.random() | crypto.getRandomValues() | Cryptographically secure, required for salt/IV |

**Key insight:** Cryptography is notoriously easy to get wrong. Using browser-native Web Crypto API eliminates entire classes of vulnerabilities (timing attacks, weak RNG, incorrect padding) and requires zero dependencies. The only custom code needed is business logic (merge strategy, pairing flow).

## Common Pitfalls

### Pitfall 1: IV/Nonce Reuse in AES-GCM

**What goes wrong:** Reusing the same IV with the same key in AES-GCM is catastrophic - it allows trivial recovery of the authentication key and message content.

**Why it happens:** Random IV generation seems safe but has birthday paradox collision risk. With 96-bit IVs, collision probability reaches 50% after 2^48 encryptions. Multiple devices encrypting independently increases collision risk.

**How to avoid:** Use counter-based IV generation stored in IndexedDB. Increment counter for each encryption and persist to database. Counter-based IVs eliminate collision risk entirely.

**Warning signs:**
- Using `crypto.getRandomValues()` for IV without tracking used IVs
- Not persisting IV state between sessions
- Generating IV independently on each device

### Pitfall 2: Insufficient PBKDF2 Iterations

**What goes wrong:** Using default or low iteration counts (e.g., 10,000) makes pairing codes vulnerable to brute-force attacks.

**Why it happens:** Higher iterations slow down legitimate users, so developers choose low values for performance.

**How to avoid:** Use minimum 100,000 iterations (OWASP 2023 recommendation was 600,000 for PBKDF2-HMAC-SHA256). Test on target devices to ensure acceptable performance (should be <500ms). 6-character alphanumeric pairing code has only ~2 billion combinations - strong KDF is essential.

**Warning signs:**
- Iteration count below 100,000
- No performance testing on mobile devices
- Choosing iteration count "because it feels fast enough"

### Pitfall 3: Clock Skew in Timestamp-Based Merge

**What goes wrong:** Devices with incorrect clocks cause newer changes to be overwritten by older ones, or prevent sync from working at all.

**Why it happens:** Client clocks drift or are manually set incorrectly. Timestamp comparison assumes synchronized clocks.

**How to avoid:**
- Accept that some clock skew is unavoidable with client-side timestamps
- Use `modified` field as best-effort ordering, not absolute truth
- Consider adding Lamport timestamp (logical clock) if issues arise
- Document limitation: "Ensure device clocks are reasonably accurate"

**Warning signs:**
- Users report "changes disappearing" after sync
- Sync behavior differs drastically between devices
- Modified timestamps in the future or distant past

### Pitfall 4: Forgetting to Store Salt and IV

**What goes wrong:** Encrypted data cannot be decrypted without the salt (for key derivation) and IV (for AES-GCM). If lost, data is permanently unrecoverable.

**Why it happens:** Developers treat salt/IV as "configuration" rather than per-ciphertext data.

**How to avoid:** Store salt and IV in the same blob as ciphertext. Use consistent format:
```typescript
interface EncryptedBlob {
  salt: string;        // Base64-encoded salt (16 bytes)
  iv: string;          // Base64-encoded IV (12 bytes)
  ciphertext: string;  // Base64-encoded encrypted data
}
```

**Warning signs:**
- Salt/IV stored in separate IndexedDB records
- Using fixed/global salt or IV
- Not testing decryption in isolation

### Pitfall 5: Exposing Pairing Code After Initial Pairing

**What goes wrong:** Storing pairing code in plaintext settings makes it accessible to XSS attacks or local file access.

**Why it happens:** Need to re-derive key on each sync, so pairing code must be stored somewhere.

**How to avoid:**
- Hash pairing code (SHA-256) for blob store key (device ID)
- Store derived CryptoKey in memory during session
- Re-prompt for pairing code if key lost (on page refresh) OR store encrypted with browser-managed key
- Never log pairing code to console or analytics

**Warning signs:**
- Pairing code visible in DevTools Application tab (IndexedDB/localStorage)
- Pairing code in URL parameters or visible in UI after pairing
- No "unpair device" function to revoke access

### Pitfall 6: Netlify Blobs Size Limits

**What goes wrong:** Individual blobs cannot exceed 5 GB. For GTD app, this is unlikely, but repeated syncs with tombstones could grow payload size.

**Why it happens:** Tombstones are never removed, deleted records accumulate over time.

**How to avoid:**
- Implement tombstone compaction: remove tombstones older than 30 days
- Monitor blob size in development (log payload size before upload)
- Set hard limit: fail sync if payload exceeds 10 MB (indicates issue)

**Warning signs:**
- Blob size grows monotonically over time
- Sync payload contains thousands of deleted records
- Sync becomes noticeably slower over time

## Code Examples

Verified patterns from official sources:

### Pairing Code Generation

```typescript
// Source: crypto.getRandomValues() - https://developer.mozilla.org/en-US/docs/Web/API/Crypto/getRandomValues
function generatePairingCode(): string {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'; // No ambiguous chars (0,O,1,I)
  const length = 6;
  const randomValues = crypto.getRandomValues(new Uint8Array(length));

  return Array.from(randomValues)
    .map(val => chars[val % chars.length])
    .join('');
}

// Hash pairing code for blob storage key (device ID)
async function hashPairingCode(pairingCode: string): Promise<string> {
  const enc = new TextEncoder();
  const hashBuffer = await crypto.subtle.digest('SHA-256', enc.encode(pairingCode));
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}
```

### Complete Encrypt/Decrypt Flow

```typescript
// Source: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto/encrypt
async function encryptData(pairingCode: string, data: string): Promise<string> {
  // Generate salt for PBKDF2
  const salt = crypto.getRandomValues(new Uint8Array(16));

  // Derive key from pairing code
  const key = await deriveKey(pairingCode, salt);

  // Get next IV from counter
  const ivCounter = new IVCounter();
  await ivCounter.loadCounter(db);
  const iv = await ivCounter.getNextIV(db);

  // Encrypt data
  const ciphertext = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    new TextEncoder().encode(data)
  );

  // Return as JSON with salt, IV, and ciphertext
  return JSON.stringify({
    salt: btoa(String.fromCharCode(...salt)),
    iv: btoa(String.fromCharCode(...iv)),
    ciphertext: btoa(String.fromCharCode(...new Uint8Array(ciphertext)))
  });
}

async function decryptData(pairingCode: string, encryptedBlob: string): Promise<string> {
  const { salt, iv, ciphertext } = JSON.parse(encryptedBlob);

  // Decode Base64 to Uint8Array
  const saltBytes = Uint8Array.from(atob(salt), c => c.charCodeAt(0));
  const ivBytes = Uint8Array.from(atob(iv), c => c.charCodeAt(0));
  const ciphertextBytes = Uint8Array.from(atob(ciphertext), c => c.charCodeAt(0));

  // Derive key from pairing code
  const key = await deriveKey(pairingCode, saltBytes);

  // Decrypt data
  const plaintext = await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv: ivBytes },
    key,
    ciphertextBytes
  );

  return new TextDecoder().decode(plaintext);
}
```

### Soft-Delete Tombstone Implementation

```typescript
// Source: https://docs.couchbase.com/sync-gateway/current/manage/managing-tombstones.html
// Add to schema.ts interface:
interface GTDItem {
  id: number;
  title: string;
  // ... existing fields
  deleted?: boolean;     // Tombstone flag
  deletedAt?: Date;      // When deleted (for compaction)
}

// Instead of db.items.delete(id):
async function softDeleteItem(id: number) {
  await db.items.update(id, {
    deleted: true,
    deletedAt: new Date(),
    modified: new Date()
  });
}

// Filter out deleted items in queries:
const activeItems = await db.items
  .where('deleted').notEqual(true)
  .toArray();

// Tombstone compaction (run periodically):
async function compactTombstones() {
  const thirtyDaysAgo = new Date(Date.now() - 30 * 24 * 60 * 60 * 1000);
  const oldTombstones = await db.items
    .where('deleted').equals(true)
    .and(item => item.deletedAt && item.deletedAt < thirtyDaysAgo)
    .toArray();

  await db.items.bulkDelete(oldTombstones.map(item => item.id));
}
```

### Netlify Function Implementation

```typescript
// Source: https://docs.netlify.com/build/data-and-storage/netlify-blobs/
// netlify/functions/sync-push.mts
import { getStore } from '@netlify/blobs';
import type { Context } from '@netlify/functions';

export default async (req: Request, context: Context) => {
  if (req.method !== 'POST') {
    return new Response('Method not allowed', { status: 405 });
  }

  try {
    const { deviceId, encryptedBlob } = await req.json();

    if (!deviceId || !encryptedBlob) {
      return new Response('Missing deviceId or encryptedBlob', { status: 400 });
    }

    const store = getStore('sync-data');
    await store.set(deviceId, encryptedBlob);

    return new Response(JSON.stringify({
      success: true,
      timestamp: new Date().toISOString()
    }), {
      status: 200,
      headers: { 'Content-Type': 'application/json' }
    });
  } catch (error) {
    return new Response(JSON.stringify({
      error: error.message
    }), {
      status: 500,
      headers: { 'Content-Type': 'application/json' }
    });
  }
};

// netlify/functions/sync-pull.mts
import { getStore } from '@netlify/blobs';
import type { Context } from '@netlify/functions';

export default async (req: Request, context: Context) => {
  if (req.method !== 'GET') {
    return new Response('Method not allowed', { status: 405 });
  }

  try {
    const url = new URL(req.url);
    const deviceId = url.searchParams.get('deviceId');

    if (!deviceId) {
      return new Response('Missing deviceId parameter', { status: 400 });
    }

    const store = getStore('sync-data');
    const encryptedBlob = await store.get(deviceId);

    if (!encryptedBlob) {
      return new Response(JSON.stringify({
        error: 'No data found for device'
      }), {
        status: 404,
        headers: { 'Content-Type': 'application/json' }
      });
    }

    return new Response(JSON.stringify({
      encryptedBlob,
      timestamp: new Date().toISOString()
    }), {
      status: 200,
      headers: { 'Content-Type': 'application/json' }
    });
  } catch (error) {
    return new Response(JSON.stringify({
      error: error.message
    }), {
      status: 500,
      headers: { 'Content-Type': 'application/json' }
    });
  }
};
```

## Open Questions

Things that couldn't be fully resolved:

1. **Netlify Blobs Pricing**
   - What we know: Unofficial estimate of 100 GB free, $0.09/GB/month beyond that (June 2024 Netlify forum post)
   - What's unclear: Official pricing not published as of January 2026
   - Recommendation: Monitor usage in production. GTD app unlikely to exceed 100 GB (typical user data <1 MB). Document pricing uncertainty for users.

2. **Multi-Device Pairing Topology**
   - What we know: Pairing code creates shared secret for encryption. Hashed pairing code serves as blob storage key (device ID).
   - What's unclear: Should all devices share same blob (overwrite each other), or should each device have separate blob with merge happening client-side?
   - Recommendation: Start with single shared blob (last device to sync wins globally). If issues arise, implement per-device blobs with multi-way merge (more complex).

3. **IV Counter Synchronization Across Devices**
   - What we know: Counter-based IVs prevent nonce reuse, but each device maintains its own counter.
   - What's unclear: Two devices encrypting simultaneously could use same counter value if not coordinated.
   - Recommendation: Prefix counter with device-specific offset (e.g., Device A uses counters 0-1M, Device B uses 1M-2M). Derive offset from hashed pairing code + device index.

4. **Handling Schema Migrations During Sync**
   - What we know: Dexie schema currently at version 6. Future migrations may add/remove fields.
   - What's unclear: How to handle syncing data between devices with different schema versions.
   - Recommendation: Include schema version in encrypted blob. On version mismatch, force newer device to downgrade sync (omit new fields) or older device to upgrade (add default values for new fields).
