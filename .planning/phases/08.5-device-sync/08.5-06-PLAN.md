  <name>Task 1: Wire auto-sync on app open and periodic compaction</name>
  <files>src/routes/+layout.svelte</files>
  <action>
In `src/routes/+layout.svelte`:

1. Import `syncState` from `$lib/stores/sync.svelte`.
2. Import `compactTombstones` from `$lib/db/operations`.

3. In the `onMount` callback, add sync initialization AFTER the existing persistence and onboarding setup:
   ```typescript
   // Initialize sync state (non-blocking)
   (async () => {
     await syncState.init();

     // Auto-sync on app open if paired and has pairing code
     if (syncState.isPaired && syncState.hasPairingCode()) {
       await syncState.sync();
     }

     // Run tombstone compaction (non-blocking, runs once on app open)
     compactTombstones(30).catch(err => {
       console.warn('Tombstone compaction failed:', err);
     });
   })();
   ```

4. Key behaviors:
   - `syncState.init()` loads pairing info from IndexedDB (sets isPaired, deviceId, lastSyncTime)
   - Auto-sync only runs if paired AND pairing code is in memory (first session after pairing OR after re-entry)
   - If pairing code is not in memory (page refresh without re-entry), sync does NOT run automatically - user sees "re-enter code" prompt in Settings
   - Tombstone compaction runs in background, errors are logged but never block the app
   - All sync operations are wrapped in async IIFEs to avoid blocking the UI

5. Add visibility change listener to sync when app becomes visible (user switches back to tab):
   ```typescript
   function handleVisibilityChange() {
     if (document.visibilityState === 'visible' && syncState.isPaired && syncState.hasPairingCode()) {
       syncState.sync();
     }
   }
   document.addEventListener('visibilitychange', handleVisibilityChange);
   ```
   Clean up in the onMount return function: `document.removeEventListener('visibilitychange', handleVisibilityChange)`.
  </action>

<task type="auto">
  <name>Task 2: Hook database operations to trigger debounced sync push</name>
  <files>src/lib/db/operations.ts, src/lib/sync/sync.ts</files>
  <action>
In `src/lib/db/operations.ts`:

1. Add a sync notification mechanism. We CANNOT directly import `syncState` into operations.ts because that would create a circular dependency (operations -> stores -> operations). Instead, use a simple event-based approach:

   At the top of operations.ts, add:
   ```typescript
   // Sync notification callback - set by sync store to avoid circular imports
   let onDataChanged: (() => void) | null = null;

   export function setSyncNotifier(callback: () => void): void {
     onDataChanged = callback;
   }

   function notifyDataChanged(): void {
     if (onDataChanged) onDataChanged();
   }
   ```

2. Add `notifyDataChanged()` calls at the end of every data-modifying operation:
   - `addItem` - after db.items.add
   - `updateItem` - after db.items.update
   - `deleteItem` (soft-delete) - after db.items.update
   - `bulkDeleteItems` - after all updates
   - `addContext` - after db.contexts.add
   - `updateContext` - after db.contexts.update
   - `deleteContext` (soft-delete) - after db.contexts.update
   - `completeAction` - after db.items.update (before returning undo fn)
   - `undoCompleteAction` - after db.items.update
   - `reorderActions` - after all updates
   - `bulkCompleteActions` - after all updates
   - `completeProject` - after db.items.update
   - `moveProjectToSomeday` - after db.items.update
   - `addProject` - after db.items.add
   - `resolveWaitingFor` - after db.items.update
   - `addWaitingFor` - after db.items.add
   - `promoteSomedayToActive` - after db.items.update
   - `addSomedayItem` - after db.items.add
   - `setSetting` - after db.settings update (but NOT for sync-related settings to prevent infinite loop)
   - `addEvent` - after db.events.add
   - `updateEvent` - after db.events.update
   - `deleteEvent` (soft-delete) - after db.events.update
   - `bulkAddEvents` - after db.events.bulkAdd

   For setSetting, add a guard: only notify if the key does NOT start with 'sync-' to prevent sync operations from triggering more sync operations.

3. In `src/lib/sync/sync.ts`, add a function to wire up the notification:
   ```typescript
   import { setSyncNotifier } from '$lib/db/operations';

   export function initSyncNotifier(): void {
     setSyncNotifier(() => {
       // Only queue sync if we have access to the sync store
       // This is lazy-loaded to avoid import order issues
       import('$lib/stores/sync.svelte').then(({ syncState }) => {
         syncState.queueSync();
       });
     });
   }
   ```

   Then in the sync store's `init()` method (sync.svelte.ts), call `initSyncNotifier()` from sync.ts.

   ALTERNATIVELY (simpler approach): In sync.svelte.ts init(), directly call `setSyncNotifier` with `() => this.queueSync()`:
   ```typescript
   async init(): Promise<void> {
     // ... existing init code ...
     setSyncNotifier(() => this.queueSync());
   }
   ```
   This avoids the dynamic import. Import setSyncNotifier at the top of sync.svelte.ts.
  </action>
