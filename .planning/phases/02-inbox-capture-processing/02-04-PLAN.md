<objective>
Implement the GTD processing workflow so users can process inbox items through the canonical decision tree and route them to appropriate lists.

Purpose: INBX-03 requires guided processing — the heart of the GTD methodology. Items enter the inbox undifferentiated; processing clarifies each into a concrete next action, delegated task, someday/maybe item, or trash. This is what transforms a pile of thoughts into an organized system. The inline expansion UI keeps the user in flow, and auto-advance through items enables efficient batch processing.

Output: ProcessingFlow component with full GTD decision tree, InboxList updated with slide transitions for inline expansion, and inbox page with "Process Inbox" sequential mode.
</objective>

  <name>Task 1: Create ProcessingFlow component with GTD decision tree</name>
  <files>
    src/lib/components/ProcessingFlow.svelte
  </files>
  <action>
    Create `src/lib/components/ProcessingFlow.svelte`:

    **Props:**
    - `item: GTDItem` — the inbox item being processed
    - `onProcessed: () => void` — callback after item is routed (parent calls advanceToNext)

    **State:**
    - `let step = $state<'actionable' | 'two-minute' | 'not-actionable' | 'delegate-or-defer' | 'done'>('actionable')`

    **GTD Decision Tree Flow:**

    Step 1: "Is this actionable?" (step === 'actionable')
    - Display the item title prominently at top
    - If item has notes, show them below title
    - Two buttons:
      - "Yes, it's actionable" -> sets step to 'two-minute'
      - "No, not actionable" -> sets step to 'not-actionable'
    - Styling: buttons side by side, clear visual weight

    Step 2a: "Can you do it in under 2 minutes?" (step === 'two-minute')
    - Two buttons:
      - "Yes, do it now" -> Mark item as done: call deleteItem(item.id) (GTD: if you can do it in 2 min, do it immediately, no need to track). Call onProcessed(). Show toast "Done! Item completed."
      - "No, it takes longer" -> sets step to 'delegate-or-defer'

    Step 2b: "What do you want to do with it?" (step === 'not-actionable')
    - Three buttons:
      - "Trash it" -> call deleteItem(item.id). Call onProcessed(). Toast "Trashed."
      - "Someday/Maybe" -> call updateItem(item.id, { type: 'someday' }). Call onProcessed(). Toast "Moved to Someday/Maybe."
      - "Keep as Reference" -> For now (no reference system yet), call updateItem(item.id, { type: 'someday' }) as placeholder. Toast "Saved for reference." Add a TODO comment: "// TODO Phase 5+: implement reference material category"

    Step 3: "Delegate or defer?" (step === 'delegate-or-defer')
    - Three buttons:
      - "Delegate to someone" -> Show inline input for "Who?" (delegatedTo field). On submit: call updateItem(item.id, { type: 'waiting', delegatedTo: name }). Call onProcessed(). Toast "Moved to Waiting For."
      - "I'll do it next" -> call updateItem(item.id, { type: 'next-action' }). Call onProcessed(). Toast "Added to Next Actions."
      - "It's part of a project" -> call updateItem(item.id, { type: 'next-action' }). Call onProcessed(). Toast "Added to Next Actions." (Project assignment comes in Phase 4)

    **Visual Design:**
    - Each step shows a step indicator / breadcrumb at top: "Step 1 of 3", "Step 2 of 3", etc.
    - Current question in bold, medium text
    - Buttons are clear, well-spaced, with descriptive text
    - Use color-coded buttons:
      - Actionable/positive actions: blue/green
      - Trash/negative: red/gray
      - Neutral/navigation: gray outline
    - "Back" link at each step (except first) to go to previous step
    - Background: slightly different from list items (bg-gray-50 dark:bg-gray-800/50) to visually distinguish the processing area
    - Padding: p-4 with rounded-b-lg

    **Import:**
    - Import toast from 'svelte-5-french-toast' for confirmation toasts
    - Import updateItem, deleteItem from db/operations.ts
    - Import type GTDItem from db/schema.ts
    - Import storageStatus from stores/storage.svelte.ts (call recordSave after each routing)

             />
           </div>
         {/if}

       - When an item is expanded, its row should have visual differentiation:
         - Left border highlight: border-l-2 border-blue-500 (already planned in Plan 02)

       **Processing mode behavior:**
       - When `inboxState.isProcessing` is true, disable checkboxes (no multi-select during sequential processing)
       - Auto-scroll to expanded item (use `scrollIntoView()` on the expanded element)

       **Important slide transition note:**
       - The expanding div MUST have `display: block` (not inline or contents) — this is a known Svelte slide transition requirement
