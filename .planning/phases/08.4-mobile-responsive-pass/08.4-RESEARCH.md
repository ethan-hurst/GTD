# Phase 08.4: Mobile Responsive Pass - Research

**Researched:** 2026-01-31
**Domain:** Mobile-first responsive web design with SvelteKit + Svelte 5 + Tailwind CSS
**Confidence:** HIGH

## Summary

Mobile responsiveness for a SvelteKit app requires coordinating multiple layers: Tailwind CSS for responsive layouts, touch gesture libraries for swipe and long-press interactions, mobile navigation patterns like hamburger drawer and FAB, and careful attention to accessibility including touch target sizes, viewport meta tags, and safe area insets.

The standard approach is mobile-first with Tailwind's breakpoint system, svelte-gestures for touch interactions, svelte-dnd-action for drag-and-drop, and custom Svelte 5 components for navigation drawer and FAB. The existing app already uses Tailwind CSS 4.1.18, Svelte 5, svelte-dnd-action, and event-calendar core with multiple view plugins.

**Primary recommendation:** Build mobile responsiveness using Tailwind's mobile-first breakpoints with custom 360px baseline, svelte-gestures (3KB) for swipe and long-press gestures, existing svelte-dnd-action for drag reordering, and custom Svelte 5 components for hamburger drawer and FAB. Follow WCAG 2.5.8 with 24px minimum touch targets and use safe-area-inset for notch and home indicator support.

### Pattern 1: Mobile-First Breakpoints with Custom Baseline
**What:** Tailwind uses mobile-first breakpoints where styles apply from breakpoint upward. Customize breakpoints to match project requirements.
**When to use:** Always for mobile-responsive design; required for custom 360px baseline.

**Custom breakpoints configuration:**
```typescript
// tailwind.config.js or app.css
@import "tailwindcss";

@theme {
  --breakpoint-phone: 22.5rem;    /* 360px - minimum supported */
  --breakpoint-phablet: 40rem;    /* 640px - large phones */
  --breakpoint-tablet: 48rem;     /* 768px - tablets */
  --breakpoint-desktop: 64rem;    /* 1024px - desktop */
}
```

**Usage in components:**
```svelte
<!-- Hidden on mobile, visible on tablet+ -->
<aside class="hidden tablet:block">...</aside>

<!-- Full width on phone, half on phablet+, third on tablet+ -->
<div class="w-full phablet:w-1/2 tablet:w-1/3">...</div>
```

**Key principle:** Unprefixed utilities apply to ALL screen sizes (mobile default). Prefixed utilities apply at that breakpoint and above.

### Pattern 2: Conditional Mobile and Desktop Layouts with $state
**What:** Use Svelte 5 $state runes with matchMedia to detect mobile or desktop and render different layouts.
**When to use:** When mobile and desktop UIs diverge significantly like sidebar vs drawer or calendar side panel vs toggle.

**Example:**
```svelte
<script>
  import { onMount } from 'svelte';

  let isMobile = $state(false);

  onMount(() => {
    const mediaQuery = window.matchMedia('(max-width: 1023px)');
    isMobile = mediaQuery.matches;

    const handler = (e) => { isMobile = e.matches; };
    mediaQuery.addEventListener('change', handler);
    return () => mediaQuery.removeEventListener('change', handler);
  });
</script>

{#if isMobile}
  <MobileNav />
{:else}
  <Sidebar />
{/if}
```

### Pattern 5: Touch Gestures with svelte-gestures
**What:** Use svelte-gestures for swipe, long-press, and other touch interactions.
**When to use:** Calendar day navigation with swipe, list item actions with swipe, tooltips with long-press, drag reordering with long-press plus drag.

**Swipe gesture example for calendar day navigation:**
```svelte
<script>
  import { swipe } from 'svelte-gestures';

  function handleSwipe(event) {
    if (event.detail.direction === 'left') {
      // Swipe left: next day
      navigateToNextDay();
    } else if (event.detail.direction === 'right') {
      // Swipe right: previous day
      navigateToPreviousDay();
    }
  }
</script>

<div use:swipe on:swipe={handleSwipe}>
  <!-- Calendar day view content -->
</div>
```

**Long-press gesture example for tooltip or preview:**
```svelte
<script>
  import { press } from 'svelte-gestures';

  let showPreview = $state(false);

  function handlePress() {
    showPreview = true;
  }

  function handleRelease() {
    showPreview = false;
  }
</script>

<div use:press on:press={handlePress} on:pressup={handleRelease}>
  <!-- List item content -->
  {#if showPreview}
    <div class="absolute ...">Preview tooltip</div>
  {/if}
</div>
```

**Swipe actions on list items like Todoist-style:**

```svelte
  <!-- List item content -->

<div class="absolute left-0 ...">✓ Complete</div>
<div class="absolute right-0 ...">🗑️ Delete</div>
```

### Pattern 6: Mobile-Specific Viewport and Safe Areas
**What:** Configure viewport meta tag and use CSS env() for safe area insets.
**When to use:** Always for mobile apps; critical for iPhone X+ and notched Android devices.

**Viewport meta tag in app.html:**
```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">
```

**Safe area insets in components or layouts:**
```svelte
<header class="fixed top-0 left-0 right-0 pt-[env(safe-area-inset-top)] ...">
  <!-- Header content -->
</header>

<nav class="fixed bottom-0 left-0 right-0 pb-[env(safe-area-inset-bottom)] ...">
  <!-- Bottom nav if used -->
</nav>

<button class="fixed bottom-[calc(1.5rem+env(safe-area-inset-bottom))] right-[calc(1.5rem+env(safe-area-inset-right))] ...">
  <!-- FAB with safe area spacing -->
</button>
```

**Fallback for non-supporting browsers:**
```css
/* Tailwind arbitrary values with fallback */
class="pb-[env(safe-area-inset-bottom,0px)]"
```

### Pattern 7: Prevent iOS Input Zoom with 16px Font Size
**What:** Use minimum 16px font size on inputs to prevent iOS Safari auto-zoom on focus.
**When to use:** All form inputs, textareas, and selects.

**CSS approach in global styles:**
```css
input[type="text"],
input[type="email"],
input[type="password"],
input[type="search"],
input[type="number"],
select,
textarea {
  font-size: max(1rem, 16px);
}
```

**Tailwind approach:**
```svelte
<input
  type="text"
  class="text-base"
  placeholder="..." />
```

Note: Tailwind's text-base is 1rem which is 16px by default.

### Pattern 8: Hover State Alternatives for Touch
**What:** Replace hover states with touch-appropriate feedback like active states, ripples, or immediate toggles.
**When to use:** All interactive elements on mobile.

**CSS media query approach:**
```css
/* Only apply hover on devices with hover capability */
@media (hover: hover) and (pointer: fine) {
  .button:hover {
    background-color: #...;
  }
}

/* Touch feedback for all devices */
.button:active {
  transform: scale(0.95);
  background-color: #...;
}
```

**Tailwind approach:**
```svelte
<button class="
  bg-blue-600
  hover:bg-blue-700
  active:bg-blue-800
  active:scale-95
  transition-all
">
  Click me
</button>
```

Note: Tailwind's hover: variant already includes media hover check in v4.

### Anti-Patterns to Avoid

- **Using sm: for mobile styles:** sm: applies at 640px+, NOT on mobile. Use unprefixed utilities for mobile.
- **Disabling zoom with user-scalable=no:** Violates WCAG 2.1; prevents accessibility for low-vision users.
- **Fixed-width layouts on mobile:** Always use w-full or percentage widths; never hardcode pixel widths.
- **Sticky hover states on touch:** iOS keeps hover active after tap; use media hover check or active states instead.
- **Small touch targets less than 24px:** Violates WCAG 2.5.8; always use minimum 44px for important actions, 24px minimum for all interactive elements.
- **Hamburger icon without label:** Many users don't recognize the icon; include Menu text or aria-label.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Touch gestures like swipe, pinch, rotate | Custom touch event handlers | svelte-gestures | Cross-device pointer events, gesture recognition algorithms, debouncing, threshold tuning, 3KB battle-tested library |
| Drag-and-drop with touch | Custom touchstart touchmove handlers | svelte-dnd-action already installed | Handles mouse plus touch, drag handles, drop zones, constraints, accessibility, sorting algorithms |
| Responsive breakpoint detection | Multiple window.matchMedia listeners | Tailwind breakpoints plus conditional state for edge cases | Tailwind handles 95% of cases; only use JS for major layout switches |
| Mobile-first CSS | Desktop-first media queries with max-width | Tailwind's mobile-first breakpoints | Mobile-first is easier to maintain and aligns with progressive enhancement |
| Safe area insets | Manual device detection plus hardcoded padding | CSS env(safe-area-inset-*) | Device-agnostic, future-proof, handles notches and home indicators and dynamic island |

**Key insight:** Mobile interactions have subtle complexity like touch vs mouse events, gesture thresholds, momentum scrolling, and passive event listeners. Libraries like svelte-gestures handle cross-browser quirks and edge cases that custom implementations miss.

## Common Pitfalls

### Pitfall 1: Small Touch Targets
**What goes wrong:** Buttons, links, or interactive elements smaller than 24x24px are difficult to tap accurately, causing user frustration and accidental mis-taps.
**Why it happens:** Desktop designs translate directly to mobile without accounting for finger size vs mouse cursor precision.
**How to avoid:**
- WCAG 2.5.8 Level AA: Minimum 24x24 CSS pixels
- WCAG 2.5.5 Level AAA: Recommended 44x44 CSS pixels
- Use Tailwind: min-h-11 min-w-11 for 44px or min-h-6 min-w-6 for 24px minimum
- Add padding to increase tap area: p-3 on buttons, py-2 on links
**Warning signs:** Users reporting accidental taps, difficulty clicking small icons, need to zoom to tap accurately.

### Pitfall 2: iOS Input Focus Zoom
**What goes wrong:** When user taps an input field with font-size less than 16px, iOS Safari auto-zooms to 16px, disrupting the layout and requiring manual zoom-out.
**Why it happens:** iOS has built-in accessibility feature to make small text readable; triggers when input font-size is less than 16px absolute.
**How to avoid:**
- Set minimum 16px font-size on all inputs: font-size: max(1rem, 16px);
- Use Tailwind text-base which is 16px default on inputs
- Never use maximum-scale=1 or user-scalable=no in viewport meta as it violates accessibility
**Warning signs:** Users report unwanted zoom when tapping inputs, layout shifts on input focus, complaints about zoom behavior.

### Pitfall 3: Sticky Hover on Touch Devices
**What goes wrong:** After tapping an element with hover styles, the hover state remains active until user taps elsewhere, creating confusing visual feedback.
**Why it happens:** Touch devices simulate a hover state on tap; CSS hover pseudo-class stays active after tap.
**How to avoid:**
- Use media hover check with pointer fine to apply hover only on pointer devices
- Rely on active states for touch feedback instead
- Tailwind v4 handles this automatically for hover variants
**Warning signs:** Buttons stay highlighted after tap, tooltips remain visible, visual states don't match interaction model.

### Pitfall 4: Ignoring Safe Area Insets
**What goes wrong:** Content gets hidden behind iPhone notch, home indicator, or Android status bar and navigation bar on fullscreen or PWA apps.
**Why it happens:** Modern phones have non-rectangular screens and system UI overlays; content needs to avoid these areas.
**How to avoid:**
- Add viewport-fit=cover to viewport meta tag
- Use env(safe-area-inset-*) for padding and margins on fixed elements
- Test on iPhone X+ simulators or real devices
**Warning signs:** Content clipped at screen edges, buttons hidden behind home indicator, header overlapped by notch.

### Pitfall 5: Desktop-Heavy Content on Mobile
**What goes wrong:** Loading full desktop assets like large images, heavy JavaScript, or unused features on mobile causes slow load times, high data usage, and poor performance.
**Why it happens:** Not optimizing bundle size, using same assets for all devices, loading desktop-only features on mobile.
**How to avoid:**
- Use responsive images: img srcset and sizes
- Lazy load off-screen content: loading="lazy"
- Code-split mobile-specific vs desktop-specific components
- Minimize JavaScript bundle size (Vite automatically does this)
**Warning signs:** Slow mobile load times, high bounce rates on mobile, poor Core Web Vitals scores, users complaining about data usage.

### Pitfall 6: Over-Reliance on JavaScript for Responsive Layout
**What goes wrong:** Using JavaScript to detect viewport size and manipulate DOM causes slower rendering, content flicker and reflow, and poor performance.
**Why it happens:** Developers default to JavaScript for everything; not leveraging CSS capabilities.
**How to avoid:**
- Use Tailwind responsive utilities for 95% of layout changes
- Use CSS media queries and container queries for responsive behavior
- Only use JavaScript for major layout switches like sidebar vs drawer component swap
**Warning signs:** Visible reflow on page load, flickering content, layout shifts, poor performance metrics.

### Pitfall 7: Long-Press Conflicts
**What goes wrong:** Long-press for drag reordering conflicts with long-press for tooltips; users can't access both features.
**Why it happens:** Single gesture type (long-press) mapped to multiple actions on the same element.
**How to avoid:**
- Differentiate by target: long-press on drag handle = reorder, long-press on item body = tooltip
- Use swipe gestures instead of long-press for one action
- Provide alternative access methods like context menu or explicit buttons
- Use different gesture thresholds: 300ms for tooltip, 500ms for drag
**Warning signs:** Users can't access tooltips, drag conflicts with other interactions, confusion about gesture behavior.

### Pitfall 8: Hamburger Menu Without Discoverability
**What goes wrong:** Users don't recognize the hamburger icon and can't find navigation, especially first-time or older users.
**Why it happens:** Assuming everyone knows the hamburger icon convention; no text label or affordance.
**How to avoid:**
- Add Menu text label next to hamburger icon
- Use aria-label for screen readers
- Show visual hint on first visit (feature hint component already exists in project)
- Consider hybrid approach: hamburger plus most important nav items visible
**Warning signs:** High bounce rates on mobile, users unable to navigate, support requests about where is the menu.

### List Item with Swipe Actions
```svelte
<script>
  import { pan } from 'svelte-gestures';

  let swipeOffset = $state(0);
  let isRevealing = $state(false);

  const SWIPE_THRESHOLD = 80;

  function handlePan(event) {
    // Only allow horizontal swipe
    const offset = event.detail.x;
    swipeOffset = Math.max(-120, Math.min(120, offset));
    isRevealing = Math.abs(swipeOffset) > 20;
  }

  function handlePanEnd(event) {
    const distance = event.detail.x;

    if (distance > SWIPE_THRESHOLD) {
      // Swipe right: complete
      completeItem();
    } else if (distance < -SWIPE_THRESHOLD) {
      // Swipe left: delete
      deleteItem();
    }

    // Reset
    swipeOffset = 0;
    isRevealing = false;
  }
</script>

<div class="relative overflow-hidden">
  <!-- Action backgrounds revealed by swipe -->
  {#if isRevealing}

    use:pan

</div>
```

### Responsive Layout with Conditional Components
```svelte
<script>
  import { onMount } from 'svelte';
  import Sidebar from '$lib/components/Sidebar.svelte';
  import MobileNav from '$lib/components/mobile/MobileNav.svelte';
  import FloatingActionButton from '$lib/components/mobile/FloatingActionButton.svelte';

  let isMobile = $state(false);

  onMount(() => {
    // Match tablet breakpoint (768px)
    const mediaQuery = window.matchMedia('(max-width: 767px)');
    isMobile = mediaQuery.matches;

    const handler = (e) => { isMobile = e.matches; };
    mediaQuery.addEventListener('change', handler);
    return () => mediaQuery.removeEventListener('change', handler);
  });
</script>

<div class="h-screen flex overflow-hidden">
  {#if isMobile}
    <MobileNav />
  {:else}
    <Sidebar />
  {/if}

  <main class="flex-1 overflow-y-auto">
    <!-- Main content -->
    {@render children()}
  </main>
</div>

{#if isMobile}
  <FloatingActionButton />
{/if}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Desktop-first media queries with max-width | Mobile-first breakpoints with min-width | Around 2015 | Tailwind standardized mobile-first; easier progressive enhancement |
| Custom touch event handlers | Pointer events plus gesture libraries | Around 2019 | Unified mouse, touch, pen handling; svelte-gestures uses pointer events |
| user-scalable=no to prevent zoom | font-size 16px on inputs | iOS 10+ in 2016 | Accessibility-compliant solution; better user experience |
| Hamburger menu-only navigation | Hybrid nav or bottom tab bars | Around 2020 | Discoverability issues led to showing key nav items; this project uses hamburger per context decisions |
| Component libraries like Bootstrap or Material UI | Utility-first like Tailwind plus custom components | Around 2019 | Smaller bundles, design control, faster development |
| jQuery Mobile or Framework7 full-stack | SvelteKit plus focused libraries | Around 2020 | Modern framework plus small libraries (svelte-gestures 3KB) beats monolithic mobile frameworks |
| Fixed viewport width like 480px or 320px | Fluid responsive with 100vw and breakpoints | Around 2014 | Device diversity requires fluid layouts |
| Passive event listeners manually | Automatic passive for touch and wheel | Chrome 51+ in 2016 | Svelte 5 makes touch listeners passive by default for performance |

**Deprecated or outdated:**
- jQuery Mobile: Last release 2016; use modern frameworks like SvelteKit
- Bootstrap Mobile: Merged into Bootstrap; use Tailwind for more control
- Hammer.js: Last update 2019; use svelte-gestures which is Svelte 5 native
- sveltejs gestures: Work in progress; use stable svelte-gestures instead
- maximum-scale=1: Accessibility violation; use 16px fonts instead
- webkit-tap-highlight-color: Now unnecessary; use active states

## Open Questions

Things that couldn't be fully resolved:

1. **event-calendar core mobile responsiveness**
   - What we know: Supports touch configuration options like eventLongPressDelay and eventDragMinDistance, has DayGrid and TimeGrid views
   - What's unclear: Specific mobile optimizations, responsive view switching, swipe gesture integration
   - Recommendation: Test on real devices; may need custom wrapper to integrate svelte-gestures for swipe navigation between days

2. **svelte-dnd-action plus svelte-gestures conflict**
   - What we know: Both handle touch events; svelte-dnd-action has Svelte 5 support but was designed for Svelte 3 and 4
   - What's unclear: Potential event conflicts when using long-press for drag with svelte-dnd-action and long-press for tooltips with svelte-gestures on same element
   - Recommendation: Differentiate by target area like drag handle vs item body or use different gesture thresholds; test thoroughly

3. **Drawer overlay behavior dim vs push**
   - What we know: Both patterns are standard; dim is more common on mobile
   - What's unclear: User preference for this specific app
   - Recommendation: Start with dim overlay which is simpler and more mobile-native feel; can switch to push if user testing shows preference

4. **Month view on phone**
   - What we know: event-calendar core has dayGridMonth view; screen real estate is limited on 360px phones
   - What's unclear: Whether month view is usable or valuable on small screens vs tablet and desktop only
   - Recommendation: Make month view available but default to day view on phone; let users discover month view if they need it
