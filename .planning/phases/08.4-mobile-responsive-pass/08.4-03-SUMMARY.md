## Decisions Made

**1. Full-screen overlays for detail panels on mobile**
- ActionDetailPanel and ProjectDetailPanel use `mobileState.isMobile` to conditionally render
- Mobile: `fixed inset-0 z-40` with sticky header/footer
- Desktop: Inline `slide` panel (unchanged)
- Rationale: Side panels don't work on 360px screens

**2. Horizontal scroll for category filters**
- SomedayMaybeList category sidebar becomes horizontal scrollbar on mobile
- Uses `flex overflow-x-auto` on mobile, `flex-col` on tablet+
- Rationale: Better UX than dropdown or hidden menu - users can see all options

**3. Hide text labels on smallest screens**
- SomedayMaybeItem promote buttons hide text labels below phablet breakpoint
- Keep icons visible with `min-h-11 min-w-11` for touch target compliance
- Rationale: Saves horizontal space while maintaining accessibility

**4. Stack metadata on mobile**
- Project created date + stalled badge: stack on phone, inline on phablet+
- Waiting For delegated-to + follow-up date: stack on phone, inline on phablet+
- Rationale: Prevents horizontal overflow and text truncation on narrow screens

## Technical Notes

### Custom Breakpoints Used
- `phone: 360px` (implied by default)
- `phablet: 640px` - Used for stacking transitions
- `tablet: 768px` - Primary desktop/mobile split
- `desktop: 1024px` - Not used in this plan

### Touch Target Strategy
- Buttons: `min-h-11 min-w-11` (44px × 44px minimum)
- List items: `min-h-11` on row container
- Completion circles: Wrapped in `min-h-11 min-w-11` button with inner circle

### iOS Zoom Prevention
- Changed all form inputs from `text-sm` to `text-base`
- Prevents iOS Safari auto-zoom on input focus (triggers at < 16px font-size)
- Does NOT use `user-scalable=no` (bad for accessibility)
